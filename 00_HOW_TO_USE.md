# 00_HOW_TO_USE — Інструкція до кожного чату

> Цей файл — твій мануал. Відкривай його коли починаєш новий чат або забув як працювати з конкретним напрямком. Не вставляй цей файл в чати — він для тебе, не для Claude.

---

## Загальні правила для ВСІХ чатів

### Що вставляти на початку чату
Завжди вставляй відповідний MD файл на початку нового чату. Claude не пам'ятає минулі розмови — кожен чат починається з нуля. Файл дає контекст: що ти вже знаєш, які gap'и, який рівень.

### Коли починати НОВИЙ чат
- Коли тема змінилась (був Collections → перейшов на Concurrency)
- Коли чат став довгим (50+ повідомлень) — Claude починає "забувати" початок
- Коли починаєш новий день/сесію

### Чого НЕ робити в жодному чаті
- Не питай "розкажи все про HashMap" — занадто широко, отримаєш поверхневу відповідь
- Не проси "дай список питань" без контексту — краще проси задавати по одному
- Не копіюй відповідь Claude в нотатки без переосмислення — це ілюзія навчання
- Не витрачай час на formatting нотаток — краще поясни тему вголос

### Як фіксувати прогрес
Наприкінці кожної сесії (не кожного повідомлення, а коли закінчуєш):
1. Скопіюй в MD файл: що вивчив (зміни статус 🔴→🟡→🟢)
2. Запиши gap'и в секцію "⚠️ Gap'и"
3. Якщо була mock сесія — запиши в "Статистика сесій"

## 📘 01_JAVA_CORE — Чат по Core Java

### Модель: Sonnet для теорії, Opus для mock interviews

### Де шукати матеріали (ДО чату, не в чаті)
- **Книга:** "Java Concurrency in Practice" (Goetz) — для Concurrency, найкраще джерело
- **Книга:** "Effective Java" (Bloch) — items по конкретних темах, не читай цілком
- **Відео:** YouTube канал "Java Brains" — короткі пояснення Spring/Java
- **Документація:** Oracle Java docs — для точних деталей API
- **Baeldung.com** — практичні приклади з кодом (але не копіюй бездумно)

### Як проводити сесію по ТЕОРІЇ (1-1.5 год)

**Етап 1 — Самостійне вивчення (30-40 хв, БЕЗ Claude)**
Прочитай тему з книги/статті. Спробуй зрозуміти сам. Намалюй схему на папері (HashMap buckets, Thread states, JVM memory). Це важливо — не йди відразу в чат.

**Етап 2 — Перевірка розуміння в чаті (20-30 хв)**
```
Ось як я розумію [тему]: [твоє пояснення 5-10 речень].
Де я помиляюсь? Що пропустив? Що б ти додав для Senior рівня?
```
Це значно ефективніше ніж "поясни мені HashMap" — бо Claude бачить ЩО ти вже знаєш і де gap.

**Етап 3 — Закріплення (10-15 хв)**
```
Задай мені 3 питання по [темі] рівня Senior. Одне за одним, 
чекай мою відповідь, потім оціни і дай правильну відповідь якщо я помилився.
```

### Як проводити MOCK (30 хв, Opus)
```
Проведи mock interview по Core Java, 30 хвилин. Рівень: Senior.
Почни з warm-up (1-2 прості питання), потім ускладнюй.
Задавай follow-up до моїх відповідей. В кінці дай оцінку 1-10
і список тем які треба підтягнути.
Вставляю контекст: [01_JAVA_CORE.md]
```

### Чого НЕ питати
- "Поясни Collections" — занадто широко
- "Дай мені все про GC" — будеш читати стіну тексту і нічого не запам'ятаєш
- "Який GC найкращий?" — немає правильної відповіді без контексту

### Що працює найкраще
- "Я думаю що X працює так: [...]. Правильно?" → targeted feedback
- "Напиши код де порушення equals/hashCode ламає HashMap, я спробую знайти баг"
- "Чим відрізняється [A] від [B] в контексті [C]?" → конкретне порівняння

## 📗 02_SPRING_HIBERNATE — Чат по Spring

### Модель: Sonnet для теорії і code review, Opus для mock

### Де шукати матеріали
- **Документація:** docs.spring.io — офіційна, найточніша (Spring Boot Reference)
- **Baeldung.com** — практичні how-to з кодом
- **Відео:** "Laurentiu Spilca" YouTube — Spring Security глибоко
- **Книга:** "Spring in Action" (Walls) — хороший overview, але ти вже досвідчений, читай вибірково
- **Vlad Mihalcea blog** — Hibernate глибоко (N+1, caching, performance)

### Як проводити сесію

**Формат 1 — Code Review (найефективніший для Spring)**
Коли пишеш код проекту і хочеш перевірити:
```
Зроби Senior code review цього сервісу. Перевір: 
правильність @Transactional, можливі N+1, exception handling,
чи можна щось спростити. Вставляю код:
[твій код]
```

**Формат 2 — Механізм під капотом**
```
Поясни як Spring створює proxy для @Transactional. 
Step by step: від анотації на класі до реального SQL в БД.
Потім задай мені питання щоб перевірити чи я зрозумів.
```

**Формат 3 — "Зламай і виправ"**
Найкращий спосіб зрозуміти @Transactional, N+1, circular dependencies:
```
Напиши Spring сервіс з 3 типовими помилками Senior рівня 
(self-invocation, N+1, неправильний cascade). 
Не кажи які помилки — я знайду сам.
```

### Чого НЕ питати
- "Як зробити REST endpoint?" — ти це знаєш за 4 роки, не витрачай час
- "Як підключити Spring Security?" — це tutorial, читай docs
- Будь-які how-to які є в документації — Claude не замінює docs

### Що працює
- "Чому мій @Transactional не ролбечить? Ось код: [...]" → debugging
- "В якому порядку Spring створює proxy? Self-invocation — поясни НА МОЄМУ сервісі"
- "N+1 в моєму Route → RouteStop → Poi — покажи де і як виправити"

## 📙 03_DATABASES — Чат по базах даних

### Модель: Sonnet

### Де шукати матеріали
- **"Use The Index, Luke"** (use-the-index-luke.com) — індекси, EXPLAIN, безкоштовна книга онлайн
- **PostgreSQL docs** — EXPLAIN output format, MVCC internals
- **Redis University** (university.redis.com) — безкоштовні курси по Redis
- **Kafka docs + Confluent blog** — Kafka deep dives
- **"Designing Data-Intensive Applications" (Kleppmann)** — найкраща книга по distributed data

### Як проводити сесію

**SQL практика:**
```
Дай мені 3 SQL задачі зростаючої складності для таблиць мого проекту 
(poi, route, route_stop, rating). Використай window functions і CTE. 
Я вирішу, ти покажеш оптимальніший варіант і поясниш EXPLAIN.
```

**Розуміння індексів:**
Не проси "поясни B-tree". Замість цього:
```
В мене таблиця poi з колонками (id, name, city, category, rating).
Типові запити: WHERE city = ? AND category = ?, ORDER BY rating DESC, LIMIT 10.
Які індекси створити? Який буде EXPLAIN? Де індекс НЕ допоможе?
```

**Redis/Kafka — через проект:**
```
Ось мій use case: кешування POI по місту, 10K запитів/хв, 
дані оновлюються раз на годину.
Яка Redis структура? Який TTL? Як invalidate? Що якщо cache stampede?
```

### Чого НЕ питати
- "Поясни SQL JOIN" — ти це знаєш
- "Що таке Redis?" — прочитай overview в docs, не витрачай час Claude

### Що працює
- Давати СВОЇ запити і просити оптимізувати
- Просити написати проблемний запит → ти знаходиш проблему
- "EXPLAIN ANALYZE показує Seq Scan — чому?" з конкретним запитом

**Формат "розбір свого проекту":**

## 📓 06_PROJECT — Чат про проект (interview prep)

### Модель: Opus

### Коли використовувати
Цей чат НЕ для написання коду (це CLAUDE.md + Claude Code). Цей чат для підготовки до РОЗМОВИ про проект.

### Формати сесій

**Формат 1 — Elevator pitch practice:**
```
Я розкажу про свій проект за 30 секунд. Оціни: 
чи зрозуміло що робить, чи чути technical depth, 
чи хочеться задати follow-up. Контекст: [06_PROJECT.md]

[Твій pitch]
```

**Формат 2 — Trade-off drilling:**
```
Задавай мені питання "чому?" про мій проект. 
Одне за одним. "Чому Redis?", "Чому не MongoDB?", 
"Чому цей алгоритм?". Я відповідаю, ти оцінюєш 
чи достатньо глибока відповідь для Senior.
```

**Формат 3 — Підготовка до конкретної співбесіди:**
```
У мене завтра технічне інтерв'ю в [тип компанії]. 
Задай мені 10 найімовірніших питань про мій проект.
Я відповідаю, ти даєш feedback.
```

### Чого НЕ робити
- Не проси написати за тебе відповіді на питання — ти маєш формулювати сам
- Не вчи відповіді напам'ять — вчи СТРУКТУРУ відповіді і trade-offs

## 🖥️ CLAUDE.md — Claude Code (написання проекту)

### Модель: Sonnet (для коду Sonnet ідеальний)

### Як працювати

**Початок сесії:**
Вставляєш CLAUDE.md. Потім конкретна задача:
```
Крок 5: Route Optimizer. Створи HaversineCalculator 
з методом distance(lat1, lng1, lat2, lng2) → meters.
Додай unit тести.
```

**Правила взаємодії:**
- Давай конкретні задачі, не "зроби все"
- Один крок за раз: HaversineCalculator → тести → NearestNeighbor → тести
- Після кожного кроку: ПРОЧИТАЙ код, ЗРОЗУМІЙ кожен рядок
- Якщо щось незрозуміло → запитай "чому тут [X]?"
- Не копіюй код який не розумієш — на співбесіді спитають

**Після Claude Code сесії:**
Відкрий проект в IntelliJ, пройдись по коду. Зможеш пояснити кожен клас? Якщо ні — запиши питання для наступної теоретичної сесії.

### ⚠️ Критично важливо
Проект — це НЕ портфоліо для показу "дивіться який гарний код". Проект — це МАТЕРІАЛ для співбесіди. Ти маєш вміти пояснити КОЖНЕ рішення в коді. Якщо Claude написав щось що ти не розумієш — зупинись і розберись.

## 📙 07_ENGLISH_BEHAVIORAL — Чат англійською

### Модель: Opus (потрібна оцінка нюансів мови)

### Як працювати

**Правило #1: ВСЕ в цьому чаті — англійською.** Навіть якщо помиляєшся. Навіть якщо незручно. Переключення на українську — заборонено.

**Формат 1 — Пояснення теми (щодня 15-20 хв):**
```
I'll explain [HashMap internals / my project / @Transactional] 
in English. Correct my grammar, suggest more natural phrasing, 
and rate my technical accuracy. Here I go:

[Твоє пояснення]
```

**Формат 2 — STAR story review:**
```
Here's my STAR story for "most complex technical problem". 
Check: (1) all 4 STAR components present, (2) grammar, 
(3) does it sound natural, (4) is it specific enough for Senior.

[Твоя story]
```

**Формат 3 — Full mock (з тижня 9):**
```
Run a full mock interview in English. You're a Senior interviewer 
at a European product company. Format: 
- 5 min: tell me about yourself + project
- 15 min: technical questions (Java/Spring)
- 10 min: behavioral (STAR)
- 5 min: my questions to you
After each answer, give brief feedback on grammar and content.
Final score: technical (1-10), communication (1-10), English (1-10).
```

### Чого НЕ робити
- Не пиши ідеальні відповіді — пиши як ГОВОРИШ, навіть з помилками
- Не перекладай з української в голові — думай англійською (прийде з практикою)
- Не витрачай час на граматичні правила — вчи через USE, не через правила

### Лайфхак
Записуй voice memo на телефон коли пояснюєш тему вголос. Послухай через 5 хвилин. Ти одразу почуєш де запинаєшся, де filler words ("um", "like"), де занадто довгі паузи.
