Еволюція Pattern Matching

Pattern matching — це перевірка типу + binding змінної в одному виразі. Java додавала його поступово, від простого instanceof до повноцінного switch з деконструкцією.

До і після — відчуй різницю

до Java 16

if (obj instanceof String) { String s = (String) obj; System.out.println(s.length()); }

Java 16+

if (obj instanceof String s) { System.out.println(s.length()); // s скопована з obj }

до Java 21

if (shape instanceof Circle c) { area = Math.PI * c.r() * c.r(); } else if (shape instanceof Rect r) { area = r.w() * r.h(); }

Java 21

area = switch (shape) { case Circle c -> Math.PI*c.r()*c.r(); case Rect r -> r.w() * r.h(); };
Java 16

Pattern matching для instanceof

Замість instanceof + явного cast — binding variable автоматично доступна в тілі if-блоку.

// Scope binding variable if (obj instanceof String s) { // s доступна тут: String System.out.println(s.toUpperCase()); } // s недоступна тут // Можна одразу використовувати в умові if (obj instanceof String s && s.length() > 5) { System.out.println(s); }

Тонкощі scope

// Negation: s доступна в else-гілці if (!(obj instanceof String s)) { // s тут НЕ доступна return; } // після early return — s доступна! System.out.println(s.length()); // Shadowing: binding variable може тінити поле class Foo { Object s = "field"; void check(Object obj) { if (obj instanceof String s) { // s тут — binding var, не поле! } } }
Binding variable не є final — її можна перепризначити. Але компілятор попередить якщо це виглядає підозріло.
Java 21

Type patterns у switch

Switch може матчити по типу, а не тільки по значенню. Кожен case перевіряє тип і одразу створює typed змінну.

sealed interface Shape permits Circle, Rectangle, Triangle {} double area(Shape s) { return switch (s) { case Circle c -> Math.PI * c.r() * c.r(); case Rectangle r -> r.width() * r.height(); case Triangle t -> t.base() * t.height() / 2; // default не потрібен — sealed є exhaustive! }; }

null у switch — змінилась поведінка

До Java 21 switch кидав NullPointerException на null. Тепер можна явно обробити:

switch (obj) { case null -> System.out.println("null!"); case String s -> System.out.println(s); case Integer i -> System.out.println(i); default -> System.out.println("other"); } // або об'єднати null з дефолтом case null, default -> System.out.println("none");
Switch-expression (switch (...) { ... } зі стрілкою і значенням) — stable з Java 14. Pattern matching у switch — stable з Java 21.
Java 21

Guarded patterns — when

До type pattern можна додати when-умову. Матч спрацює тільки якщо тип збігається і умова true.

String classify(Object obj) { return switch (obj) { case Integer i when i < 0 -> "negative int"; case Integer i when i == 0 -> "zero"; case Integer i -> "positive int"; case String s when s.isEmpty() -> "empty string"; case String s -> "string: " + s; default -> "other"; }; }

Порядок важливий — dominance

Case з when є вужчим ніж case без нього. Якщо поставити broader case вище — компілятор видасть помилку.

// ПОМИЛКА компіляції: dominance violation switch (obj) { case Integer i -> ... // catches all Integer case Integer i when i > 0 -> ... // dead code! } // ПРАВИЛЬНО: narrower вище switch (obj) { case Integer i when i > 0 -> ... case Integer i -> ... }
when замінив старий && у case, але читається значно краще і компілятор статично перевіряє порядок.
Java 21

Record patterns — деконструкція

При матчингу по record-типу можна одразу деструктурувати його компоненти. Не треба окремо викликати accessor-методи.

record Point(int x, int y) {} record Circle(Point center, double radius) {} // Без record patterns if (obj instanceof Circle c) { int x = c.center().x(); int y = c.center().y(); } // З record patterns — деконструкція одразу if (obj instanceof Circle(Point(int x, int y), double r)) { System.out.println(x + ", " + y + ", r=" + r); }

Вкладені record patterns у switch

sealed interface Expr permits Num, Add, Mul {} record Num(int v) implements Expr {} record Add(Expr l, Expr r) implements Expr {} record Mul(Expr l, Expr r) implements Expr {} int eval(Expr e) { return switch (e) { case Num(int v) -> v; case Add(Expr l, Expr r) -> eval(l) + eval(r); case Mul(Expr l, Expr r) -> eval(l) * eval(r); // exhaustive — default не потрібен }; }
Це і є "Expression Problem" — найкращий приклад для демонстрації sealed + record + pattern matching разом на співбесіді.
нюанс

Dominance — порядок case має значення

Один pattern домінує над іншим якщо він матчить всі значення, що матчить другий. Компілятор забороняє ставити broader pattern вище narrower.

// Dominance між типами в ієрархії switch (animal) { case Animal a -> ... // ERROR: домінує над Cat case Cat c -> ... // dead code } // Правильно switch (animal) { case Cat c -> ... // спочатку вузький case Animal a -> ... // потім широкий } // Dominance: guarded vs unguarded switch (n) { case Integer i -> ... // ERROR: домінує case Integer i when i > 0 -> ... // dead code }

Чому це compile-time, а не runtime?

Компілятор будує граф dominance між усіма case і визначає мертвий код статично. Це одна з головних переваг pattern matching — помилки порядку ловляться до запуску.

Правило: завжди розставляй case від найвужчого до найширшого. Guarded pattern (з when) — вужчий за unguarded того самого типу. Subtype — вужчий за supertype.

Dominance між record patterns

record Box(Object content) {} switch (box) { // вужчий — String всередині case Box(String s) when s.length() > 5 -> ... // ширший — будь-який String case Box(String s) -> ... // найширший case Box b -> ... }
гарантія

Exhaustiveness — повнота switch

Switch expression зобов'язаний покривати всі можливі значення. Для sealed ієрархій компілятор перевіряє це статично.

sealed interface Status permits Ok, Error, Pending {} // ПОМИЛКА: не покрито Pending String msg = switch (status) { case Ok ok -> "ok"; case Error e -> "err"; // compile error: не всі варіанти покриті }; // ПРАВИЛЬНО String msg = switch (status) { case Ok ok -> "ok"; case Error e -> "err: " + e.msg(); case Pending p -> "pending"; };

Що дає exhaustiveness з sealed

Якщо додати новий subtype до sealed interface — всі switch без default перестануть компілюватись. Ти знайдеш всі місця де треба обробити новий випадок.

Антипатерн: додавати default у sealed switch. Ти втрачаєш compile-time safety — новий subtype буде мовчки проходити через default замість того щоб викинути помилку компіляції.

Exhaustiveness для non-sealed типів

// Для Object або не-sealed — треба default switch (obj) { case String s -> ...; case Integer i -> ...; default -> ...; // обов'язково } // Для sealed — default опціональний (і небажаний) switch (shape) { // sealed Shape case Circle c -> ...; case Rectangle r -> ...; // НЕ додавай default тут }
антипатерни

Типові помилки

1. default у sealed switch

// ПОГАНО: втрачаємо compile-time safety switch (shape) { case Circle c -> c.area(); default -> 0; // новий Shape тихо повертає 0 } // ДОБРЕ: exhaustive без default switch (shape) { case Circle c -> c.area(); case Rectangle r -> r.area(); // compile error якщо додати новий Shape }

2. Неправильний порядок case

// ПОМИЛКА КОМПІЛЯЦІЇ: broader pattern вище switch (n) { case Number n -> ...; // covers Integer теж case Integer i -> ...; // ERROR: dominated! }

3. Забули null у switch expression

// NullPointerException якщо obj == null switch (obj) { // кидає NPE на null! case String s -> ...; default -> ...; } // Правильно: явний case null switch (obj) { case null -> handleNull(); case String s -> ...; default -> ...; }

4. Змішувати старий і новий синтаксис

// НЕ можна: pattern + traditional case labels switch (obj) { case String s -> ...; // новий case "hello": ...; // старий — ПОМИЛКА } // Вибери один стиль — або arrow, або colon
Підсумок для співбесіди: instanceof (binding var) → type patterns у switch → guarded patterns (when) → record deconstruction → sealed exhaustiveness. Кожен крок будується на попередньому.
