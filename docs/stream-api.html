основи

Lazy evaluation — як Stream виконується

Stream операції діляться на intermediate (ліниві, повертають Stream) і terminal (запускають виконання). Без terminal-операції pipeline не виконується взагалі.

// Intermediate: filter, map, flatMap, sorted, distinct, // limit, skip, peek, takeWhile, dropWhile // Terminal: collect, forEach, count, sum, min, max, // findFirst, findAny, anyMatch, reduce, toList // Цей код нічого не виводить — немає terminal! Stream.of(1, 2, 3) .filter(n -> { System.out.println(n); return n > 1; }); // Додаємо terminal — тепер виконується Stream.of(1, 2, 3) .filter(n -> n > 1) .map(n -> n * 10) .forEach(System.out::println); // 20, 30

Short-circuit: виконання зупиняється раніше

// limit, findFirst, anyMatch — short-circuit Stream.iterate(0, n -> n + 1) // нескінченний стрім! .filter(n -> n % 2 == 0) .limit(5) // зупиниться після 5 .toList(); // [0, 2, 4, 6, 8] // anyMatch зупиняється при першому true boolean hasEven = list.stream() .anyMatch(n -> n % 2 == 0); // не обходить всі!
Stream — це pipeline, не структура даних. Після terminal операції стрім закритий — його не можна перевикористати. Новий виклик stream() створює новий pipeline.
трансформація

map vs flatMap

map — 1 елемент → 1 елемент. flatMap — 1 елемент → N елементів (flatten вкладених стрімів).

List<List<String>> nested = List.of( List.of("a", "b"), List.of("c", "d") ); // map → Stream<List<String>> — не те! nested.stream() .map(List::stream) // Stream<Stream<String>> .toList(); // [[a,b], [c,d]] // flatMap → Stream<String> — плоский список nested.stream() .flatMap(Collection::stream) // Stream<String> .toList(); // [a, b, c, d]

Реальний приклад: всі теги з постів

record Post(String title, List<String> tags) {} List<Post> posts = ...; // Всі унікальні теги по всіх постах Set<String> allTags = posts.stream() .flatMap(post -> post.tags().stream()) .collect(Collectors.toSet()); // flatMapToInt — для примітивів (без boxing) int[] allLengths = posts.stream() .flatMapToInt(p -> p.tags().stream() .mapToInt(String::length)) .toArray();

mapMulti (Java 16+) — альтернатива flatMap

// mapMulti — ефективніше ніж flatMap для малих списків posts.stream() .<String>mapMulti((post, consumer) -> post.tags().forEach(consumer)) .toList();
collectors

groupingBy — найпотужніший collector

Групує елементи в Map<K, List<V>>. З downstream collector може агрегувати довільним чином.

record Employee(String name, String dept, double salary) {} List<Employee> employees = ...; // Базовий: Map<String, List<Employee>> Map<String, List<Employee>> byDept = employees.stream() .collect(Collectors.groupingBy(Employee::dept)); // Рахуємо кількість у кожній групі Map<String, Long> countByDept = employees.stream() .collect(Collectors.groupingBy( Employee::dept, Collectors.counting())); // Середня зарплата по відділах Map<String, Double> avgSalaryByDept = employees.stream() .collect(Collectors.groupingBy( Employee::dept, Collectors.averagingDouble(Employee::salary)));

partitioningBy — спеціальний groupingBy для boolean

// Ділить на true/false групи Map<Boolean, List<Employee>> seniorVsJunior = employees.stream() .collect(Collectors.partitioningBy( e -> e.salary() > 5000)); seniorVsJunior.get(true); // seniors seniorVsJunior.get(false); // juniors // partitioningBy завжди повертає обидва ключі // groupingBy — тільки ті ключі що є в даних
Порядок ключів у Map: groupingBy за замовчуванням повертає HashMap. Якщо потрібен порядок — передай TreeMap::new як другий аргумент або використовуй LinkedHashMap::new.
// Sorted map Map<String, List<Employee>> sorted = employees.stream() .collect(Collectors.groupingBy( Employee::dept, TreeMap::new, // ← map factory Collectors.toList()));
collectors

toMap і joining

toMap — ключ/значення з елементів

record User(long id, String name) {} // id → User Map<Long, User> byId = users.stream() .collect(Collectors.toMap( User::id, // keyMapper u -> u)); // valueMapper // id → name Map<Long, String> idToName = users.stream() .collect(Collectors.toMap(User::id, User::name)); // ОБЕРЕЖНО: дублікати ключів → IllegalStateException! // Треба mergeFunction: Map<String, User> byName = users.stream() .collect(Collectors.toMap( User::name, u -> u, (existing, duplicate) -> existing)); // keep first

joining — конкатенація рядків

List<String> names = List.of("Alice", "Bob", "Carol"); // Просто joined names.stream().collect(Collectors.joining()); // → "AliceBobCarol" // З роздільником names.stream().collect(Collectors.joining(", ")); // → "Alice, Bob, Carol" // З prefix і suffix names.stream().collect(Collectors.joining(", ", "[", "]")); // → "[Alice, Bob, Carol]" // Те саме через String.join (простіше) String.join(", ", names); // → "Alice, Bob, Carol"
toMap кидає IllegalStateException при дублікаті ключа — завжди передавай mergeFunction якщо дублікати можливі. Це дуже часта помилка в продакшн коді.
collectors

Downstream collectors — вкладена агрегація

Другий аргумент groupingBy — downstream collector, що застосовується до кожної групи. Дозволяє будь-яку агрегацію всередині груп.

// Всі downstream collectors Collectors.counting() // кількість → Long Collectors.summingInt/Long/Double // сума Collectors.averagingDouble() // середнє → Double Collectors.summarizingInt() // count+sum+min+max+avg Collectors.minBy(comparator) // Optional<T> Collectors.maxBy(comparator) // Optional<T> Collectors.joining() // для String Collectors.toSet() // Set замість List Collectors.mapping(f, downstream) // трансформація перед Collectors.toUnmodifiableList() // immutable результат

Складні комбінації

// Імена співробітників по відділах (не об'єкти) Map<String, List<String>> namesByDept = employees.stream() .collect(Collectors.groupingBy( Employee::dept, Collectors.mapping( Employee::name, // трансформація Collectors.toList()))); // Топ зарплата по відділах Map<String, Optional<Employee>> topPaidByDept = employees.stream() .collect(Collectors.groupingBy( Employee::dept, Collectors.maxBy( Comparator.comparingDouble(Employee::salary)))); // Вкладений groupingBy Map<String, Map<String, Long>> countByDeptAndLevel = employees.stream() .collect(Collectors.groupingBy( Employee::dept, Collectors.groupingBy( Employee::level, Collectors.counting())));
агрегація

reduce і collect

reduce — fold операція

// reduce(identity, accumulator) int sum = Stream.of(1, 2, 3, 4) .reduce(0, Integer::sum); // → 10 // reduce без identity → Optional (стрім може бути порожнім) Optional<String> longest = names.stream() .reduce((a, b) -> a.length() >= b.length() ? a : b); // reduce(identity, accumulator, combiner) — для parallel int parallelSum = numbers.parallelStream() .reduce(0, Integer::sum, Integer::sum);

collect vs reduce — коли що

// reduce — immutable results, не змінює контейнер int total = list.stream().reduce(0, Integer::sum); // collect — mutable reduction, ефективніше для складних агрегацій // Ніколи не роби так (неефективно + не паралелізується): List<String> bad = stream.reduce( new ArrayList<>(), (list, e) -> { list.add(e); return list; }, // мутуємо! (l1, l2) -> { l1.addAll(l2); return l1; }); // Правильно — collect: List<String> good = stream.collect(Collectors.toList());

Примітивні стріми — без boxing

// IntStream, LongStream, DoubleStream int sum = IntStream.rangeClosed(1, 100).sum(); double avg = IntStream.of(1,2,3).average().orElse(0); IntSummaryStatistics stats = list.stream() .mapToInt(String::length) .summaryStatistics(); // count, sum, min, max, avg
parallel

Parallel streams

Parallel stream розбиває дані на частини і обробляє у ForkJoinPool.commonPool(). Не завжди швидше — overhead на split/merge і координацію потоків.

// Увімкнути parallel list.parallelStream()... stream.parallel()... // Вимкнути всередині pipeline stream.parallel().filter(...).sequential().forEach(...);

Коли parallel допомагає, а коли шкодить

УмоваParallel
Великий обсяг даних (100k+)допомагає
CPU-intensive операціїдопомагає
Малі колекції (< 10k)overhead перебільшує
IO-bound операціїForkJoinPool не для цього
Stateful операції (sorted)втрата переваги
forEach з side effectsrace condition
ArrayList (splittable)добре розбивається
LinkedList (не splittable)погано розбивається
// НЕБЕЗПЕЧНО: stateful lambda в parallel List<Integer> result = new ArrayList<>(); list.parallelStream() .filter(n -> n > 0) .forEach(result::add); // race condition на ArrayList! // ПРАВИЛЬНО: collect thread-safe List<Integer> safe = list.parallelStream() .filter(n -> n > 0) .collect(Collectors.toList()); // thread-safe ✓
Parallel stream використовує ForkJoinPool.commonPool() — спільний для всього застосунку. Заблокувати його IO операцією = заблокувати інші частини програми.
антипатерни

Типові помилки

1. Стрім використаний двічі

Stream<String> stream = list.stream().filter(s -> s.length() > 3); stream.forEach(System.out::println); stream.count(); // IllegalStateException: stream already operated upon // Рішення: використай Supplier<Stream> Supplier<Stream<String>> ss = () -> list.stream().filter(s -> s.length() > 3); ss.get().forEach(System.out::println); ss.get().count();

2. toMap без merge function при дублікатах

// IllegalStateException: Duplicate key users.stream().collect(Collectors.toMap(User::name, u -> u)); // Якщо дублікати можливі — завжди передавай merge users.stream().collect(Collectors.toMap( User::name, u -> u, (a, b) -> a));

3. forEach замість map/collect

// ПОГАНО: side-effect forEach для трансформації List<String> result = new ArrayList<>(); list.stream().forEach(s -> result.add(s.toUpperCase())); // ДОБРЕ: декларативно List<String> result = list.stream() .map(String::toUpperCase) .collect(Collectors.toList());
На співбесіді: groupingBy + downstream — найчастіше питання. Обов'язково знай різницю map/flatMap, toMap з merge function, і чому parallel не завжди швидше.
