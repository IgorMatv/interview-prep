контекст

Навіщо це взагалі

В Java кожен об'єкт успадковує equals() і hashCode() від Object. За замовчуванням вони працюють по ідентичності (адреса в пам'яті). Але колекції типу HashMap, HashSet, LinkedHashSet потребують семантичної рівності.

Object за замовчуванням

// equals: == new Point(1,2).equals( new Point(1,2)) // → false (різні об'єкти!) // hashCode: System.identityHashCode // → різні значення

після перевизначення

// equals: по полях new Point(1,2).equals( new Point(1,2)) // → true // hashCode: по тих самих полях // → однакові значення
Наслідок порушення: set.contains(new Point(1,2)) повертає false, навіть якщо такий Point вже є в Set — бо шукаємо не в тому бакеті.

контракт

equals() — 5 правил

Контракт визначений у Javadoc Object.equals(). Порушення будь-якого правила ламає поведінку колекцій.

рефлексивність
x.equals(x) == true завжди. Об'єкт рівний сам собі.
симетричність
x.equals(y) == y.equals(x). Якщо x рівний y — то y рівний x.
транзитивність
x.equals(y) && y.equals(z) → x.equals(z). Якщо x=y і y=z, то x=z.
консистентність
Повторні виклики дають той самий результат поки поля не змінились.
null
x.equals(null) == false завжди. Ніколи не кидати NPE при порівнянні з null.

Де найчастіше ламається симетричність

Типова проблема: клас A знає про клас B і вважає їх рівними, але B не знає про A.

class CaseInsensitiveString { private final String s; @Override public boolean equals(Object o) { if (o instanceof CaseInsensitiveString c) return s.equalsIgnoreCase(c.s); // ПОРУШЕННЯ: порівнює з String теж! if (o instanceof String str) return s.equalsIgnoreCase(str); return false; } } // cis.equals("hello") → true // "hello".equals(cis) → false ← порушення!
Правило Блоха: equals між різними типами практично завжди порушує симетричність або транзитивність. Порівнюй тільки об'єкти одного типу.
контракт

hashCode() — 3 правила

консистентність
Повторні виклики на незмінному об'єкті повертають однакове значення протягом одного запуску JVM.
equals → hash
Якщо x.equals(y) == true, то x.hashCode() == y.hashCode(). Це обов'язкова умова.
hash → equals
Якщо x.hashCode() == y.hashCode() — це не означає x.equals(y). Колізії допустимі, просто погіршують продуктивність.

Що відбувається при порушенні

// Клас з equals але без hashCode class BadKey { int id; @Override public boolean equals(Object o) { return o instanceof BadKey k && k.id == id; } // hashCode() не перевизначений! } var map = new HashMap<BadKey, String>(); var k1 = new BadKey(); k1.id = 42; map.put(k1, "hello"); var k2 = new BadKey(); k2.id = 42; map.get(k2); // → null ← ЗАГУБЛЕНО! // k1.equals(k2) == true, але різні бакети
IntelliJ / IDE завжди попередить якщо перевизначено лише один з двох методів. Ніколи не ігноруй це попередження.
механіка

Зв'язок між equals і hashCode

HashMap шукає ключ у два кроки: спочатку знаходить бакет по hashCode(), потім перебирає ланцюжок через equals().

Ключове: однаковий hashCode → один бакет → equals() порівнює вузли в ланцюжку. Різний hashCode → різний бакет → equals() навіть не викликається.
реалізація

Правильна реалізація

Сучасний Java-стиль — використовувати Objects.equals() і Objects.hash(). Для record це генерується автоматично.

class Point { private final int x, y; @Override public boolean equals(Object o) { if (this == o) return true; // 1. identity fast-path if (!(o instanceof Point p)) return false; // 2. тип + null return x == p.x && y == p.y; // 3. поля } @Override public int hashCode() { return Objects.hash(x, y); // ті самі поля! } }

Record — автоматично

// Record генерує коректний equals + hashCode сам record Point(int x, int y) {} new Point(1, 2).equals(new Point(1, 2)) // → true

Objects.hash() під капотом

// Objects.hash(x, y) розкривається в: Arrays.hashCode(new Object[]{x, y}) // ≈ 31 * (31 * 1 + x) + y // Ручний варіант (швидше, без boxing): int result = 31 * x + y; // для int полів
Правило: у equals() і hashCode() використовуй рівно ті самі поля. Пропустив поле в hashCode — отримаєш надмірні колізії. Пропустив у equals — отримаєш неправильну рівність.
нюанси

Підводні камені

1. Mutable ключі в HashMap

Якщо змінити поле об'єкта-ключа після вставки — hashCode() зміниться, і ключ "загубиться" в іншому бакеті.

var key = new Point(1, 2); map.put(key, "value"); key.x = 99; // поле змінилось! map.get(key); // → null ← загублено map.containsKey(key); // → false // але значення досі є у старому бакеті
Ключі HashMap мають бути immutable або принаймні їх поля, що беруть участь у hashCode/equals, не мають змінюватись після вставки.

2. Кешування hashCode

Для immutable класів з дорогим hashCode (наприклад, String) — кешуємо результат.

class ImmutableKey { private final String value; private int hash; // кеш, 0 = не обчислено @Override public int hashCode() { int h = hash; if (h == 0 && value != null) { h = value.hashCode(); hash = h; // racy but safe для int } return h; } }

3. Float і double

// equals: використовуй compareTo для float/double Double.compare(a.val, b.val) == 0 // не == // hashCode: спеціальні методи Double.hashCode(val) // або Double.doubleToLongBits Float.hashCode(val) // або Float.floatToIntBits // Бо: Double.NaN != Double.NaN, але // Double.compare(NaN, NaN) == 0
спадкування

equals і спадкування — проблема транзитивності

Неможливо розширити instantiable клас, додати значуще поле і при цьому зберегти контракт equals. Це фундаментальна проблема.

class Point { int x, y; } class ColorPoint extends Point { Color color; } Point p = new Point(1, 2); ColorPoint cp = new ColorPoint(1, 2, RED); // Варіант A: cp.equals(p) перевіряє колір cp.equals(p) // false (немає кольору) p.equals(cp) // true ← порушення симетричності! // Варіант B: ігнорувати колір при mixed compare cp.equals(p) // true p.equals(cp) // true // але тоді: cp1.equals(p) && p.equals(cp2) // cp1(RED) і cp2(BLUE) — порушення транзитивності!

Рішення: composition замість inheritance

// Замість extends Point — містить Point class ColorPoint { private final Point point; // composition private final Color color; public Point asPoint() { return point; } @Override public boolean equals(Object o) { if (!(o instanceof ColorPoint cp)) return false; return cp.point.equals(point) && cp.color.equals(color); // чисто } }
Саме тому java.sql.Timestamp (extends java.util.Date) порушує контракт equals — і Javadoc явно попереджає не змішувати їх у колекціях.
антипатерни

Типові помилки

1. getClass() замість instanceof

// ПОГАНО: ламає Liskov — підклас не може бути рівним @Override public boolean equals(Object o) { if (o == null || getClass() != o.getClass()) return false; // Point і ColorPoint завжди різні ... } // ДОБРЕ: instanceof дозволяє поліморфізм if (!(o instanceof Point p)) return false;

2. equals але без hashCode

// HashSet і HashMap стають непередбачуваними var set = new HashSet<Point>(); set.add(new Point(1, 2)); set.contains(new Point(1, 2)); // → false!

3. hashCode повертає константу

// "Правильно" з точки зору контракту, але... @Override public int hashCode() { return 42; } // Всі об'єкти в один бакет → O(n) HashMap // Технічно не порушення, але катастрофа для perf
На співбесіді: якщо запитають про equals/hashCode — завжди згадай контракт (5 правил + 3 правила), mutable keys у HashMap, і проблему спадкування з транзитивністю. Це три кути, які відрізняють підготовлену людину.
