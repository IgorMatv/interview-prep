структура

Як B-tree індекс працює

B-tree (Balanced Tree) — збалансоване дерево. Листові ноди містять значення ключа і вказівник на heap (фізичне розташування рядку). Пошук за O(log n).

-- Звичайний пошук без індексу: Sequential Scan -- Читає кожен рядок таблиці: O(n) SELECT * FROM orders WHERE user_id = 42; -- При 1M рядків → 1M читань -- З індексом: Index Scan → O(log n) CREATE INDEX idx_orders_user_id ON orders(user_id); -- B-tree: пройти дерево → знайти листову ноду → heap lookup -- При 1M рядків → ~20 читань -- Переваги B-tree: -- = equality: WHERE id = 5 -- Range: WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31' -- ORDER BY: може уникнути Sort вузла -- IS NULL / IS NOT NULL
B-tree підтримує: =, <, <=, >, >=, BETWEEN, IN, LIKE 'prefix%'. Не підтримує: LIKE '%suffix', LIKE '%middle%' — для цього Full-text search або GIN.
composite

Composite index і left-prefix rule

Composite index по (A, B, C) — відсортований спочатку за A, потім B, потім C. Left-prefix rule: індекс використовується тільки якщо є фільтр по крайньому лівому стовпцю.

CREATE INDEX idx_orders_user_status_date ON orders(user_id, status, created_at); -- ✓ Використовує індекс (є user_id — лівий крайній) WHERE user_id = 42 WHERE user_id = 42 AND status = 'PAID' WHERE user_id = 42 AND status = 'PAID' AND created_at > '2024-01-01' WHERE user_id = 42 AND created_at > '2024-01-01' -- частково (пропускає status) -- ✗ НЕ використовує індекс WHERE status = 'PAID' -- нема user_id WHERE created_at > '2024-01-01' -- нема user_id WHERE status = 'PAID' AND created_at > '2024-01-01' -- теж ні
Порядок колонок у composite index важливий. Правило: спочатку колонки з = фільтрами, потім з range (<, >, BETWEEN). Range колонка має бути останньою — після неї індекс вже не допомагає.

Порядок стовпців — практика

-- Запит: WHERE user_id = ? AND status = ? AND created_at > ? -- Оптимальний порядок: CREATE INDEX ON orders(user_id, status, created_at); -- user_id і status — equality (=), created_at — range (>) -- Range колонка останньою! -- Якщо порядок неправильний: CREATE INDEX ON orders(created_at, user_id, status); -- Запит WHERE user_id=42 AND status='PAID' AND created_at>... -- Не зможе ефективно використати цей індекс
covering

Covering index (покриваючий)

Якщо всі колонки запиту є в індексі — PostgreSQL не йде в heap таблиці (Index Only Scan). Найшвидший варіант.

-- Запит: SELECT status, total FROM orders WHERE user_id = 42 -- Звичайний індекс: Index Scan → heap lookup для status, total CREATE INDEX ON orders(user_id); -- Covering index: Index Only Scan — heap не потрібен! CREATE INDEX ON orders(user_id) INCLUDE (status, total); -- або включити в ключ (але впливає на сортування) CREATE INDEX ON orders(user_id, status, total); -- INCLUDE (PostgreSQL 11+): додає колонки в листову ноду -- без включення в ключ сортування CREATE INDEX ON orders(user_id, status) INCLUDE (total, created_at);
Covering index особливо цінний для часто виконуваних запитів. Перевір у EXPLAIN: Index Only Scan замість Index Scan — значить heap не читається.

EXPLAIN

EXPLAIN ANALYZE

-- EXPLAIN: план без виконання EXPLAIN SELECT * FROM orders WHERE user_id = 42; -- EXPLAIN ANALYZE: виконує і показує реальні цифри EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) SELECT * FROM orders WHERE user_id = 42; -- Приклад виводу: -- Index Scan using idx_orders_user_id on orders -- (cost=0.43..8.45 rows=5 width=200) -- (actual time=0.031..0.045 rows=5 loops=1) -- Index Cond: (user_id = 42) -- Buffers: shared hit=4 -- Planning Time: 0.5 ms -- Execution Time: 0.1 ms
Ключові вузли EXPLAIN: Seq Scan — sequential scan (може бути OK для малих таблиць), Index Scan — використовує індекс, Index Only Scan — covering index, Bitmap Heap Scan — для IN з багатьма значеннями, Nested Loop / Hash Join / Merge Join — join стратегії.

Читання cost

-- cost=0.43..8.45 rows=5 width=200 -- cost: startup_cost..total_cost -- startup_cost: час до першого рядку -- total_cost: оціночна вартість всього -- rows: оціночна кількість рядків -- width: середній розмір рядку в байтах -- Порівняй estimated rows vs actual rows -- Якщо сильно відрізняються → застаріла статистика -- Рішення: ANALYZE table_name
антипатерни

Коли індекс не використовується

-- 1. Функція над колонкою WHERE LOWER(email) = 'alice@ex.com' -- індекс на email не допомагає -- Рішення: functional index CREATE INDEX ON users (LOWER(email)); WHERE LOWER(email) = 'alice@ex.com' -- тепер використовує -- 2. Implicit type cast WHERE user_id = '42' -- user_id INT, '42' TEXT → cast → no index WHERE user_id = 42 -- правильно -- 3. OR між різними колонками WHERE user_id = 42 OR status = 'PAID' -- Рішення: UNION або окремі індекси + Bitmap OR -- 4. Negation WHERE status != 'DELETED' -- зазвичай Seq Scan (крім partial index) WHERE status NOT IN (...) -- 5. Low selectivity (мало унікальних значень) WHERE is_active = true -- якщо 90% рядків active → Seq Scan вигідніший -- PostgreSQL сам вирішує: якщо <~5% рядків → Index Scan -- 6. LIKE з leading wildcard WHERE name LIKE '%Alice%' -- no index WHERE name LIKE 'Alice%' -- B-tree допомагає!
overhead

Індекси і write overhead

Кожен індекс потребує оновлення при INSERT/UPDATE/DELETE. Забагато індексів → повільні writes.

-- Перевірити використання індексів SELECT schemaname, tablename, indexname, idx_scan, idx_tup_read, idx_tup_fetch FROM pg_stat_user_indexes WHERE schemaname = 'public' ORDER BY idx_scan ASC; -- idx_scan=0 → невикористаний індекс -- Розмір індексів SELECT indexname, pg_size_pretty(pg_relation_size(indexname::regclass)) FROM pg_indexes WHERE tablename = 'orders'; -- REINDEX: перебудувати пошкоджений або bloated індекс REINDEX INDEX CONCURRENTLY idx_orders_user_id; -- CREATE INDEX CONCURRENTLY: без блокування writes CREATE INDEX CONCURRENTLY idx_orders_status ON orders(status); -- Займає більше часу, але не блокує INSERT/UPDATE/DELETE
На співбесіді: B-tree = balanced tree O(log n), left-prefix rule для composite, covering index = Index Only Scan, EXPLAIN показує план і actual time, функції над колонкою ламають індекс → functional index.
