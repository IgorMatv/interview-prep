навіщо

IoC і Dependency Injection

Inversion of Control — контроль над створенням залежностей передається контейнеру. Dependency Injection — конкретна реалізація IoC: залежності "вприскуються" ззовні.

// Без IoC: клас сам створює залежності class RouteService { private RouteRepository repo = new JpaRouteRepository(); // жорстка прив'язка, не можна підмінити в тесті } // З IoC: Spring створює і вприскує @Service class RouteService { private final RouteRepository repo; RouteService(RouteRepository repo) { // constructor injection this.repo = repo; } }
Constructor injection — найкращий варіант: поля final, залежності явні, клас легко тестується без Spring контексту, неможливо створити об'єкт без залежностей.

Три способи DI

// 1. Constructor injection (рекомендовано) @Service class MyService { private final MyRepo repo; MyService(MyRepo repo) { this.repo = repo; } } // 2. Field injection (не рекомендовано) @Autowired private MyRepo repo; // приховані залежності, final неможливий // 3. Setter injection (для опціональних залежностей) @Autowired public void setRepo(MyRepo repo) { this.repo = repo; }

BeanFactory vs ApplicationContext

У Spring Boot завжди ApplicationContext. BeanFactory — базовий інтерфейс для розуміння архітектури, але не для прямого використання.

Реалізації ApplicationContext

// Spring Boot створює автоматично: AnnotationConfigServletWebServerApplicationContext // або для reactive: AnnotationConfigReactiveWebServerApplicationContext // Доступ до контексту @Autowired ApplicationContext ctx; ctx.getBean(MyService.class); ctx.getBeansOfType(MyInterface.class); ctx.getEnvironment().getProperty("app.name");

Способи визначення beans

// 1. Анотації на класі (найпоширеніше) @Component // загальний @Service // бізнес-логіка (семантика) @Repository // DAO, + exception translation @Controller // MVC контролер @RestController // = @Controller + @ResponseBody // 2. @Bean в @Configuration класі @Configuration class AppConfig { @Bean public RoutingStrategy routingStrategy(HaversineCalculator calc) { return new TwoOptStrategy(calc); } } // 3. XML (legacy, майже не використовується) // <bean id="myService" class="com.example.MyService"/> // @ComponentScan — де шукати @Component @SpringBootApplication // включає @ComponentScan для поточного пакету
@Component vs @Bean: @Component — клас є bean'ом (Spring сам викличе конструктор). @Bean — метод повертає bean (ти контролюєш створення). @Bean потрібен для third-party класів або складної ініціалізації.

2

3

4

5
@PostConstruct — ініціалізація після DI
6
InitializingBean.afterPropertiesSet()
7
@Bean(initMethod) — кастомний init
8
BeanPostProcessor.postProcessAfterInitialization ← AOP proxy тут
9
Bean готовий до використання
10
@PreDestroy при завершенні контексту
// Практичне використання lifecycle hooks @Component class CacheWarmer { @Autowired CacheService cache; @PostConstruct void warmUp() { cache.preload(); // виконується після DI, перед використанням } @PreDestroy void cleanup() { cache.clear(); // виконується при зупинці } }
@PostConstruct vs InitializingBean: @PostConstruct рекомендований — це JSR-250 стандарт, не прив'язаний до Spring. InitializingBean — Spring-specific. @Bean(initMethod) — для third-party класів.
BPP

BeanPostProcessor

Перехоплює кожен bean після створення. Може модифікувати або замінити bean. AOP proxy, @Autowired, @Value — все через BPP.

public interface BeanPostProcessor { // Перед @PostConstruct Object postProcessBeforeInitialization(Object bean, String name); // Після @PostConstruct — AOP proxy створюється тут Object postProcessAfterInitialization(Object bean, String name); } // Приклад: логуємо створення всіх beans @Component class LoggingBPP implements BeanPostProcessor { public Object postProcessAfterInitialization(Object bean, String name) { log.debug("Bean created: {}", name); return bean; // повертаємо той самий або замінений } }
Вбудовані BPP у Spring: AutowiredAnnotationBeanPostProcessor (обробляє @Autowired), CommonAnnotationBPP (@PostConstruct, @PreDestroy), AbstractAdvisorAutoProxyCreator (AOP proxy для @Transactional тощо).
BFPP

BeanFactoryPostProcessor

Виконується до створення beans — після читання bean definitions. Може модифікувати самі definitions. PropertySourcesPlaceholderConfigurer — класичний приклад: підставляє ${prop} значення.

public interface BeanFactoryPostProcessor { void postProcessBeanFactory(ConfigurableListableBeanFactory factory); } // Порядок виконання: // 1. BeanDefinition читаються (XML, @Component scan, @Bean) // 2. BeanFactoryPostProcessor спрацьовує ← тут // 3. Beans створюються // 4. BeanPostProcessor спрацьовує // PropertySourcesPlaceholderConfigurer замінює: @Value("${app.name}") String name; // на реальне значення з application.properties
BPP ≠ BFPP. BeanPostProcessor — обробляє готові bean інстанси (після створення). BeanFactoryPostProcessor — обробляє bean definitions (до створення). Це найчастіша плутанина на співбесіді.
антипатерни

Типові помилки і нюанси

1. Field injection замість constructor injection

// ПОГАНО: приховані залежності, неможливий final @Autowired private MyService service; // ДОБРЕ private final MyService service; MyBean(MyService service) { this.service = service; }

2. Circular dependency

// A → B → A → StackOverflowError або BeanCurrentlyInCreationException // З constructor injection виявляється одразу при старті // Рішення: рефакторинг, @Lazy, або переглянути дизайн @Lazy @Autowired ServiceB b; // proxy-заглушка до першого використання

3. ApplicationContext в бізнес-логіці

// ПОГАНО: service залежить від контейнера @Autowired ApplicationContext ctx; void doWork() { ctx.getBean(SomeService.class).call(); } // Легітимний кейс: self-invocation обхід (але краще рефакторинг)
На співбесіді: BPP vs BFPP (instance vs definition), lifecycle послідовність (@PostConstruct між BPP before і after), чому constructor injection кращий за field, AOP proxy створюється в postProcessAfterInitialization.
