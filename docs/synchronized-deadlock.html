monitor

Monitor і intrinsic lock

Кожен Java об'єкт має вбудований lock (intrinsic/monitor lock). synchronized захоплює цей lock — тільки один потік може тримати його одночасно.

// Що гарантує synchronized: // 1. Mutual exclusion — тільки 1 потік в блоці // 2. Visibility — бачить всі зміни до unlock попереднього власника // 3. Ordering — happens-before між unlock і lock // Reentrant: той самий потік може захопити лок знову synchronized(this) { synchronized(this) { // ОК — не deadlock doWork(); } }
Intrinsic lock — reentrant: потік що вже тримає lock може захопити його ще раз без блокування. Лічильник входжень зростає, lock звільняється коли лічильник дійде до 0.
синтаксис

Варіанти synchronized

// 1. Instance method — lock на this public synchronized void increment() { count++; } // еквівалент: public void increment() { synchronized(this) { count++; } } // 2. Static method — lock на Class object public static synchronized void reset() { staticCount = 0; } // еквівалент: synchronized(MyClass.class) { staticCount = 0; } // 3. Block — мінімальний критичний розділ (краще) private final Object lock = new Object(); // приватний lock public void update() { prepare(); // поза локом synchronized(lock) { apply(); } // тільки потрібне cleanup(); // поза локом }
Синхронізація на this або Class — небезпечна: зовнішній код може захопити той самий lock. Приватний final Object lock — єдиний безпечний варіант.
координація

wait() / notify() / notifyAll()

Механізм координації між потоками. Тільки всередині synchronized блоку — інакше IllegalMonitorStateException.

// Правильний патерн wait(): synchronized(lock) { while (!condition()) { // while, не if! lock.wait(); // звільняє lock і чекає } doWork(); } // Повідомити один потік synchronized(lock) { setCondition(true); lock.notify(); // будить один (довільний) } // Повідомити всі потоки synchronized(lock) { setCondition(true); lock.notifyAll(); // будить всіх }
Завжди while, не if. Spurious wakeups — потік може прокинутись без notify(). while (!condition()) захищає від цього.
У сучасному коді wait/notify майже не використовується — є ReentrantLock + Condition, BlockingQueue, CountDownLatch, Semaphore.
deadlock

Deadlock

Deadlock виникає коли два або більше потоки чекають на локи які тримають один одного. Чотири необхідні умови (Coffman):

  • Mutual exclusion — ресурс тримає тільки один потік
  • Hold and wait — тримає один lock і чекає іншого
  • No preemption — lock не можна забрати примусово
  • Circular wait — A чекає B, B чекає A
// Класичний deadlock Object lockA = new Object(), lockB = new Object(); // Thread 1 synchronized(lockA) { sleep(10); synchronized(lockB) { ... } // чекає lockB } // Thread 2 synchronized(lockB) { sleep(10); synchronized(lockA) { ... } // чекає lockA } // Thread 1 тримає A, чекає B // Thread 2 тримає B, чекає A — deadlock!

Три способи уникнути deadlock

// 1. Фіксований порядок захоплення локів // Завжди спочатку lockA, потім lockB synchronized(lockA) { synchronized(lockB) { ... } } // 2. tryLock() з таймаутом (ReentrantLock) if (lockA.tryLock(100, TimeUnit.MS)) { try { if (lockB.tryLock(100, TimeUnit.MS)) { try { doWork(); } finally { lockB.unlock(); } } } finally { lockA.unlock(); } } // 3. Уникати вкладених локів взагалі // Захоплювати тільки один lock за раз
Знайти deadlock: jstack <pid> або VisualVM покаже "found one Java-level deadlock" з трасуванням потоків.
суміжні проблеми

Livelock і starvation

Livelock

Потоки активні (не заблоковані) але не роблять корисної роботи — постійно реагують один на одного.

// Аналогія: два люди в коридорі // кожен поступається, зміщуючись в одну сторону, // і блокують один одного нескінченно // У коді: два потоки відступають при конфлікті // і постійно потрапляють в одну ситуацію

Starvation

Потік постійно не отримує доступ до ресурсу бо інші потоки з вищим пріоритетом завжди його займають.

// Причини starvation: // - unfair lock (notify() будить довільний потік) // - Thread.setPriority() — низький пріоритет // - довгі synchronized блоки // Рішення: fair lock new ReentrantLock(true); // fair=true — FIFO порядок
race condition

Race condition

Результат залежить від відносного порядку виконання операцій кількома потоками. Непередбачуваний і важко відтворюваний баг.

// Check-then-act race condition if (!map.containsKey(key)) { // Thread 1 перевіряє // Thread 2 теж перевіряє — true map.put(key, value); // обидва кладуть! } // Рішення: атомарна операція map.putIfAbsent(key, value); // ConcurrentHashMap map.computeIfAbsent(key, k -> expensive()); // lazy init // Read-modify-write race condition counter++; // read(0), read(0), write(1), write(1) → 1 замість 2 // Рішення synchronized(this) { counter++; } // або atomicCounter.incrementAndGet();
антипатерни

Типові помилки

1. Синхронізація на змінному об'єкті

// ПОГАНО: якщо str перепризначити — різні локи! String lock = "lock"; synchronized(lock) { ... } // ДОБРЕ: final, не може змінитись private final Object lock = new Object();

2. Синхронізація на String literal або Integer

// ПОГАНО: String literals — interned, спільний між класами synchronized("LOCK") { ... } // небезпечно! // ПОГАНО: Integer кешується для -128..127 Integer i = 42; synchronized(i) { ... } // той самий об'єкт у всьому JVM!

3. if замість while у wait()

// ПОГАНО: spurious wakeup обходить перевірку synchronized(lock) { if (!ready) lock.wait(); // spurious wake → продовжує без ready! process(); } // ДОБРЕ synchronized(lock) { while (!ready) lock.wait(); process(); }
На співбесіді: 4 умови deadlock, фіксований порядок локів як prevention, різниця deadlock/livelock/starvation, check-then-act race condition.
