контекст

Навіщо Optional

Optional — контейнер що може містити або значення, або нічого. Мета: явно показати в сигнатурі методу що результат може бути відсутнім, і змусити caller обробити обидва випадки.

до Optional

// повертає null — неявно User findById(long id) { return db.get(id); // null? } // caller може забути перевірити user.getName(); // NPE!

з Optional

// явно в сигнатурі Optional<User> findById(long id) { return Optional.ofNullable(db.get(id)); } // компілятор змушує обробити findById(1).ifPresent(u -> ...);
Optional призначений тільки для return type методів. Не для полів, не для параметрів, не для колекцій — це антипатерни з конкретними наслідками.

Три способи створити Optional

Optional.of(value) // NPE якщо value == null Optional.ofNullable(value) // safe — null → empty Optional.empty() // порожній // of vs ofNullable Optional.of(null); // NullPointerException! Optional.ofNullable(null); // Optional.empty()
API

Всі методи Optional

// Перевірка opt.isPresent() // true якщо є значення opt.isEmpty() // Java 11+, !isPresent() // Отримання (небезпечні) opt.get() // NoSuchElementException якщо empty opt.orElse(defaultVal) // значення або default opt.orElseGet(() -> calc()) // lazy default opt.orElseThrow() // Java 10+, або з supplier opt.orElseThrow(NotFoundException::new) // Трансформація opt.map(u -> u.getName()) // Optional<String> opt.flatMap(u -> u.getAddr()) // якщо getAddr() → Optional opt.filter(u -> u.isActive()) // Optional<User> або empty // Side effects opt.ifPresent(u -> log(u)) opt.ifPresentOrElse( // Java 9+ u -> process(u), () -> handleEmpty()) // Stream (Java 9+) opt.stream() // Stream з 0 або 1 елементом // Fallback Optional (Java 9+) opt.or(() -> fallback()) // Optional якщо empty
антипатерн

get() без перевірки

Optional.get() кидає NoSuchElementException якщо Optional порожній. Викликати його без попередньої перевірки — той самий ризик що й разіменування null.

погано

Optional<User> opt = findById(id); // NoSuchElementException якщо empty! String name = opt.get().getName();

добре

// явно кидаємо свій exception User user = findById(id) .orElseThrow(() -> new UserNotFoundException(id));
// Правильні варіанти замість get() // 1. orElseThrow — якщо відсутність = помилка User user = opt.orElseThrow(UserNotFoundException::new); // 2. orElse — якщо є розумне значення за замовчуванням User user = opt.orElse(User.anonymous()); // 3. ifPresent — якщо дія тільки при наявності opt.ifPresent(u -> sendEmail(u)); // 4. map + orElse — трансформація з дефолтом String name = opt.map(User::getName).orElse("Unknown");
Якщо бачиш opt.isPresent() + opt.get() — це code smell. Завжди можна замінити на orElse, orElseThrow або ifPresent.
антипатерн

Optional як поле класу

Optional не реалізує Serializable. Якщо клас з Optional-полем серіалізувати (JPA entity, Kafka message, HTTP session) — NotSerializableException.

погано

@Entity class User { // Not Serializable! private Optional<String> phone; }

добре

@Entity class User { // nullable поле @Nullable private String phone; // Optional тільки в getter public Optional<String> getPhone() { return Optional.ofNullable(phone); } }
Правило: зберігай null, повертай Optional. Поле завжди nullable, getter обгортає в Optional. Так і серіалізація працює, і API зрозуміле.

Ще одна проблема: пам'ять

Кожен Optional — окремий об'єкт у heap (~16 байт + посилання). Для мільйонів об'єктів з Optional-полями це помітний overhead. Nullable поле — нульовий overhead.

Optional в колекціях

List<Optional<T>> або Map<K, Optional<V>> — надмірне ускладнення. Колекція вже може бути порожньою — це і є "відсутність значень".

// ПОГАНО: List<Optional<String>> List<Optional<String>> names = users.stream() .map(u -> Optional.ofNullable(u.getName())) .collect(Collectors.toList()); // ДОБРЕ: відфільтруй null одразу List<String> names = users.stream() .map(User::getName) .filter(Objects::nonNull) .collect(Collectors.toList()); // або через Optional.stream() (Java 9+) List<String> names = users.stream() .map(User::getOptionalName) // → Optional<String> .flatMap(Optional::stream) // розгортаємо в Stream<String> .collect(Collectors.toList());
Optional::stream (Java 9+) — єдиний гарний спосіб з'єднати Optional і Stream: перетворює Stream<Optional<T>> на Stream<T> через flatMap.
антипатерн

Optional як параметр методу

Optional як параметр змушує caller завжди обгортати значення — навіть коли воно точно є. Плюс caller може передати null замість Optional.empty() — і NPE все одно можливий.

погано

void createUser( String name, Optional<String> phone) { // caller мусить: createUser("Alice", Optional.of("+380...")); createUser("Bob", Optional.empty()); }

добре — overloading

void createUser(String name) { createUser(name, null); } void createUser( String name, String phone) { // phone може бути null }
Альтернативи Optional-параметру: overloading методів, @Nullable анотація (Lombok/JetBrains), Builder pattern для багатьох опціональних параметрів.
нюанс

orElse vs orElseGet — найчастіше питання

orElse(value) — аргумент обчислюється завжди, навіть якщо Optional не порожній. orElseGet(supplier) — supplier викликається тільки якщо empty.

// orElse: createDefault() викликається ЗАВЖДИ User user = findById(id).orElse(createDefault()); // якщо findById повернув значення — createDefault() все одно викликано! // orElseGet: createDefault() тільки якщо empty User user = findById(id).orElseGet(() -> createDefault());
Практичне правило: якщо default — це простий літерал або вже існуючий об'єкт → orElse. Якщо default потребує обчислення, IO або створення нового об'єкту → orElseGet.
// ОК: константа, нема overhead opt.orElse("Unknown") opt.orElse(Collections.emptyList()) // ОБЕРЕЖНО: DB запит виконається завжди! opt.orElse(userRepository.findDefault()) // ← погано opt.orElseGet(userRepository::findDefault) // ← добре // ОБЕРЕЖНО: new об'єкт завжди opt.orElse(new User()) // ← непотрібна алокація opt.orElseGet(User::new) // ← тільки якщо треба
правильне використання

Chaining: map, flatMap, filter

Сила Optional — можливість будувати pipeline без explicit перевірок на null. Якщо будь-який крок повертає empty — весь ланцюжок повертає empty.

// Без Optional — пірамідний null-check String city = null; User user = findById(id); if (user != null) { Address addr = user.getAddress(); if (addr != null) { city = addr.getCity(); } } // З Optional — лінійний ланцюжок String city = findById(id) .map(User::getAddress) // Optional<Address> .map(Address::getCity) // Optional<String> .orElse("Unknown");

flatMap — коли метод повертає Optional

// getAddress() повертає Optional<Address> // map дав би Optional<Optional<Address>> — не те! findById(id) .flatMap(User::getAddress) // розгортає вкладений Optional .map(Address::getCity) .orElse("Unknown");
На співбесіді: orElse vs orElseGet (lazy evaluation) — найчастіше питання. Потім: чому не поле, не параметр, не в колекцію. І завжди — що Optional не замінює всі null, а тільки return type.
