Стани потоку (Thread.State)

NEW → start() → RUNNABLE
RUNNABLE → BLOCKED → lock acquired → RUNNABLE
RUNNABLE → WAITING → notify/join → RUNNABLE
RUNNABLE → TIMED_WAITING → timeout → RUNNABLE
RUNNABLE → run() returns → TERMINATED

Що переводить у кожен стан

СтанЩо викликає
NEWnew Thread() — ще не запущений
RUNNABLEthread.start(), повернення з BLOCKED/WAITING
BLOCKEDчекає на synchronized lock зайнятий іншим потоком

RUNNABLE ≠ реально виконується. JVM позначає потік RUNNABLE як тільки він готовий — але OS scheduler вирішує коли насправді дати йому CPU. Потік може бути RUNNABLE і чекати на CPU.
створення

Способи створити потік

// 1. Extends Thread (не рекомендується) class MyThread extends Thread { @Override public void run() { System.out.println("running"); } } new MyThread().start(); // 2. Implements Runnable (кращий варіант) Thread t = new Thread(() -> System.out.println("running")); t.start(); // 3. ExecutorService (правильний варіант у продакшн) ExecutorService exec = Executors.newFixedThreadPool(4); exec.submit(() -> System.out.println("running")); // 4. Virtual Thread (Java 21) Thread.ofVirtual().start(() -> System.out.println("virtual"));
Ніколи не викликай run() напряму — це просто виклик методу в поточному потоці. Тільки start() створює новий потік. Це класичне питання-пастка на співбесіді.

interrupt() — кооперативне зупинення

// interrupt() встановлює прапор — не зупиняє одразу Thread t = new Thread(() -> { while (!Thread.currentThread().isInterrupted()) { doWork(); } // cleanup }); t.interrupt(); // встановлює interrupted flag // sleep/wait кидають InterruptedException при interrupted try { Thread.sleep(1000); } catch (InterruptedException e) { Thread.currentThread().interrupt(); // відновлюємо прапор! return; }
daemon

Daemon threads

JVM завершується коли всі non-daemon потоки завершились. Daemon потоки — фонові сервіси що не блокують завершення JVM. GC, finalizer — daemon.

Thread t = new Thread(() -> { while (true) { monitorMetrics(); sleep(1000); } }); t.setDaemon(true); // до start()! t.start(); // JVM завершиться не чекаючи цього потоку // Virtual threads — завжди daemon Thread.ofVirtual().start(task); // daemon by default
setDaemon() треба викликати до start(). Після запуску — IllegalThreadStateException.
JMM

Visibility проблема

Кожен потік може кешувати змінні у своєму CPU cache або регістрах. Без явної синхронізації один потік може не побачити зміни зроблені іншим.

// Класичний приклад: нескінченний цикл! class StopTask implements Runnable { private boolean stopped = false; // не volatile! public void stop() { stopped = true; } @Override public void run() { while (!stopped) { // може читати кешоване false вічно doWork(); } } } // Потік може ніколи не побачити stopped = true! // JIT може навіть оптимізувати цикл в while(true)
Це не баг JVM — це специфікація. CPU і компілятор мають право переупорядковувати і кешувати операції для оптимізації. Java Memory Model визначає коли зміни гарантовано видимі.
volatile

volatile

Гарантує два ефекти: visibility (запис видимий усім потокам одразу) і ordering (заборона переупорядкування через memory barrier).

class StopTask implements Runnable { private volatile boolean stopped = false; public void stop() { stopped = true; } // запис у main memory @Override public void run() { while (!stopped) { // кожен раз читає з main memory doWork(); } } }

Коли volatile достатньо: один потік пише, інші читають. Або прапори (boolean). Якщо кілька потоків пишуть — потрібен synchronized або Atomic*.
JMM

happens-before

Happens-before — відношення між операціями. Якщо A happens-before B, то всі ефекти A видимі B. Це єдиний спосіб гарантувати visibility між потоками.

// Правила happens-before в Java: // 1. Program order: в одному потоці // кожна операція happens-before наступної // 2. Monitor lock: unlock happens-before lock того ж монітора synchronized(lock) { x = 1; } // unlock synchronized(lock) { y = x; } // lock — бачить x=1 // 3. volatile: запис volatile happens-before читання volatile int v; v = 42; // write int r = v; // read — гарантовано бачить 42 // 4. Thread start: все до start() happens-before run() data = prepare(); thread.start(); // thread бачить data // 5. Thread join: run() happens-before join() повертає thread.join(); // ми бачимо все що thread зробив readResult(); // 6. Transitivity: A hb B, B hb C → A hb C

Практичний приклад: safe publication

// БЕЗ happens-before: небезпечна публікація Object obj; // Thread 1: obj = new Object(); // Thread 2 може побачити частково ініціалізований! // З volatile: safe publication volatile Object obj; obj = new Object(); // запис volatile — HB для читачів // З synchronized: safe publication synchronized(this) { obj = new Object(); } // final поля — безпечні без синхронізації після конструктора class Immutable { final int x; Immutable(int x) { this.x = x; } // safe після return з ctor }
антипатерни

Типові помилки

1. run() замість start()

// ПОГАНО: виконується в поточному потоці new Thread(task).run(); // ДОБРЕ new Thread(task).start();

2. volatile для non-atomic compound ops

// ПОГАНО: volatile не захищає ++ volatile int count = 0; count++; // race condition між потоками // ДОБРЕ AtomicInteger count = new AtomicInteger(); count.incrementAndGet();

3. Проковтнути InterruptedException

// ПОГАНО: прапор скинуто, вищий код не дізнається try { Thread.sleep(1000); } catch (InterruptedException e) { /* ignore */ } // ДОБРЕ: відновлюємо прапор try { Thread.sleep(1000); } catch (InterruptedException e) { Thread.currentThread().interrupt(); return; }
