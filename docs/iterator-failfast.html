основи

Що таке Iterator

Iterator — патерн для обходу колекції без знання її внутрішньої структури. В Java кожна колекція що реалізує Iterable повертає свій Iterator через iterator(). Саме його використовує for-each під капотом.

interface Iterator<E> { boolean hasNext(); E next(); default void remove() { // опціональний throw new UnsupportedOperationException(); } } // for-each розкривається в: Iterator<String> it = list.iterator(); while (it.hasNext()) { String s = it.next(); }

Дві стратегії при модифікації під час ітерації

fail-fast
кидає ConcurrentModificationException одразу
fail-safe
працює на snapshot або tolerate changes
Назва "fail-safe" неофіційна — в Javadoc її немає. Офіційні терміни: weakly consistent (ConcurrentHashMap) і snapshot (CopyOnWriteArrayList).
fail-fast

Fail-fast Iterator

ArrayList, LinkedList, HashMap, HashSet та більшість колекцій з java.util. Зберігають лічильник modCount — скільки разів структура колекції змінювалась. При кожному next() порівнюють з expectedModCount.

// Спрощено: ArrayList.Itr private class Itr implements Iterator<E> { int expectedModCount = modCount; // snapshot при створенні public E next() { if (modCount != expectedModCount) throw new ConcurrentModificationException(); // ... повертає елемент } }

Що кидає CME

List<String> list = new ArrayList<>(List.of("a", "b", "c")); // ConcurrentModificationException! for (String s : list) { if (s.equals("b")) list.remove(s); // модифікуємо список під час ітерації } // Також CME при паралельній модифікації з іншого потоку // (але не гарантовано — best-effort detection)
CME — не thread-safety механізм. Це best-effort детекція помилки програмування. При реальному concurrent доступі CME може не спрацювати.

Колекції з fail-fast iterator

  • ArrayList, LinkedList, ArrayDeque
  • HashMap, LinkedHashMap, HashSet, LinkedHashSet
  • TreeMap, TreeSet
  • PriorityQueue
fail-safe

Fail-safe / Weakly consistent Iterator

Колекції з java.util.concurrent. Два механізми: snapshot (CopyOnWriteArrayList) і weakly consistent (ConcurrentHashMap).

// CopyOnWriteArrayList — snapshot підхід CopyOnWriteArrayList<String> cowList = new CopyOnWriteArrayList<>(List.of("a", "b", "c")); for (String s : cowList) { cowList.add("d"); // ОК — не кидає CME // але "d" НЕ буде в цій ітерації (snapshot) } // ConcurrentHashMap — weakly consistent ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>(); map.put("a", 1); map.put("b", 2); for (var entry : map.entrySet()) { map.put("c", 3); // може або не може з'явитись в ітерації }

практика

Безпечне видалення під час ітерації

Якщо треба видалити елементи під час обходу звичайного ArrayList — є кілька безпечних способів.

List<String> list = new ArrayList<>(List.of("a", "bb", "c", "dd")); // ✓ Спосіб 1: Iterator.remove() Iterator<String> it = list.iterator(); while (it.hasNext()) { if (it.next().length() > 1) it.remove(); // оновлює expectedModCount } // ✓ Спосіб 2: removeIf() (Java 8+) — найчистіший list.removeIf(s -> s.length() > 1); // ✓ Спосіб 3: collect в новий список list = list.stream() .filter(s -> s.length() == 1) .collect(Collectors.toList()); // ✓ Спосіб 4: ListIterator для заміни ListIterator<String> lit = list.listIterator(); while (lit.hasNext()) { if (lit.next().equals("a")) lit.set("A"); // set — без CME }
Рекомендація: у продакшн коді завжди використовуй removeIf() — це найчитабельніше і найбезпечніше рішення.
immutability

unmodifiableList vs List.of

Обидва повертають "незмінний" список, але по-різному. Це одне з найчастіших питань на співбесіді.

Collections.unmodifiableList()

List<String> mutable = new ArrayList<>(List.of("a","b")); List<String> view = Collections.unmodifiableList(mutable); view.add("c"); // UOE ✓ mutable.add("c"); // ОК! // view тепер ["a","b","c"] ← змінився!

List.of() / List.copyOf()

List<String> immutable = List.of("a", "b"); immutable.add("c"); // UOE ✓ // Жодного способу змінити // Справді immutable // null елементи — NPE

List.copyOf(collection) — найбезпечніший варіант коли хочеш справді незмінну копію. Робить defensive copy і гарантує що оригінал не може вплинути на результат.
// Типовий патерн: defensive copy в конструкторі class ImmutableContainer { private final List<String> items; ImmutableContainer(List<String> items) { this.items = List.copyOf(items); // копія + immutable } public List<String> getItems() { return items; // вже immutable, безпечно повертати } }
Collections API

Корисні Collections утиліти

// Сортування і пошук Collections.sort(list); // natural order Collections.sort(list, comparator); // custom order Collections.binarySearch(sortedList, key); // O(log n) Collections.reverse(list); Collections.shuffle(list); // Min / Max Collections.min(collection); Collections.max(collection, comparator); // Frequency і disjoint Collections.frequency(list, element); // скільки разів Collections.disjoint(list1, list2); // немає спільних? // Заповнення Collections.fill(list, "default"); Collections.nCopies(5, "x"); // [x, x, x, x, x] // Singleton і empty (immutable) Collections.singletonList("only"); Collections.emptyList(); // краще List.of() // Thread-safe обгортки (краще ConcurrentHashMap) Collections.synchronizedList(list); Collections.synchronizedMap(map);
Collections.synchronizedList() синхронізує кожну операцію окремо, але не ітерацію — for-each треба синхронізувати вручну. Зазвичай краще використовувати CopyOnWriteArrayList або ConcurrentHashMap.
антипатерни

Типові помилки

1. Модифікація в for-each

// ПОГАНО: ConcurrentModificationException for (String s : list) { if (condition(s)) list.remove(s); } // ДОБРЕ list.removeIf(s -> condition(s));

2. unmodifiableList і мутація оригіналу

// ПАСТКА: view, не копія var original = new ArrayList<>(List.of("a", "b")); var view = Collections.unmodifiableList(original); original.add("c"); view.size(); // → 3 ← несподівано! // ПРАВИЛЬНО: справжня копія var safe = List.copyOf(original);

3. null в List.of()

// NPE при створенні! List.of("a", null, "b"); // NullPointerException // Якщо null можливий — ArrayList або перевірка var list = new ArrayList<>(); list.add("a"); list.add(null); // ОК
На співбесіді: ключова різниця — unmodifiableList це view (оригінал може змінитись), List.of — справді immutable структура. І CME — це best-effort, не thread-safety гарантія.
