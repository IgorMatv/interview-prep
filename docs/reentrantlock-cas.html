explicit lock

ReentrantLock

Явний lock з більшим контролем ніж synchronized: tryLock з таймаутом, fair режим, Condition, interruptible lock.

private final ReentrantLock lock = new ReentrantLock(); // Базове використання — завжди unlock у finally lock.lock(); try { doWork(); } finally { lock.unlock(); // обов'язково! } // tryLock — не блокує якщо lock зайнятий if (lock.tryLock()) { try { doWork(); } finally { lock.unlock(); } } else { fallback(); } // tryLock з таймаутом — для уникнення deadlock if (lock.tryLock(100, TimeUnit.MILLISECONDS)) { ... }

Condition — заміна wait/notify

private final ReentrantLock lock = new ReentrantLock(); private final Condition notEmpty = lock.newCondition(); private final Condition notFull = lock.newCondition(); // Producer lock.lock(); try { while (isFull()) notFull.await(); add(item); notEmpty.signal(); // тільки consumers } finally { lock.unlock(); } // Consumer lock.lock(); try { while (isEmpty()) notEmpty.await(); take(); notFull.signal(); // тільки producers } finally { lock.unlock(); }
Перевага над wait/notify: можна мати кілька Condition на один lock і сигналізувати тільки потрібній групі потоків.
read-write

ReentrantReadWriteLock

Два логічних локи на одному: read lock (кілька читачів одночасно) і write lock (ексклюзивний). Ідеально для read-heavy сценаріїв.

private final ReadWriteLock rwLock = new ReentrantReadWriteLock(); private final Lock readLock = rwLock.readLock(); private final Lock writeLock = rwLock.writeLock(); private Map<String, Data> cache = new HashMap<>(); public Data get(String key) { readLock.lock(); // багато потоків можуть читати try { return cache.get(key); } finally { readLock.unlock(); } } public void put(String key, Data data) { writeLock.lock(); // ексклюзивний доступ try { cache.put(key, data); } finally { writeLock.unlock(); } }
При write-heavy навантаженні ReadWriteLock може бути повільнішим за synchronized через overhead координації. Вимірюй перед використанням.
optimistic

StampedLock (Java 8+)

Три режими: write lock, read lock, і optimistic read — читає без захоплення lock, потім перевіряє чи не змінились дані. Якщо так — повторно з read lock.

private final StampedLock sl = new StampedLock(); private double x, y; // Optimistic read — ідеально для read-heavy double distanceFromOrigin() { long stamp = sl.tryOptimisticRead(); // без блокування double cx = x, cy = y; // читаємо if (!sl.validate(stamp)) { // чи не змінилось? stamp = sl.readLock(); // ні → повторюємо з lock try { cx = x; cy = y; } finally { sl.unlockRead(stamp); } } return Math.sqrt(cx*cx + cy*cy); } // Write lock void move(double dx, double dy) { long stamp = sl.writeLock(); try { x += dx; y += dy; } finally { sl.unlockWrite(stamp); } }
StampedLock не reentrant. Якщо потік захопив write lock і спробує захопити read lock — deadlock. Не підходить для рекурсивних сценаріїв.
lock-free

CAS — Compare-And-Swap

Атомарна CPU інструкція: "якщо значення в пам'яті = expected, запиши new value". Основа lock-free структур. В Java — через Unsafe або VarHandle.

// Псевдокод CAS (атомарно на рівні CPU): boolean compareAndSet(int expected, int newValue) { if (current == expected) { current = newValue; return true; } return false; // хтось змінив між тим — retry } // Типовий CAS loop (optimistic concurrency) int oldValue, newValue; do { oldValue = atomicInt.get(); newValue = transform(oldValue); } while (!atomicInt.compareAndSet(oldValue, newValue));

ABA проблема

CAS бачить що значення = expected (A), але між read і write воно було змінено A → B → A. Операція проходить хибно успішно.

// Рішення: AtomicStampedReference — додає version counter AtomicStampedReference<Integer> ref = new AtomicStampedReference<>(0, 0); int[] stampHolder = new int[1]; Integer val = ref.get(stampHolder); // читаємо з версією int stamp = stampHolder[0]; ref.compareAndSet(val, newVal, stamp, stamp + 1); // CAS пройде тільки якщо і значення і версія збіглись

AtomicInteger та інші

// AtomicInteger — найчастіше використовується AtomicInteger ai = new AtomicInteger(0); ai.get() // читання ai.set(42) // запис ai.getAndIncrement() // i++ (повертає старе) ai.incrementAndGet() // ++i (повертає нове) ai.addAndGet(5) // += 5 ai.compareAndSet(0, 1) // CAS ai.updateAndGet(x -> x * 2) // lambda (Java 8+) ai.accumulateAndGet(5, Integer::sum) // AtomicLong, AtomicBoolean — аналогічно // AtomicReference — для об'єктів AtomicReference<String> ref = new AtomicReference<>("init"); ref.compareAndSet("init", "updated"); // AtomicIntegerArray — масив атомарних int AtomicIntegerArray arr = new AtomicIntegerArray(10); arr.incrementAndGet(3); // атомарний ++ для елемента 3
high throughput

LongAdder vs AtomicLong

При сильній конкуренції на лічильнику AtomicLong витрачає багато часу на retry. LongAdder розподіляє запис по кількох cell, сумує при читанні.

// AtomicLong: один лічильник, contention при конкуренції AtomicLong counter = new AtomicLong(); counter.incrementAndGet(); // CAS retry при конкуренції // LongAdder: stripe per thread, менше contention LongAdder adder = new LongAdder(); adder.increment(); // майже без contention adder.sum(); // читання — сумує всі cells // LongAccumulator — для довільних операцій LongAccumulator max = new LongAccumulator(Long::max, Long.MIN_VALUE); max.accumulate(42); max.get();
Коли що: AtomicLong — коли потрібен точний поточний стан (CAS чи read after write). LongAdder — коли просто накопичуєш статистику з багатьох потоків.
