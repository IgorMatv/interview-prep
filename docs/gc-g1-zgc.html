основи

Stop-The-World і паузи GC

Stop-The-World (STW) — всі потоки додатку зупиняються поки GC виконує роботу. Мета сучасних GC: мінімізувати тривалість STW пауз.

// Три виміри GC: // Throughput: % часу додатку (не GC) // Latency: тривалість STW пауз // Footprint: обсяг пам'яті для GC структур // Не можна мати все — trade-off: // Serial/Parallel: high throughput, high latency // G1: balanced, predictable latency // ZGC/Shenandoah: ultra-low latency, lower throughput
Для більшості веб-сервісів важлива latency (час відповіді), тому G1 або ZGC. Для batch jobs — throughput, тому Parallel GC.

Алгоритми GC: mark, sweep, compact, copy

  • Mark: позначаємо живі об'єкти від GC roots
  • Sweep: звільняємо пам'ять мертвих об'єктів (фрагментація)
  • Compact: зсуваємо живі об'єкти разом (без фрагментації, але повільно)
  • Copy: копіюємо живих у новий регіон (Young Gen використовує це)

CMS видалено в Java 14. Якщо бачиш його у legacy конфігах — треба мігрувати на G1 або ZGC.
G1 GC

G1 — regions замість суцільних зон

G1 (Garbage-First) — default з Java 9. Heap поділений на рівні regions (~1-32MB кожен). Regions динамічно призначаються ролі: Eden, Survivor, Old, Humongous.

// G1 flags -XX:+UseG1GC // explicit (default Java 9+) -XX:MaxGCPauseMillis=200 // target pause (не гарантія) -XX:G1HeapRegionSize=4m // розмір region (1-32MB) -XX:G1NewSizePercent=5 // мін % Young Gen -XX:G1MaxNewSizePercent=60 // макс % Young Gen -XX:InitiatingHeapOccupancyPercent=45 // IHOP: поріг для concurrent marking
Ключова ідея G1: він вибирає для збору regions з найбільшою кількістю сміття (Garbage-First). STW фаза займається тільки вибраними regions, а не всім heap одразу.

Remembered Sets (RSets)

G1 підтримує RSets — для кожного region зберігає посилання з інших regions. Це дозволяє збирати окремий region без сканування всього heap.

// RSets overhead: ~1-20% heap // При великих heap може бути значним // G1RSetSparseRegionEntries, G1RSetRegionEntries — тюнінг
G1 GC

G1 — цикл збору

// 1. Young-only GC (часто) // STW: evacuate Eden + Survivor → нові Survivor/Old regions // Пауза: зазвичай <50ms // 2. Concurrent Marking Cycle (при IHOP %) // Initial Mark: STW, позначити root objects // Root Region Scan: concurrent, сканувати Survivor // Concurrent Mark: concurrent, обійти heap (без STW!) // Remark: STW, завершити marking // Cleanup: STW, оновити RSet, вибрати пусті regions // 3. Mixed GC (після marking) // STW: evacuate Young + вибрані Old regions // Продовжується поки Old не стане достатньо чистим // 4. Full GC (fallback, уникати!) // Serial STW, mark+compact всього heap // Відбувається при to-space exhaustion або metadata OOM
Full GC в G1 — найгірший сценарій. Симптом: "Pause Full" в GC логах. Причини: to-space exhaustion (немає місця для evacuation), concurrent marking не встигає.
humongous

Humongous objects

Об'єкти розміром > 50% розміру region — Humongous. Розміщуються в окремих Humongous regions, обробляються інакше.

// При G1HeapRegionSize=4MB: // Humongous threshold: > 2MB // Великий масив, великий StringBuilder — humongous // Проблеми з Humongous: // - Займають кілька суміжних regions // - Можуть викликати Full GC якщо немає суміжних regions // - Не переміщуються (non-movable) до Java 9 // - Збираються тільки при concurrent marking або Full GC // Виявити Humongous allocation в логах: // [GC pause (G1 Humongous Allocation)...] // Рішення: збільшити G1HeapRegionSize -XX:G1HeapRegionSize=16m // якщо є великі об'єкти // або уникати великих short-lived об'єктів
ZGC

ZGC — sub-millisecond паузи

Scalable low-latency GC. Stable з Java 15. Паузи <1ms незалежно від розміру heap (навіть терабайти). Concurrent mark + concurrent relocation.

// Увімкнути ZGC -XX:+UseZGC -Xmx16g -XX:SoftMaxHeapSize=12g // soft limit, ZGC намагається не перевищувати // ZGC colored pointers: кожен pointer має кольорові біти // GC metadata вбудована в pointer → не потрібні RSets // Паузи ZGC: // Init Mark: STW <1ms // Concurrent Mark: concurrent // Relocate Start: STW <1ms // Concurrent Reloc: concurrent

Trade-off ZGC: нижчий throughput (~10-15% vs G1) і вищий CPU overhead. Не підходить для CPU-bound задач де кожен % CPU критичний.

Shenandoah GC (Red Hat)

Схожа мета що й ZGC — ultra-low latency. Concurrent compaction. Доступний в OpenJDK, стабільний з Java 15. Вибір між ZGC і Shenandoah — тест на реальному навантаженні.

На співбесіді: G1 = regions (не зони), Garbage-First (вибирає регіони з найбільшим сміттям), IHOP%, Mixed GC після concurrent marking, Humongous = об'єкти >50% region. ZGC = sub-ms паузи, colored pointers, concurrent relocation.
