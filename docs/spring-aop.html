концепти

AOP термінологія

  • Aspect — модуль що реалізує cross-cutting concern (логування, транзакції, безпека)
  • Join Point — точка виконання де можна вставити логіку (в Spring AOP — тільки виклик методу)
  • Pointcut — вираз що визначає які join points обробляти
  • Advice — код що виконується на join point (before, after, around)
  • Weaving — процес вставки advice. Spring — runtime weaving через proxy
// Мінімальний Aspect @Aspect @Component class LoggingAspect { // Pointcut: всі методи в пакеті service @Pointcut("execution(* com.example.service.*.*(..))") void serviceMethods() {} // Advice: before кожного виклику @Before("serviceMethods()") void logBefore(JoinPoint jp) { log.info("Calling: {}", jp.getSignature().getName()); } }

JDK vs CGLIB proxy

JDK Dynamic ProxyCGLIB Proxy
Механізмреалізує інтерфейсextends клас (підклас)
Умоваклас має інтерфейсбудь-який клас
final методине перехоплюєне може override
final класине застосовноне може extend
Конструкторне викликаєтьсявикликається
Spring Boot defaultнітак (з Spring 5)
Spring Boot 2+ використовує CGLIB за замовчуванням для всіх beans (навіть з інтерфейсами). Раніше JDK proxy для beans з інтерфейсами, CGLIB без. Зміна зменшила кількість сюрпризів.

Що не можна проксіювати

// final клас — CGLIB не може extend final class MyService { ... } // @Transactional не спрацює! // final метод — не можна override public final void doWork() { ... } // @Cacheable, @Transactional на final методі — мовчки ігнорується // private метод — не видимий proxy private void internalWork() { ... } // @Transactional на private — не спрацює

Pointcut і Advice типи

// Advice типи @Before("pointcut()") // до методу @AfterReturning // після успішного return @AfterThrowing // після exception @After // завжди (як finally) @Around // повний контроль // Pointcut expressions "execution(* com.example..*(..))" // будь-який метод в пакеті "execution(public * *(..))" // всі public методи "@annotation(org.springframework.transaction.annotation.Transactional)" "@within(org.springframework.stereotype.Service)" "bean(userService)" // конкретний bean "args(String, ..)" // перший аргумент String // Комбінування "execution(* *.find*(..)) && @within(Service)" "serviceLayer() || repositoryLayer()"
@Around

@Around — найпотужніший advice

Повний контроль: може не викликати метод, змінити аргументи, перехопити результат або exception.

@Around("@annotation(timed)") public Object measureTime(ProceedingJoinPoint pjp, Timed timed) throws Throwable { long start = System.nanoTime(); try { Object result = pjp.proceed(); // виклик реального методу return result; } catch (Exception ex) { metrics.recordError(timed.value()); throw ex; // перекидаємо exception } finally { long elapsed = System.nanoTime() - start; metrics.record(timed.value(), elapsed); } } // Доступ до аргументів методу Object[] args = pjp.getArgs(); String methodName = pjp.getSignature().getName(); Class<?> targetClass = pjp.getTarget().getClass();
pjp.proceed() треба викликати явно. Якщо забути — реальний метод не виконається! І завжди кидай exception далі якщо не хочеш його "проковтнути".
пастка

Self-invocation — найпоширеніша пастка AOP

При виклику методу через this — proxy обходиться. AOP advice, @Transactional, @Cacheable не спрацьовують.

@Service class OrderService { public void processOrder(Order order) { validateOrder(order); saveOrder(order); // ← self-invocation! } @Transactional // НЕ СПРАЦЮЄ при self-invocation public void saveOrder(Order order) { repo.save(order); } } // Що відбувається: // Зовнішній код викликає proxy.processOrder() // proxy делегує в реальний OrderService.processOrder() // всередині this.saveOrder() — вже НЕ через proxy! // @Transactional на saveOrder() ігнорується
Це стосується будь-якої Spring AOP анотації: @Transactional, @Cacheable, @Async, @Retry, @Timed, власних @Around aspects — всіх.
рішення

Рішення self-invocation

1. Рефакторинг (найкраще)

// Винести метод в окремий bean @Service class OrderSaver { @Transactional // спрацює — виклик через proxy public void save(Order o) { repo.save(o); } } @Service class OrderService { @Autowired OrderSaver saver; public void processOrder(Order o) { saver.save(o); // через proxy OrderSaver ✓ } }

2. Self-injection через ApplicationContext

@Service class OrderService { @Autowired ApplicationContext ctx; public void processOrder(Order o) { ctx.getBean(OrderService.class).saveOrder(o); // proxy ✓ } }

3. @EnableAspectJAutoProxy(exposeProxy=true)

// application.properties spring.aop.expose-proxy=true // У коді: ((OrderService) AopContext.currentProxy()).saveOrder(o);
Перший варіант (рефакторинг) — єдиний правильний з точки зору дизайну. Інші — workarounds. Якщо потрібен self-call → це сигнал що клас робить забагато (SRP).
обмеження

Обмеження Spring AOP

  • Тільки методи — Spring AOP не перехоплює доступ до полів або конструктори (на відміну від AspectJ)
  • Тільки Spring beans — якщо об'єкт створений через new — AOP не спрацює
  • final методи і класи — не можна проксіювати
  • private методи — proxy не бачить
  • Self-invocation — обходить proxy
// НЕ через proxy → AOP не спрацює MyService svc = new MyService(); // не Spring bean! svc.annotatedMethod(); // @Transactional ігнорується // AspectJ weaving (alternative до Spring AOP) // Compile-time або load-time weaving // Перехоплює конструктори, поля, private методи, self-call // Складніша конфігурація
На співбесіді: JDK proxy = інтерфейс, CGLIB = підклас (Boot default), self-invocation обходить proxy, final методи/класи не проксіюються, @Around потрібен pjp.proceed().
