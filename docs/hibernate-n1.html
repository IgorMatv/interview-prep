N+1

N+1 проблема

Завантажуємо N entities, потім для кожної — окремий SQL для lazy association. Разом: 1 + N запитів.

// Entity @Entity class Order { @OneToMany(fetch = LAZY) private List<Item> items; } // Repository List<Order> orders = repo.findAll(); // 1 SQL: SELECT * FROM orders // Тепер обходимо for (Order o : orders) { o.getItems().size(); // N SQL: SELECT * FROM items WHERE order_id=? // 100 orders → 100 запитів тут! } // Разом: 1 + 100 = 101 SQL запит
N+1 — одна з найчастіших причин деградації продуктивності в Hibernate застосунках. Особливо небезпечно коли прихована в сервісному шарі або маппері (MapStruct/ModelMapper).
рішення

JOIN FETCH — один SQL для всього

// JPQL з JOIN FETCH @Query("SELECT o FROM Order o JOIN FETCH o.items WHERE o.userId = :userId") List<Order> findWithItems(@Param("userId") long userId); // Один SQL: SELECT o.*, i.* FROM orders o JOIN items i ON ... // ОБЕРЕЖНО: кілька JOIN FETCH колекцій → MultipleBagFetchException // або декартовий добуток (дублювання рядків) // Рішення для 2+ колекцій: DISTINCT або Set замість List @Query("SELECT DISTINCT o FROM Order o JOIN FETCH o.items JOIN FETCH o.tags") // або окремі запити // HQL JOIN FETCH з пагінацією — небезпечно! // Hibernate завантажує ВСЕ в пам'ять, потім пагінує // HHH90003004: firstResult/maxResults specified with collection fetch
JOIN FETCH + пагінація = Hibernate завантажує всі рядки в пам'ять і пагінує там. При великих даних — OOM. Рішення: спочатку вибрати ids з пагінацією, потім JOIN FETCH по ids.

@EntityGraph

Альтернатива JOIN FETCH — визначаємо граф завантаження без зміни запиту. Зручно в Spring Data.

// Named EntityGraph на entity @Entity @NamedEntityGraph( name = "Order.withItems", attributeNodes = @NamedAttributeNode("items") ) class Order { ... } // Spring Data: використовуємо граф @EntityGraph("Order.withItems") List<Order> findByUserId(long userId); // Або inline (без @NamedEntityGraph) @EntityGraph(attributePaths = {"items", "items.product"}) List<Order> findByStatus(String status); // EntityGraph.EntityGraphType @EntityGraph(value = "Order.withItems", type = EntityGraph.EntityGraphType.FETCH) // FETCH: вказані → EAGER, решта → LAZY (default) // LOAD: вказані → EAGER, решта → за анотаціями mapping

@BatchSize — пакетне завантаження

Замість N окремих SELECT — завантажує кількох associations одним IN запитом. Компроміс між LAZY і JOIN FETCH.

@Entity class Order { @OneToMany @BatchSize(size = 25) // замість 100 запитів → 4 (100/25) private List<Item> items; } // SQL при 100 orders: // SELECT * FROM items WHERE order_id IN (1,2,...,25) -- 1 // SELECT * FROM items WHERE order_id IN (26,27,...,50) -- 2 // ... разом 4 запити замість 100 // Глобально для всіх lazy associations: // spring.jpa.properties.hibernate.default_batch_fetch_size=25
default_batch_fetch_size=25 в application.properties — простий спосіб суттєво зменшити N+1 без змін коду. Хороший початковий default для більшості проектів.

DTO через JPQL — найефективніше

Якщо не потрібна entity для модифікації — вибирай одразу DTO. Hibernate не відслідковує DTO, менше пам'яті і SQL.

// DTO record (Java 16+) record OrderSummary(long id, String status, BigDecimal total) {} // JPQL constructor expression @Query(""" SELECT new com.example.dto.OrderSummary(o.id, o.status, o.total) FROM Order o WHERE o.userId = :userId """) List<OrderSummary> findSummariesByUser(@Param("userId") long userId); // Spring Data Projections (інтерфейс) interface OrderSummaryProjection { long getId(); String getStatus(); BigDecimal getTotal(); } List<OrderSummaryProjection> findByUserId(long userId); // Spring Data генерує SELECT o.id, o.status, o.total FROM ... // Тільки потрібні колонки!
На співбесіді: N+1 = 1 SELECT + N SELECT для кожної lazy association, три рішення: JOIN FETCH (один SQL), @EntityGraph (декларативно), @BatchSize (IN замість N), DTO проекції — найефективніше для read-only.

Виявлення N+1 і моніторинг

// Логування SQL (development) spring.jpa.show-sql=true spring.jpa.properties.hibernate.format_sql=true logging.level.org.hibernate.SQL=DEBUG logging.level.org.hibernate.type.descriptor.sql.BasicBinder=TRACE // Статистика Hibernate spring.jpa.properties.hibernate.generate_statistics=true logging.level.org.hibernate.stat=DEBUG // Покаже: кількість запитів, час, cache hits // Бібліотека datasource-proxy / p6spy // Логує кожен SQL з параметрами і стек-трейсом // Hypersistence Utils — NPE + N+1 detection implementation 'io.hypersistence:hypersistence-utils-hibernate-63' // Кидає exception при N+1 в тестах
Перевірка в тестах: бібліотека hypersistence-utils або db-util дозволяє анотувати тест @SQLStatementCountValidator і валідувати точну кількість SQL запитів. Це ловить N+1 regression автоматично.
