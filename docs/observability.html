три стовпи

Три стовпи Observability

  • Metrics — агреговані числові дані в часі. "Скільки запитів/сек?", "p99 latency?". Prometheus → Grafana.
  • Traces — шлях одного запиту через систему. "Де саме повільно?". OpenTelemetry → Jaeger/Zipkin/Tempo.
  • Logs — детальні текстові події. "Що саме сталось?". Logback → ELK/Loki.
Тріада в дії: Alert по metric (error rate підвищився) → знайти проблемний trace → прочитати logs того request'у. Кожен стовп доповнює інші.

metrics

Metrics — Micrometer + Prometheus

// application.properties management.endpoints.web.exposure.include=prometheus,health,info,metrics management.metrics.export.prometheus.enabled=true // Типи метрик: // Counter: тільки зростає (кількість запитів, помилок) // Gauge: поточне значення (активні з'єднання, queue size) // Timer: тривалість + кількість (latency histogram) // DistributionSummary: розподіл значень (request size)
// Кастомні метрики @Autowired MeterRegistry registry; // Counter Counter routesCreated = Counter.builder("routes.created") .tag("status", "success") .register(registry); routesCreated.increment(); // Timer Timer optimizationTime = Timer.builder("route.optimization.duration") .register(registry); optimizationTime.record(() -> optimizer.optimize(stops)); // Gauge для queue size Gauge.builder("queue.size", pendingQueue, Queue::size) .register(registry);
tracing

Distributed Tracing (OpenTelemetry)

Trace = дерево spans. Span = одна операція з часом початку/кінця, атрибутами. TraceId — єдиний для всього запиту через усі сервіси.

// Автоматична інструментація (Spring Boot Actuator) // HTTP requests, DB queries, Kafka — автоматично // Кастомний span @Autowired Tracer tracer; Span span = tracer.nextSpan().name("route-optimization").start(); try (var scope = tracer.withSpan(span)) { span.tag("stops.count", stops.size()); var result = optimizer.optimize(stops); span.tag("algorithm", "two-opt"); return result; } catch (Exception e) { span.error(e); throw e; } finally { span.end(); } // application.properties management.tracing.sampling.probability=1.0 // 100% sampling (dev) management.tracing.sampling.probability=0.1 // 10% sampling (prod)
logging

Structured logging

JSON логи замість plain text. Кожне поле — окремий ключ. Дозволяє ефективний пошук і фільтрацію в ELK/Loki.

// Поганий лог (plain text): // 2024-01-15 10:23:45 ERROR RouteService - Failed to create route for user 42 // Важко фільтрувати за userId // Добрий лог (structured JSON): { "timestamp": "2024-01-15T10:23:45Z", "level": "ERROR", "logger": "com.example.RouteService", "message": "Failed to create route", "userId": 42, "requestId": "550e8400-...", "traceId": "abc123", "spanId": "def456", "exception": "IllegalStateException: No available pois" } // Logback + logstash-logback-encoder // logback-spring.xml: <appender name="JSON" class="ch.qos.logback.core.ConsoleAppender"> <encoder class="net.logstash.logback.encoder.LogstashEncoder"/> </appender>
Spring Boot 3.4+ підтримує structured logging нативно через logging.structured.format.console=ecs — без додаткових залежностей.
MDC

Correlation ID і MDC

MDC (Mapped Diagnostic Context) — thread-local map для додавання контексту до всіх логів у рамках запиту.

// Filter: додаємо requestId до MDC на початку запиту @Component public class RequestIdFilter extends OncePerRequestFilter { @Override protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res, FilterChain chain) throws ServletException, IOException { try { String requestId = req.getHeader("X-Request-Id"); if (requestId == null) requestId = UUID.randomUUID().toString(); MDC.put("requestId", requestId); MDC.put("userId", extractUserId(req)); res.setHeader("X-Request-Id", requestId); chain.doFilter(req, res); } finally { MDC.clear(); // ВАЖЛИВО: очистити перед поверненням в пул } } } // Тепер КОЖЕН лог в рамках запиту автоматично містить requestId: // {"requestId":"abc123","userId":"42","message":"Route created"}
MDC.clear() у finally — обов'язково. При thread pool без clear — наступна задача успадкує MDC попередньої.
best practices

Log рівні і best practices

// Рівні — що куди // ERROR: помилка що вимагає негайної уваги log.error("Payment failed for order {}: {}", orderId, ex.getMessage(), ex); // WARN: щось підозріле, але сервіс продовжує log.warn("Slow query detected: {}ms for {}", duration, query); // INFO: важлива бізнес-подія log.info("Route {} created for user {}", routeId, userId); // DEBUG: детально для розробки log.debug("Optimization step {}: {} stops", step, stops.size()); // TRACE: найдетальніше (майже не використовується в prod) log.trace("Processing stop: {}", stop); // application.properties logging.level.root=INFO logging.level.com.example=DEBUG // наш код — більше деталей logging.level.org.hibernate.SQL=DEBUG // SQL логування
  • Ніколи не логуй паролі, токени, PAN (credit cards), PII у plain text
  • Структуровані поля замість конкатенації рядків
  • Exception завжди передавай останнім параметром (буде stack trace)
  • INFO для бізнес-логіки, DEBUG для технічних деталей
alerting

Alerting і Grafana

// Ключові метрики для alerting (Spring Boot) // Error rate rate(http_server_requests_seconds_count{status=~"5.."}[5m]) / rate(http_server_requests_seconds_count[5m]) > 0.01 // Alert якщо > 1% 5xx за 5 хвилин // p99 latency histogram_quantile(0.99, rate(http_server_requests_seconds_bucket[5m]) ) > 1.0 // Alert якщо p99 > 1 секунда // Connection pool exhausted hikaricp_connections_pending > 0 // Alert якщо є потоки що чекають на з'єднання // Heap usage jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} > 0.85 // Alert якщо heap > 85%
На співбесіді: три стовпи (metrics/traces/logs), Micrometer = vendor-neutral metrics facade (як SLF4J для логів), traceId через усі сервіси для кореляції, structured logging + MDC для context, не логуй sensitive data, p99 важливіший за average.
