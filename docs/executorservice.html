навіщо

Навіщо пул потоків

Створення потоку — дорога операція (~1MB stack, OS context). Пул потоків перевикористовує потоки, обмежує їх кількість і управляє чергою задач.

// БЕЗ пулу: new Thread на кожну задачу for (Request req : requests) { new Thread(() -> process(req)).start(); // 10000 запитів = 10000 потоків = OOM } // З пулом: обмежена кількість потоків ExecutorService pool = Executors.newFixedThreadPool(10); for (Request req : requests) { pool.submit(() -> process(req)); // задачі стають у чергу, 10 потоків обробляють }
ExecutorService — інтерфейс. Основна реалізація — ThreadPoolExecutor. Executors — factory що створює готові конфігурації.
factory

Executors factory методи

// Fixed pool: n потоків, LinkedBlockingQueue Executors.newFixedThreadPool(4); // Коли: CPU-bound задачі, відомий parallelism // Cached pool: потоки на вимогу, idle 60s Executors.newCachedThreadPool(); // Коли: короткі IO-bound задачі, невідома кількість // РИЗИК: необмежена кількість потоків → OOM // Single thread: 1 потік, гарантований порядок Executors.newSingleThreadExecutor(); // Коли: задачі мають виконуватись послідовно // Scheduled: таймер і периодичні задачі ScheduledExecutorService sch = Executors.newScheduledThreadPool(2); sch.scheduleAtFixedRate(task, 0, 5, TimeUnit.SECONDS); // Work-stealing pool (Java 8+): ForkJoinPool Executors.newWorkStealingPool(); // Коли: рекурсивні задачі, divide-and-conquer // Virtual thread executor (Java 21) Executors.newVirtualThreadPerTaskExecutor(); // Коли: IO-bound, тисячі конкурентних задач
newCachedThreadPool() і newFixedThreadPool() мають unbounded queue — задачі накопичуються нескінченно. У продакшн завжди створюй ThreadPoolExecutor з bounded queue явно.
API

submit() vs execute()

// execute(): fire and forget, Runnable тільки pool.execute(() -> doWork()); // exception в задачі → UncaughtExceptionHandler або тихо губиться // submit(): повертає Future, Runnable або Callable Future<String> future = pool.submit(() -> compute()); String result = future.get(); // блокує, кидає ExecutionException future.get(5, TimeUnit.SECONDS); // з таймаутом future.cancel(true); // скасування // invokeAll: всі задачі, чекає завершення всіх List<Future<String>> futures = pool.invokeAll(tasks); // invokeAny: повертає першу успішну String first = pool.invokeAny(tasks);
Exception swallowing з execute(). Якщо Runnable кидає RuntimeException — вона губиться якщо не встановлено UncaughtExceptionHandler. Завжди використовуй submit() і перевіряй Future.get().

7 параметрів ThreadPoolExecutor

new ThreadPoolExecutor( 4, // corePoolSize 8, // maximumPoolSize 60, TimeUnit.SECONDS, // keepAliveTime new LinkedBlockingQueue<>(100), // workQueue new ThreadFactory() { ... }, // threadFactory new ThreadPoolExecutor.CallerRunsPolicy() // handler );

Логіка додавання задачі

Порядок важливий — нова задача проходить три перевірки:

// 1. threads < corePoolSize → створити новий потік // 2. queue не повна → додати в чергу // 3. threads < maximumPoolSize → створити non-core потік // 4. queue повна + max threads → RejectedExecutionHandler
Нові потоки понад core створюються тільки коли черга повна. З unbounded queue (LinkedBlockingQueue без ліміту) — maximumPoolSize ніколи не досягається. Це типова пастка.

keepAliveTime

Час через який idle non-core потік завершується. За замовчуванням core потоки не завершуються. Можна змінити: allowCoreThreadTimeOut(true).

rejection

Rejection Policies

Викликається коли задачу неможливо виконати: пул зупинено або черга і всі потоки зайняті.

PolicyПоведінка
AbortPolicyкидає RejectedExecutionException (default)
CallerRunsPolicyвиконує в потоці caller — природній backpressure
DiscardPolicyмовчки ігнорує задачу
DiscardOldestPolicyвидаляє найстарішу з черги і retry
// Своя політика: логуємо відхилення new ThreadPoolExecutor.RejectedExecutionHandler() { public void rejectedExecution(Runnable r, ThreadPoolExecutor e) { log.warn("Task rejected: {}", r); metrics.increment("rejected_tasks"); // або відправити в fallback queue } };
CallerRunsPolicy — найрозумніший дефолт для більшості сервісів: уповільнює producer природно без втрати задач.
lifecycle

shutdown() vs shutdownNow()

// shutdown(): graceful — дочекатись всіх задач pool.shutdown(); // нові задачі не приймаються, черга допрацьовується // Правильний graceful shutdown: pool.shutdown(); try { if (!pool.awaitTermination(30, TimeUnit.SECONDS)) { pool.shutdownNow(); // примусово якщо не завершилось if (!pool.awaitTermination(10, TimeUnit.SECONDS)) log.error("Pool did not terminate"); } } catch (InterruptedException e) { pool.shutdownNow(); Thread.currentThread().interrupt(); } // shutdownNow(): агресивний — interrupt всіх потоків List<Runnable> unfinished = pool.shutdownNow(); // повертає задачі що не почались // виконуючі задачі отримують interrupt (cooperative!)
налаштування

Розмір пулу і формула

Немає універсальної відповіді — залежить від типу задач і CPU. Дві базові формули:

// CPU-bound задачі (обчислення, без IO): int cpus = Runtime.getRuntime().availableProcessors(); int poolSize = cpus + 1; // +1 для компенсації page faults // IO-bound задачі (DB, HTTP, файли): // poolSize = cpus * targetCpuUtilization * (1 + W/C) // W = wait time, C = compute time // Якщо 90% часу чекаємо IO: int poolSize = cpus * 1 * (1 + 9); // = 10 * cpus
Ці формули — відправна точка, не відповідь. Завжди вимірюй з реальним навантаженням. Занадто великий пул → context switching overhead. Занадто малий → idle CPU або черга росте.

Моніторинг пулу

ThreadPoolExecutor tpe = (ThreadPoolExecutor) pool; tpe.getPoolSize() // поточна кількість потоків tpe.getActiveCount() // зайнятих потоків tpe.getQueue().size() // задач у черзі tpe.getCompletedTaskCount() // виконано всього tpe.getTaskCount() // заплановано всього
На співбесіді: логіка додавання задачі (core → queue → max → reject), різниця shutdown/shutdownNow, CallerRunsPolicy як backpressure, unbounded queue пастка.
