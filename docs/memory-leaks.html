основи

Що таке memory leak в Java

Memory leak у Java — це об'єкти що більше не потрібні, але GC не може їх зібрати бо є strong reference з "живого" коду. Heap росте, врешті — OutOfMemoryError.

// GC roots — звідки починається reachability аналіз: // - Static поля // - Локальні змінні активних методів (Stack frames) // - Активні Thread об'єкти // - JNI references // Leak = об'єкт досяжний від GC root, але більше не потрібен static List<Object> CACHE = new ArrayList<>(); void process(Object obj) { CACHE.add(obj); // obj ніколи не видаляється → leak }
Симптоми: heap поступово зростає з часом, Full GC стає частішим і менш ефективним, врешті java.lang.OutOfMemoryError: Java heap space.
причина

Static поля і необмежені колекції

Static поля живуть весь час роботи JVM. Колекція в static полі що тільки росте і ніколи не очищається — найпоширеніший leak.

// ПОГАНО: static кеш без eviction static final Map<String, Object> CACHE = new HashMap<>(); void handle(Request req) { CACHE.put(req.getId(), process(req)); // ніколи не видаляємо } // Рішення 1: обмежений кеш з eviction static final Map<String, Object> CACHE = new LinkedHashMap<String, Object>(100, 0.75f, true) { protected boolean removeEldestEntry(Map.Entry<?,?> e) { return size() > 100; } }; // Рішення 2: WeakHashMap — ключ = WeakReference Map<Object, Data> cache = new WeakHashMap<>(); // entry видаляється коли немає strong ref на ключ // Рішення 3: Caffeine/Guava з TTL і max size Cache<K, V> cache = Caffeine.newBuilder() .maximumSize(1000) .expireAfterWrite(10, TimeUnit.MINUTES) .build();
причина

ThreadLocal leak

ThreadLocal зберігає значення per-thread. Якщо потік з ThreadPoolExecutor не видаляє значення після задачі — воно залишається і може "протікати" на наступну задачу або просто займати пам'ять.

// Типова помилка: ThreadLocal у thread pool без cleanup static final ThreadLocal<UserContext> CTX = new ThreadLocal<>(); void handleRequest(User user) { CTX.set(new UserContext(user)); doWork(); // ЗАБУЛИ CTX.remove() → значення залишається в потоці } // Правильно: завжди remove() у finally void handleRequest(User user) { CTX.set(new UserContext(user)); try { doWork(); } finally { CTX.remove(); // обов'язково! } }
ThreadLocal в thread pool: потоки повертаються в пул і використовуються знову. Без remove() наступна задача може побачити дані попередньої. І це security issue, не тільки memory leak.
причина

Listener і callback leak

Об'єкт реєструє себе як listener, але забуває зареєструватись. Publisher тримає strong reference → об'єкт не збирається GC навіть коли "логічно" вже не потрібен.

// ПОГАНО: listener ніколи не видаляється class Widget { Widget(EventBus bus) { bus.subscribe(this); // bus тримає strong ref на Widget } // Widget "живе" поки bus не видалить його } // Правильно: unsubscribe при завершенні class Widget implements AutoCloseable { private final EventBus bus; Widget(EventBus bus) { this.bus = bus; bus.subscribe(this); } @Override public void close() { bus.unsubscribe(this); // прибираємо себе } } // Альтернатива: WeakReference listeners // EventBus тримає WeakReference на listeners // GC може зібрати listener навіть без unsubscribe
причина

Inner class і неявне посилання

Non-static inner class неявно тримає посилання на outer class instance. Якщо inner class "виходить" назовні — outer не збереться GC.

// Non-static inner class: неявний this$0 class Outer { private byte[] bigData = new byte[1024 * 1024]; // 1MB class Inner implements Runnable { @Override public void run() { System.out.println("done"); } // Inner неявно тримає Outer.this → bigData не збереться } } Outer outer = new Outer(); executor.submit(new Outer.Inner()); // outer тримається executor'ом! outer = null; // не допомагає — executor тримає Inner → Outer // Рішення: static inner class (немає посилання на Outer) static class Inner implements Runnable { @Override public void run() { ... } } // Або lambda/anonymous class обережно з capture executor.submit(() -> doWork()); // лямбда не захоплює this якщо не використовує
інструменти

Heap dump і аналіз

// Отримати heap dump jmap -dump:format=b,file=heap.hprof <pid> jcmd <pid> GC.heap_dump /tmp/heap.hprof // Автоматично при OOM -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/var/log/heap.hprof // Аналізатори heap dump: // Eclipse MAT (Memory Analyzer Tool) — найпотужніший // VisualVM — простіший, вбудований // IntelliJ IDEA — вбудований аналізатор // JDK Mission Control (JMC)
Алгоритм аналізу в MAT: відкрити heap dump → Leak Suspects Report (автоматично знаходить підозрілі об'єкти) → Dominator Tree (які об'єкти утримують найбільше пам'яті) → Path to GC Roots (чому об'єкт не збирається).

Ключові запити в MAT

// OQL (Object Query Language) в MAT: SELECT * FROM java.util.HashMap WHERE size > 10000 SELECT * FROM java.lang.ThreadLocal$ThreadLocalMap$Entry // Знайти великі масиви: SELECT * FROM byte[] WHERE size > 1048576
інструменти

JFR і VisualVM

Java Flight Recorder (JFR)

Вбудований в JDK профайлер з мінімальним overhead (~1%). Записує events: GC, allocation, thread, CPU. Найкращий для продакшн профілювання.

// Запустити JFR профілювання jcmd <pid> JFR.start name=rec duration=60s filename=rec.jfr // або з флагами при старті: -XX:StartFlightRecording=duration=60s,filename=rec.jfr // Аналіз у JDK Mission Control (JMC): // Memory → Object Allocation — де і що алокується // GC → GC Events — тривалість і частота GC // Threads → Thread States — де блокуються потоки

Корисні jcmd команди

jcmd <pid> VM.flags // JVM флаги jcmd <pid> GC.heap_info // heap regions jcmd <pid> Thread.print // thread dump jcmd <pid> VM.native_memory // native memory (треба -XX:+NativeMemoryTracking=summary) jcmd <pid> GC.class_stats // статистика класів // jstat — моніторинг GC jstat -gcutil <pid> 1000 10 // кожну секунду, 10 разів
На співбесіді: типові причини leak (static колекції, ThreadLocal без remove, listener без unsubscribe, non-static inner class), діагностика: heap dump + MAT, JFR для production profiling.
