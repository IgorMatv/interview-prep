створення

Способи створити CompletableFuture

// runAsync: Runnable, повертає CompletableFuture<Void> CompletableFuture<Void> cf1 = CompletableFuture.runAsync(() -> doWork()); // supplyAsync: Supplier, повертає CompletableFuture<T> CompletableFuture<String> cf2 = CompletableFuture.supplyAsync(() -> compute()); // completedFuture: вже завершений з результатом CompletableFuture<String> done = CompletableFuture.completedFuture("result"); // failedFuture (Java 9+): вже завершений з помилкою CompletableFuture<String> failed = CompletableFuture.failedFuture(new RuntimeException()); // Ручне завершення CompletableFuture<String> manual = new CompletableFuture<>(); manual.complete("done"); // завершити з результатом manual.completeExceptionally(ex); // завершити з помилкою
За замовчуванням supplyAsync і runAsync використовують ForkJoinPool.commonPool(). Для IO-bound задач завжди передавай власний Executor — інакше блокуєш спільний пул.
chaining

thenApply vs thenCompose

thenApply — трансформує результат (Function). thenCompose — якщо трансформація сама повертає CompletableFuture (як flatMap у Stream).

// thenApply: T → U (синхронна трансформація) CompletableFuture<String> upper = CompletableFuture.supplyAsync(() -> "hello") .thenApply(String::toUpperCase) // → "HELLO" .thenApply(s -> s + "!"); // → "HELLO!" // thenCompose: T → CompletableFuture<U> (уникає вкладення) CompletableFuture<User> user = getOrderId() // CF<Long> .thenCompose(id -> getUser(id)); // id → CF<User> // без thenCompose вийшло б CF<CF<User>> // thenAccept: споживає результат без повернення cf.thenAccept(result -> log(result)); // thenRun: виконує дію після завершення (немає доступу до result) cf.thenRun(() -> cleanup());

Async суфікс — виконання в іншому потоці

// thenApply: виконується в потоці попереднього stage cf.thenApply(s -> transform(s)); // thenApplyAsync: виконується в ForkJoinPool.commonPool() cf.thenApplyAsync(s -> transform(s)); // thenApplyAsync з executor: у заданому пулі cf.thenApplyAsync(s -> transform(s), myExecutor);
Async варіанти корисні якщо transformation важка і не хочеш навантажувати потік попереднього stage. Для легких трансформацій — просто thenApply.
комбінування

thenCombine, allOf, anyOf

// thenCombine: результати двох CF CompletableFuture<String> greeting = CompletableFuture.supplyAsync(() -> "Hello") .thenCombine( CompletableFuture.supplyAsync(() -> "World"), (a, b) -> a + " " + b); // → "Hello World" // allOf: чекати всіх (немає результату напряму) CompletableFuture<Void> all = CompletableFuture.allOf(cf1, cf2, cf3); all.thenRun(() -> { String r1 = cf1.join(); // join() не кидає checked String r2 = cf2.join(); }); // allOf + collect результатів List<CompletableFuture<String>> futures = tasks.stream() .map(t -> CompletableFuture.supplyAsync(() -> process(t))) .toList(); List<String> results = CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])) .thenApply(v -> futures.stream().map(CompletableFuture::join).toList()) .join(); // anyOf: перший завершений (будь-якого типу) CompletableFuture<Object> first = CompletableFuture.anyOf(cf1, cf2, cf3);
помилки

Exception handling: exceptionally, handle, whenComplete

// exceptionally: тільки при помилці — як catch cf.exceptionally(ex -> { log.error("Failed", ex); return "default"; // відновлення }); // handle: завжди — і результат і помилка cf.handle((result, ex) -> { if (ex != null) return "error"; return result.toUpperCase(); }); // whenComplete: side effect без трансформації cf.whenComplete((result, ex) -> { if (ex != null) metrics.error(); else metrics.success(); // результат не змінюється });

Exception propagation у ланцюжку

// Помилка пропагується через весь ланцюжок CompletableFuture.supplyAsync(() -> { throw new RuntimeException("oops"); }) .thenApply(s -> s.toUpperCase()) // пропускається .thenApply(s -> s + "!") // пропускається .exceptionally(ex -> "recovered"); // ловить тут // join() vs get() cf.get(); // checked: ExecutionException, InterruptedException cf.join(); // unchecked: CompletionException (зручніше в ланцюжку)
Якщо не додати exceptionally або handle — виключення в CF мовчки ковтається. Завжди обробляй помилки явно, особливо в fire-and-forget сценаріях.
executor

Executor для задач

За замовчуванням усі async операції використовують ForkJoinPool.commonPool(). Для IO-bound задач це небезпечно — блокується спільний пул всього застосунку.

// ПОГАНО: IO в commonPool CompletableFuture.supplyAsync(() -> httpClient.get(url)); // блокує потік commonPool на час HTTP запиту // ДОБРЕ: окремий executor для IO Executor ioExecutor = Executors.newFixedThreadPool(20); CompletableFuture.supplyAsync(() -> httpClient.get(url), ioExecutor) .thenApplyAsync(response -> parse(response)) // commonPool ОК .thenAcceptAsync(result -> save(result), ioExecutor); // IO → свій
Гарна практика: два executor — cpuExecutor (N+1 потоків, CPU задачі) і ioExecutor (10-50 потоків, IO задачі). Передавай потрібний явно.
антипатерни

Типові помилки

1. get() без таймауту

// ПОГАНО: може чекати вічно String result = cf.get(); // ДОБРЕ: завжди з таймаутом String result = cf.get(5, TimeUnit.SECONDS);

2. Блокуючий код у supplyAsync без IO executor

// ПОГАНО: блокує ForkJoinPool CompletableFuture.supplyAsync(() -> { return jdbcTemplate.query(...); // blocking IO! }); // ДОБРЕ CompletableFuture.supplyAsync(() -> jdbcTemplate.query(...), ioPool);

3. Не обробляти помилки

// ПОГАНО: exception загублений CompletableFuture.runAsync(() -> riskyWork()); // якщо кинуло exception — нічого не відбудеться // ДОБРЕ CompletableFuture.runAsync(() -> riskyWork()) .exceptionally(ex -> { log.error("Failed", ex); return null; });
На співбесіді: thenApply vs thenCompose (flatMap аналогія), allOf + collect results патерн, exceptionally vs handle, чому не використовувати commonPool для IO.
