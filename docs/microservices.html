Distributed transactions проблема

В мікросервісах кожен сервіс має свою БД. Немає глобального ACID. 2PC (two-phase commit) — теоретично вирішує, але на практиці крихкий і повільний.

// Класична проблема: оформлення замовлення // OrderService.createOrder() → INSERT order (success) // PaymentService.charge() → charge card (success) // InventoryService.reserve() → reserve items (FAIL!) // Що тепер? Order і Payment вже записані... // 2PC проблема: // Coordinator SPOF (single point of failure) // При падінні coordinator → всі учасники заблоковані // Несумісний з BASE (поки обидва учасники в prepare state)
Сучасний підхід: Saga для розподіленої "транзакції" і Outbox для надійної публікації подій. 2PC — уникати.
Saga

Saga патерн

Послідовність локальних транзакцій. При невдачі — компенсаційні транзакції відміняють попередні кроки. Два варіанти: Choreography і Orchestration.

Choreography — через события

// OrderService: CREATE order → publish OrderCreated // PaymentService: listens OrderCreated → charge → publish PaymentDone // InventoryService: listens PaymentDone → reserve → publish Reserved // // При невдачі: // InventoryService: publish ReservationFailed // PaymentService: listens ReservationFailed → refund → publish Refunded // OrderService: listens Refunded → cancel order

Orchestration — через оркестратор

// SagaOrchestrator знає всі кроки і координує OrderSagaOrchestrator: step1: paymentService.charge() on_failure: → rollback (нема попередніх кроків) step2: inventoryService.reserve() on_failure: paymentService.refund() → rollback // Orchestration: ясніша логіка, але SPOF orchestrator // Choreography: слабка зв'язність, але важко відстежити flow
Outbox

Outbox патерн

Надійна публікація подій: пишемо в DB і Kafka атомарно. Без Outbox — між persist і publish може бути збій → подія загублена.

// Проблема без Outbox: @Transactional void createOrder(Order order) { orderRepo.save(order); // success kafkaTemplate.send(event); // FAIL тут → order saved, event lost! } // Outbox рішення: @Transactional void createOrder(Order order) { orderRepo.save(order); outboxRepo.save(new OutboxEvent("OrderCreated", order.getId())); // обидва в одній транзакції → атомарно } // Окремий поллер (або CDC) читає outbox і публікує в Kafka @Scheduled(fixedDelay = 1000) void publishPendingEvents() { outboxRepo.findUnpublished() .forEach(e -> { kafkaTemplate.send(e.getTopic(), e.getPayload()); e.markPublished(); }); }
Debezium (CDC) — читає WAL (Write-Ahead Log) PostgreSQL і публікує зміни в Kafka. Без поллера, мінімальна затримка, production-ready.
Circuit Breaker

Circuit Breaker

Якщо залежний сервіс повільний або падає — Circuit Breaker відкривається і запити не йдуть до нього. Після таймауту — пробний запит.

// 3 стани: // CLOSED: нормальна робота, запити проходять // OPEN: CB відкритий, запити відхиляються одразу (fallback) // HALF-OPEN: один пробний запит, якщо success → CLOSED // CLOSED → OPEN: failure rate > threshold (50%) // OPEN → HALF-OPEN: після wait duration (60s) // HALF-OPEN → CLOSED: пробний запит успішний // HALF-OPEN → OPEN: пробний запит невдалий
Без Circuit Breaker: повільний сервіс → накопичення threads що чекають → thread pool exhaustion → наш сервіс теж падає. Це cascading failure. CB ізолює проблему.
resilience

Retry і Bulkhead

Retry з exponential backoff

// Без jitter: всі retries одночасно → thundering herd // З jitter: random delay → розподілене навантаження delay = min(cap, base * 2^attempt) + random_jitter // Коли retry допомагає: // - Network blip (transient errors) // - 429 Too Many Requests (rate limited) // - 503 Service Unavailable // // Коли retry шкодить: // - Non-idempotent operations без idempotency key // - 4xx client errors (400, 401, 404) // - При circuit breaker OPEN

Bulkhead — ізоляція ресурсів

// Окремий thread pool для кожного залежного сервісу // Payment service pool: 10 threads // Inventory service pool: 10 threads // // Якщо Inventory зависне → тільки його 10 threads зайняті // Payment продовжує працювати в своєму pool // Без bulkhead → один повільний сервіс вичерпує всі threads
Resilience4j

Resilience4j в Spring Boot

// application.properties resilience4j.circuitbreaker.instances.paymentService.failure-rate-threshold=50 resilience4j.circuitbreaker.instances.paymentService.wait-duration-in-open-state=60s resilience4j.circuitbreaker.instances.paymentService.sliding-window-size=10 resilience4j.retry.instances.paymentService.max-attempts=3 resilience4j.retry.instances.paymentService.wait-duration=500ms resilience4j.retry.instances.paymentService.exponential-backoff-multiplier=2
// Використання @CircuitBreaker(name = "paymentService", fallbackMethod = "fallbackCharge") @Retry(name = "paymentService") public PaymentResult charge(PaymentRequest req) { return paymentClient.charge(req); } private PaymentResult fallbackCharge(PaymentRequest req, Exception ex) { log.warn("Payment service unavailable, queuing", ex); pendingQueue.enqueue(req); return PaymentResult.pending(); }
service discovery

Service Discovery

В мікросервісах адреси сервісів динамічні (контейнери). Service Discovery дозволяє знайти сервіс за іменем без hardcoded IP.

// Client-side discovery (Eureka, Consul): // 1. Сервіс реєструється в registry при старті // 2. Клієнт питає registry: "де payment-service?" // 3. Registry повертає список інстансів // 4. Клієнт сам балансує (Ribbon/LoadBalancer) // Server-side discovery (Kubernetes): // 1. kube-proxy, DNS або Ingress router балансує // 2. Клієнт просто: http://payment-service/charge // 3. K8s Service → знаходить pods за label selector // Spring Cloud LoadBalancer (Spring Boot 3): @LoadBalanced @Bean public RestTemplate restTemplate() { return new RestTemplate(); } // "http://payment-service/charge" → resolve через discovery
На співбесіді: Saga = локальні транзакції + компенсація (Choreography vs Orchestration), Outbox = атомарний persist + event (DB → CDC/poller → Kafka), Circuit Breaker = 3 стани + cascading failure prevention, Bulkhead = ізоляція thread pools.
