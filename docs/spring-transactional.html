Як @Transactional працює

Spring створює AOP proxy навколо bean. Proxy перехоплює виклик, відкриває транзакцію, викликає реальний метод, робить commit або rollback.

// Спрощено що робить proxy: void saveOrder(Order order) { TransactionStatus tx = txManager.getTransaction(def); // BEGIN try { realService.saveOrder(order); // реальний виклик txManager.commit(tx); // COMMIT } catch (RuntimeException ex) { txManager.rollback(tx); // ROLLBACK throw ex; } } // Анотація на класі — застосовується до всіх public методів @Transactional class OrderService { ... } // Анотація на методі — більш специфічна, переважає над класом @Transactional(readOnly = true) public Order findById(long id) { ... }

propagation

REQUIRED vs REQUIRES_NEW

Найчастіші на співбесіді. Ключова різниця — що відбувається при rollback.

// REQUIRED (default): одна спільна транзакція @Transactional // REQUIRED void outer() { repo.save(orderA); inner(); // приєднується до тієї ж tx repo.save(orderB); } @Transactional // REQUIRED — використовує outer tx void inner() { repo.save(log); throw new RuntimeException(); // rollback ВСІЄЇ tx: orderA, orderB, log — всі скасовані }
// REQUIRES_NEW: окрема транзакція @Transactional // REQUIRED void outer() { repo.save(orderA); auditLogger.log(event); // REQUIRES_NEW throw new RuntimeException(); // outer rollback: orderA скасований // але log вже committed в окремій tx! } @Transactional(propagation = Propagation.REQUIRES_NEW) void log(Event e) { auditRepo.save(e); // commit незалежно від outer }
REQUIRES_NEW типовий use case: audit logging, відправка email-підтвердження — речі що мають зберегтись навіть якщо основна операція провалилась.
rollback

Rollback rules

За замовчуванням rollback тільки при unchecked exceptions (RuntimeException і Error). Checked exceptions — commit!

// Default: rollback тільки при RuntimeException @Transactional void process() throws IOException { repo.save(data); throw new IOException("file error"); // COMMIT відбудеться! IOException = checked = no rollback } // Явний rollback для checked exceptions @Transactional(rollbackFor = IOException.class) void process() throws IOException { ... } // Або для всіх checked @Transactional(rollbackFor = Exception.class) // Не робити rollback при конкретному unchecked @Transactional(noRollbackFor = EntityNotFoundException.class) // Програмний rollback TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
Checked exception + @Transactional без rollbackFor = один з найпоширеніших прихованих багів. Дані зберігаються навіть при помилці.
оптимізація

readOnly і isolation

readOnly = true

Підказка для бази даних і Hibernate. Hibernate не відстежує зміни (no dirty checking), може використовуватись read replica, DB може оптимізувати.

// Для read-only методів @Transactional(readOnly = true) public List<Order> findAll() { return repo.findAll(); } // Типовий патерн: клас з readOnly, методи write override @Transactional(readOnly = true) class OrderService { @Transactional // override: readOnly=false для write public void save(Order o) { ... } public Order findById(long id) { ... } // readOnly=true з класу }

isolation

// Рівні ізоляції (від найменшого до найбільшого) @Transactional(isolation = Isolation.READ_UNCOMMITTED) // dirty read @Transactional(isolation = Isolation.READ_COMMITTED) // default PostgreSQL @Transactional(isolation = Isolation.REPEATABLE_READ) // default MySQL InnoDB @Transactional(isolation = Isolation.SERIALIZABLE) // найсильніший // Зазвичай DEFAULT (береться від бази даних) @Transactional(isolation = Isolation.DEFAULT)
пастки

Self-invocation і private методи

// Self-invocation: @Transactional не спрацює @Service class OrderService { public void processOrder(Order o) { saveOrder(o); // this.saveOrder() — обходить proxy! } @Transactional // НЕ спрацює public void saveOrder(Order o) { repo.save(o); } } // Private метод: @Transactional не спрацює @Transactional // CGLIB не може override private private void doSave(Order o) { ... } // Рішення: перемістити в окремий @Service // або використати ApplicationContext.getBean()
Три умови щоб @Transactional спрацювала: метод викликається через Spring proxy (не this), метод public, клас не final і метод не final.
антипатерни

Типові помилки

1. Транзакція занадто широка

// ПОГАНО: HTTP виклик всередині транзакції @Transactional void processPayment() { order = repo.findById(id); paymentResult = paymentGateway.charge(order); // HTTP! Тримаємо з'єднання DB repo.updateStatus(order, paymentResult); } // ДОБРЕ: мінімальний scope транзакції void processPayment() { order = getOrder(id); // окрема tx result = paymentGateway.charge(order); // поза tx updateOrderStatus(order, result); // окрема tx }

2. Catch exception всередині @Transactional

// ПОГАНО: Spring не бачить exception → commit замість rollback @Transactional void save(Order o) { try { repo.save(o); } catch (Exception e) { log.error("Error", e); // проковтнули exception! } } // ДОБРЕ: логуй але перекидай або помічай rollbackOnly catch (Exception e) { log.error("Error", e); TransactionAspectSupport.currentTransactionStatus().setRollbackOnly(); }
На співбесіді: REQUIRED vs REQUIRES_NEW (спільна tx vs окрема), rollback тільки для unchecked (додай rollbackFor = Exception.class), readOnly = Hibernate без dirty checking, self-invocation обходить proxy.
