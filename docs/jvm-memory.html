Регіони пам'яті JVM

Heap
Об'єкти і масиви. Спільний для всіх потоків. GC управляє. Young Gen + Old Gen.
Stack (per thread)
Stack frames: локальні змінні, operand stack, посилання на поточний метод. За замовчуванням ~512KB–1MB.
Metaspace (Java 8+)
Метадані класів, bytecode, constant pool. Native memory, не heap. Замінив PermGen.
Code Cache
JIT-скомпільований native код. Обмежений, при переповненні JIT вимикається.
Direct / Off-heap
ByteBuffer.allocateDirect(), MappedByteBuffer. Поза heap — не збирається GC напряму.

Ключові JVM флаги для розміру

// Heap -Xms512m // початковий heap -Xmx2g // максимальний heap // Metaspace -XX:MetaspaceSize=128m -XX:MaxMetaspaceSize=256m // Thread stack -Xss512k // розмір стеку кожного потоку // Code cache -XX:ReservedCodeCacheSize=256m
heap

Heap: Young Generation і Old Generation

Heap поділений на зони для ефективного GC. Більшість об'єктів живе коротко — Young Gen збирається часто і швидко.

// Young Generation (Minor GC) // ├── Eden: нові об'єкти створюються тут // ├── Survivor S0: вижили з Eden // └── Survivor S1: альтернативний survivor // // Old Generation (Major/Full GC) // └── Tenured: об'єкти що пережили N minor GC // Розміри (приблизно) // Young : Old = 1 : 2 (за замовчуванням) -XX:NewRatio=2 // Old/Young ratio -XX:SurvivorRatio=8 // Eden/Survivor ratio (8:1:1)
Promotion: об'єкт переходить з Young → Old після N minor GC (за замовчуванням 15). Великі об'єкти (humongous) можуть одразу потрапити в Old. Налаштовується: -XX:MaxTenuringThreshold=15.

Minor GC vs Major GC vs Full GC

// Minor GC: Young Gen тільки, часто, швидко (мс) // - Eden повна → очистити Eden + S0 → виживших → S1 // - Stop-The-World, але дуже короткий // Major GC: Old Gen, рідко, повільно (сотні мс) // - спрацьовує коли Old Gen заповнена // Full GC: весь heap + Metaspace, найповільніший // - System.gc(), або коли Major не допоміг // - тривалий STW — уникай!
System.gc() — лише підказка JVM, не гарантія. У продакшн його не використовують. GC Overhead Limit Exceeded: JVM витрачає >98% часу на GC і звільняє <2% heap.
stack

Stack і Stack Frame

Кожен потік має свій стек. При кожному виклику методу — новий Stack Frame. При поверненні — frame знімається. Переповнення → StackOverflowError.

// Що зберігається в Stack Frame: // - Локальні змінні (primitives + references) // - Operand stack (для обчислень) // - Frame data (constant pool ref, exception table) // Приклад: що де живе void method() { int x = 42; // у Stack Frame (primitive) String s = "hello"; // посилання в Stack, об'єкт в Heap Object o = new Object(); // посилання в Stack, об'єкт в Heap } // StackOverflowError: нескінченна рекурсія void infinite() { infinite(); } // StackOverflowError // Збільшити стек потоку new Thread(null, task, "name", 4 * 1024 * 1024); // 4MB
Stack vs Heap: примітивні типи (int, long, double...) зберігаються у Stack Frame як значення. Об'єкти — завжди в Heap, у Stack лише посилання. Але Java не гарантує де саме зберігається примітив — JIT може помістити його в регістр CPU.
metaspace

Metaspace (Java 8+)

Замінив PermGen. Зберігає метадані класів: bytecode, constant pool, method descriptors, annotations. Розміщується у native memory (поза heap).

// PermGen (до Java 8) — фіксований розмір, часто OOM: // java.lang.OutOfMemoryError: PermGen space -XX:MaxPermSize=256m // застарілий флаг // Metaspace (Java 8+) — росте динамічно: // java.lang.OutOfMemoryError: Metaspace -XX:MetaspaceSize=64m // initial committed -XX:MaxMetaspaceSize=256m // ліміт (за замовч. необмежений!) // Що займає Metaspace: // - Class metadata (bytecode, vtable) // - Constant pool // - Method descriptors // - Annotations // - НЕ: String literals (вони в Heap, String Pool)
Без -XX:MaxMetaspaceSize Metaspace необмежений — може з'їсти всю нативну пам'ять процесу. Завжди встановлюй ліміт у продакшн. Metaspace leak — симптом: класи не вивантажуються, зазвичай через ClassLoader leak.

String Pool (String Interning)

// String literals — автоматично інтернуються String a = "hello"; // з String Pool String b = "hello"; // той самий об'єкт a == b; // true (same reference) String c = new String("hello"); // новий об'єкт в heap a == c; // false a.equals(c); // true // String Pool в Heap (Java 7+, раніше був у PermGen) c.intern(); // додає в pool або повертає існуючий
object layout

Розташування об'єкта в heap

Кожен Java об'єкт в heap має фіксований header + поля. Header містить mark word (hashcode, lock state, GC age) і class pointer.

// Object header (64-bit JVM, без CompressedOops): // Mark word: 8 байт (hashcode, lock, GC age) // Class pointer: 8 байт (посилання на Class метадані) // ────────────────────────────────────────────── // Мінімум об'єкт: 16 байт (тільки header, без полів) // З CompressedOops (-XX:+UseCompressedOops, default on <32GB heap): // Class pointer: 4 байт → header = 12 байт // + padding до кратного 8 // Приклад: Integer object // header: 12 байт // int value: 4 байт // ───────────────── // Разом: 16 байт (vs 4 байти для int примітиву!)
Boxing overhead: int = 4 байти, Integer = 16 байт. При мільйонах елементів різниця між int[] і Integer[] — в 4 рази по пам'яті, плюс GC pressure.
references

Типи посилань

Java має 4 типи посилань — відрізняються тим коли GC може зібрати об'єкт.

// Strong: звичайне посилання — GC не чіпає Object obj = new Object(); // strong reference // SoftReference: збирається при нестачі пам'яті // Ідеально для memory-sensitive cache SoftReference<Image> ref = new SoftReference<>(image); Image img = ref.get(); // null якщо зібрано // WeakReference: збирається при наступному GC // Використовується в WeakHashMap WeakReference<Object> weak = new WeakReference<>(obj); // PhantomReference: збирається, але в ReferenceQueue // Для cleanup після збору (як finalizer, але краще) ReferenceQueue<Object> queue = new ReferenceQueue<>(); PhantomReference<Object> phantom = new PhantomReference<>(obj, queue);
WeakHashMap — map де ключі є WeakReference. При відсутності strong посилань на ключ — entry автоматично видаляється. Типовий use case: cache де ключ — це сам об'єкт.

Корисні флаги і інструменти

// Логування GC -Xlog:gc*:file=gc.log:time,uptime,level,tags -XX:+PrintGCDetails // застарілий, але досі зустрічається // Heap dump при OOM -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/heapdump.hprof // Показати всі JVM флаги -XX:+PrintFlagsFinal // Діагностичні команди (jcmd) jcmd <pid> VM.flags // поточні флаги jcmd <pid> GC.heap_info // стан heap jcmd <pid> GC.run // викликати GC jcmd <pid> Thread.print // всі потоки jcmd <pid> VM.native_memory // native memory (NMT) // jstat — моніторинг GC в реальному часі jstat -gcutil <pid> 1000 // кожну секунду // S0 S1 E O M CCS YGC YGCT FGC FGCT GCT // 0 87 42 65 94 90 15 0.3 2 0.5 0.8
На співбесіді: heap = Young(Eden+S0+S1) + Old, metaspace = native (не heap), object header = 12-16 байт, Strong/Soft/Weak/Phantom references і коли GC збирає кожен. Boxing overhead: Integer = 4x більше ніж int.
