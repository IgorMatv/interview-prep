behavioural

Strategy

Виносимо алгоритм в окремий об'єкт. Дозволяє міняти поведінку в runtime без зміни контексту. Замінює if/else або switch на поліморфізм.

// Інтерфейс стратегії interface RoutingStrategy { List<Stop> optimize(List<Stop> stops); } // Реалізації class NearestNeighborStrategy implements RoutingStrategy { public List<Stop> optimize(List<Stop> stops) { ... } } class TwoOptStrategy implements RoutingStrategy { public List<Stop> optimize(List<Stop> stops) { ... } } // Контекст — не знає деталей реалізації class RouteOptimizer { private RoutingStrategy strategy; RouteOptimizer(RoutingStrategy strategy) { this.strategy = strategy; } List<Stop> optimize(List<Stop> stops) { return strategy.optimize(stops); } }
У твоєму проекті: RoutingStrategy — саме цей патерн. NearestNeighborStrategy і TwoOptStrategy — різні стратегії. RouteOptimizer — контекст.
behavioural

Template Method

Базовий клас визначає скелет алгоритму, підкласи реалізують конкретні кроки. Схожий на Strategy але через спадкування.

abstract class DataProcessor { // Template method — фіксований алгоритм public final void process() { List<Record> data = readData(); // abstract List<Record> filtered = filter(data); // abstract writeResult(filtered); // abstract } protected abstract List<Record> readData(); protected abstract List<Record> filter(List<Record> data); protected abstract void writeResult(List<Record> data); } // Spring JdbcTemplate — класичний Template Method // jdbcTemplate.query() → manages connection/statement/resultset // ти реалізуєш тільки RowMapper
behavioural

Observer

Publisher (subject) повідомляє Subscribers (observers) про зміни. Слабка зв'язність — publisher не знає про конкретних observers.

// Spring ApplicationEvent — вбудований Observer // Publisher: @Service class RouteService { @Autowired ApplicationEventPublisher publisher; public void createRoute(Route route) { routeRepo.save(route); publisher.publishEvent(new RouteCreatedEvent(route)); } } // Subscriber: @Component class RouteNotificationListener { @EventListener void onRouteCreated(RouteCreatedEvent event) { sendNotification(event.getRoute()); } } // @TransactionalEventListener — після commit транзакції @TransactionalEventListener(phase = AFTER_COMMIT)
creational

Builder

Покроково будуємо складний об'єкт. Вирішує проблему telescoping constructor (конструктор з багатьма опціональними параметрами).

// Lombok @Builder — генерує автоматично @Builder @Data class RouteRequest { private List<Long> poiIds; private LocalTime startTime; private int maxDurationMinutes; private RoutingStrategy strategy; } // Використання RouteRequest req = RouteRequest.builder() .poiIds(List.of(1L, 2L, 3L)) .startTime(LocalTime.of(9, 0)) .maxDurationMinutes(240) .build(); // Ручна реалізація (якщо не Lombok): public static class Builder { private List<Long> poiIds; // ... public Builder poiIds(List<Long> ids) { this.poiIds = ids; return this; } public RouteRequest build() { return new RouteRequest(this); } }
creational

Factory Method

Базовий клас/інтерфейс визначає метод створення об'єкту, підкласи вирішують який саме. Делегує інстанціювання.

// Factory Method через інтерфейс interface NotificationFactory { Notification create(String message); } class EmailNotificationFactory implements NotificationFactory { public Notification create(String msg) { return new EmailNotification(msg); } } // Static factory method (Effective Java pattern) class Connection { public static Connection of(String url) { ... } public static Connection inMemory() { ... } private Connection() {} // конструктор прихований } // Spring: @Bean метод = Factory Method @Bean public RoutingStrategy routingStrategy() { return new TwoOptStrategy(haversine); }
structural

Proxy

Об'єкт-замінник що контролює доступ до реального об'єкту. Spring AOP, @Transactional, @Cacheable — всі через proxy.

// JDK Dynamic Proxy: тільки для інтерфейсів MyService proxy = (MyService) Proxy.newProxyInstance( MyService.class.getClassLoader(), new Class[]{MyService.class}, (p, method, args) -> { System.out.println("before: " + method.getName()); Object result = method.invoke(realService, args); System.out.println("after: " + method.getName()); return result; } ); // CGLIB Proxy: для класів (Spring використовує коли немає інтерфейсу) // Створює підклас і override методи // Обмеження: final класи і методи не можна проксіювати
Self-invocation пастка: виклик методу всередині того самого класу обходить proxy. this.annotatedMethod() — анотація (@Transactional, @Cacheable) не спрацює. Потрібен ctx.getBean(MyService.class).annotatedMethod().
structural

Decorator і Facade

Decorator

Динамічно додає поведінку об'єкту не змінюючи клас. Реалізує той самий інтерфейс і делегує + додає.

// InputStream stack — класичний Decorator new BufferedReader( new InputStreamReader( new GZIPInputStream( new FileInputStream("file.gz")))); // Свій Decorator class LoggingPoiService implements PoiService { private final PoiService delegate; public List<Poi> findNearby(Location loc) { log.info("findNearby called"); List<Poi> result = delegate.findNearby(loc); log.info("found: {}", result.size()); return result; } }

Facade

Спрощує складну підсистему через єдину точку входу. Controller у MVC — це Facade.

// Facade: один метод замість виклику кількох сервісів class RouteOrchestrator { // Facade public RouteResult createOptimizedRoute(RouteRequest req) { List<Poi> pois = poiService.findByIds(req.getPoiIds()); Route route = optimizer.optimize(pois, req); Route saved = routeService.save(route); eventPublisher.publishEvent(new RouteCreatedEvent(saved)); return mapper.toResult(saved); } }
На співбесіді: Strategy vs Template Method (composition vs inheritance), Builder vs Constructor (telescoping), Proxy = Spring AOP (@Transactional, @Cacheable), Decorator vs Proxy (обидва обгортають, але Decorator додає, Proxy контролює доступ).
