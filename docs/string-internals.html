immutability

Immutability і String Pool

String — immutable: кожна операція що "змінює" рядок створює новий об'єкт. String Pool (Heap) зберігає інтерновані рядки — literal однакового значення → один об'єкт.

// String literal → автоматично в String Pool String a = "hello"; // Pool String b = "hello"; // той самий об'єкт з Pool a == b; // true String c = new String("hello"); // новий об'єкт у heap (не Pool) a == c; // false a.equals(c); // true // Чому immutable? // 1. Thread-safe без синхронізації // 2. Безпечний ключ HashMap (hashCode не змінюється) // 3. String Pool можливий тільки через immutability // 4. Security: не можна змінити URL, ім'я класу після перевірки
порівняння

== vs equals, intern()

// НІКОЛИ не порівнюй String через == String s1 = new String("test"); String s2 = new String("test"); s1 == s2; // false (різні об'єкти) s1.equals(s2); // true (однаковий вміст) // intern(): додає в Pool або повертає існуючий String s3 = s1.intern(); String s4 = s2.intern(); s3 == s4; // true (обидва з Pool) // Конкатенація compile-time → в Pool String x = "hel" + "lo"; // компілятор оптимізує в "hello" → Pool x == a; // true // Runtime конкатенація → НЕ в Pool String part = "hel"; String y = part + "lo"; // runtime → new String, не Pool y == a; // false
Класичне питання-пастка на співбесіді. Відповідь завжди: порівнюй рядки через equals(), а ще краще — Objects.equals(a, b) (null-safe).
Java 9+

Compact Strings (Java 9+)

До Java 9: String зберігав char[] — 2 байти на символ. Java 9+: якщо рядок містить тільки Latin-1 символи → byte[] (1 байт/символ). Економія ~50% пам'яті для ASCII.

// До Java 9 private final char[] value; // завжди 2 байти/символ // "hello" = 10 байт + header // Java 9+ Compact Strings private final byte[] value; // 1 або 2 байти залежно від CODER private final byte coder; // LATIN1=0 або UTF16=1 // "hello" → LATIN1 → 5 байт (2x менше) // "Привіт" → UTF16 → 12 байт (як раніше) // Прозоро для програміста — API не змінився // Вмикається автоматично, -XX:-CompactStrings вимикає
конкатенація

StringBuilder vs StringBuffer

// StringBuilder: mutable, НЕ thread-safe, швидкий StringBuilder sb = new StringBuilder(); sb.append("Hello").append(", ").append("World"); String result = sb.toString(); // StringBuffer: mutable, thread-safe (synchronized), повільніший // Майже не використовується в сучасному коді // Якщо потрібна thread-safety → StringJoiner або Stream joining // Корисні методи StringBuilder: sb.insert(0, "prefix"); // вставити на позицію sb.delete(2, 5); // видалити діапазон sb.reverse(); // перевернути sb.replace(0, 2, "XX"); // замінити діапазон sb.length(); // поточна довжина sb.setLength(0); // очистити (reuse об'єкту)
Коли StringBuilder: циклічна конкатенація (більше 3-4 операцій). Для "a" + "b" + "c" в одному виразі — компілятор сам використовує StringBuilder або String.concat.
під капотом

+ оператор — що генерує компілятор

// Java 8: + розкривається в StringBuilder String s = a + b + c; // → new StringBuilder().append(a).append(b).append(c).toString() // Java 9+: invokedynamic (StringConcatFactory) // Ефективніше, але принцип той самий // ПОГАНО: + в циклі = O(n²) алокацій String result = ""; for (String s : list) { result += s; // кожна ітерація → новий StringBuilder + toString } // ДОБРЕ: один StringBuilder StringBuilder sb = new StringBuilder(); for (String s : list) sb.append(s); String result = sb.toString(); // АБО через Stream String result = String.join(", ", list); String result = list.stream().collect(Collectors.joining(", "));
API

String методи — важливі для співбесіди

// Порівняння s.equals(other) // content equality s.equalsIgnoreCase(other) s.compareTo(other) // lexicographic (для sort) s.startsWith("pre") / endsWith("suf") s.contains("sub") // Трансформація s.toUpperCase() / toLowerCase() s.trim() // видалити leading/trailing whitespace s.strip() // Java 11+, Unicode-aware (краще) s.stripLeading() / stripTrailing() s.replace("old", "new") // literal replace s.replaceAll("regex", "new") // regex replace // Розбиття і аналіз s.split(",") // split за regex s.split(",", -1) // зберігає trailing empty strings s.charAt(i) // символ за індексом s.indexOf("sub") // -1 якщо не знайдено s.substring(from, to) // [from, to) // Java 11+ s.isBlank() // "" або тільки whitespace s.lines() // Stream<String> по рядках s.repeat(3) // "ab".repeat(3) → "ababab" // Форматування String.format("%s has %d items", name, count) // старий спосіб "%s has %d items".formatted(name, count) // Java 15+
антипатерни

Типові помилки

// 1. == для порівняння if (status == "ACTIVE") { ... } // WRONG: може не спрацювати if ("ACTIVE".equals(status)) { ... } // OK: константа зліва → null-safe // 2. + в циклі String result = ""; for (int i = 0; i < 1000; i++) result += i; // O(n²)! // 3. split без ліміту на керовані вхідні дані "a,,b".split(",") // → ["a", "", "b"] "a,,b,".split(",") // → ["a", "", "b"] — trailing пустий відкидається! "a,,b,".split(",", -1) // → ["a", "", "b", ""] — зберігається // 4. substring довгого рядка і утримання всього // До Java 7: substring розділяв char[], тримав посилання на великий масив // Java 7+: substring копіює → OK
На співбесіді: String immutable (thread-safe, hashCode стабільний, pool), == порівнює посилання а не вміст, StringBuilder для циклічної конкатенації, Compact Strings з Java 9 (1 байт для Latin-1).
