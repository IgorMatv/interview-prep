ConcurrentHashMap — як влаштований

Java 8+: масив Node[] (як HashMap), але синхронізація на рівні окремих бакетів через synchronized(bin) і CAS для порожніх бакетів. Немає глобального lock.

// Java 8+ — синхронізація per-bucket // Порожній бакет: CAS (lock-free) // Зайнятий бакет: synchronized(firstNode) // Concurrency level: ефективно = кількість бакетів // При 16 бакетах — 16 потоків можуть писати паралельно // size() — не точна при concurrent updates // використовує LongAdder-подібний механізм map.size(); // ~точна map.mappingCount(); // long, точніша для великих мап
На відміну від Java 7 (сегменти), Java 8 CHM не має фіксованого concurrency level. Lock granularity = кожен бакет окремо. Це дає кращий throughput.
атомарні операції

Атомарні операції CHM

ConcurrentHashMap має атомарні методи для типових concurrent патернів. Використовуй їх замість ручної синхронізації.

// putIfAbsent: атомарний check-then-act map.putIfAbsent(key, value); // повертає null якщо додано, або існуюче значення // computeIfAbsent: lazy ініціалізація map.computeIfAbsent(key, k -> expensiveCompute(k)); // функція викликається тільки якщо ключа немає // АТОМАРНА — гарантія одноразового виклику для ключа // computeIfPresent: оновити тільки якщо є map.computeIfPresent(key, (k, v) -> v + 1); // compute: create or update map.compute(key, (k, v) -> v == null ? 1 : v + 1); // merge: combine або put map.merge(key, 1, Integer::sum); // якщо ключа немає → put(key, 1) // якщо є → put(key, oldValue + 1) // replace: атомарний conditional update map.replace(key, oldValue, newValue); // CAS для values
merge(key, 1, Integer::sum) — найелегантніший спосіб атомарного лічильника в Map. Жодного synchronized не потрібно.

Project Loom — навіщо Virtual Threads

Platform thread = OS thread (~1MB stack, контекстний перемикач дорогий). Мільйон потоків = OOM. Virtual thread — легкий потік JVM (~handful of KB), може бути мільйони.

Virtual Threads не роблять код швидшим — вони усувають scalability bottleneck при IO-bound навантаженні. Thread-per-request модель стає практичною.
API

Створення Virtual Threads

// Thread.ofVirtual() Thread vt = Thread.ofVirtual() .name("worker-", 0) // ім'я з автоінкрементом .start(() -> doWork()); // ExecutorService з virtual threads try (ExecutorService exec = Executors.newVirtualThreadPerTaskExecutor()) { for (int i = 0; i < 1_000_000; i++) { exec.submit(() -> handleRequest()); } } // try-with-resources: чекає завершення всіх // Перевірити чи virtual Thread.currentThread().isVirtual(); // → true // Virtual threads = daemon threads за замовчуванням // JVM не чекає їх завершення автоматично
Virtual thread при блокуючому IO (socket read, DB query) unmounts від OS потоку, звільняє його для інших VT, і remounts коли IO завершується. Це і є магія Loom.
обмеження

Pinning і обмеження Virtual Threads

Pinning — virtual thread не може unmount від carrier (OS) thread. Це блокує весь carrier.

// Pinning відбувається при: // 1. synchronized блок/метод synchronized(this) { httpClient.get(url); // pinned! VT не відпускає carrier } // 2. native методи (JNI) // 3. Foreign Function calls // Рішення: замінити synchronized на ReentrantLock private final ReentrantLock lock = new ReentrantLock(); lock.lock(); try { httpClient.get(url); // VT може unmount під час IO } finally { lock.unlock(); }
Виявляти pinning: -Djdk.tracePinnedThreads=full або JFR event jdk.VirtualThreadPinned. Spring Boot 3.2+ автоматично конфігурується для VT без pinning.

На співбесіді: CHM — per-bucket lock, CAS для порожніх бакетів, compute/merge/putIfAbsent. Virtual Threads — unmount при IO, pinning при synchronized, мільйони паралельних задач.
