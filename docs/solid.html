Порушення і сигнали
SOLID у Spring проекті
S

Single Responsibility Principle

Клас повинен мати тільки одну причину для зміни. Один клас — одна відповідальність. Не "один метод", а одна зона відповідальності.

порушення

class UserService { // бізнес-логіка createUser(dto) { ... } // email — не його справа sendWelcomeEmail() { ... } // PDF — не його справа generateReport() { ... } }

добре

class UserService { createUser(dto) { ... } } class EmailService { sendWelcome(user) { ... } } class ReportService { generatePdf(user) { ... } }
Практичний тест: якщо можеш описати що робить клас без слова "і" — SRP виконано. "UserService реєструє користувачів і надсилає email і генерує звіти" — порушення.
O

Open/Closed Principle

Відкритий для розширення, закритий для модифікації. Новий функціонал — через додавання нових класів, а не зміну існуючих.

порушення — switch/if

class NotificationSender { send(String type, msg) { if (type.equals("email")) sendEmail(msg); else if (type.equals("sms")) sendSms(msg); // нова нотифікація → правимо } }

добре — поліморфізм

interface Notification { void send(String msg); } class EmailNotification implements Notification { ... } class SmsNotification implements Notification { ... } // нова → новий клас
OCP — це і є Strategy патерн. Замінюємо if/else на ієрархію класів з поліморфізмом.
L

Liskov Substitution Principle

Підклас має бути взаємозамінним з батьківським класом. Код що використовує базовий тип має працювати з будь-яким підтипом без змін і без сюрпризів.

// Класичне порушення LSP: Rectangle → Square class Rectangle { protected int width, height; void setWidth(int w) { this.width = w; } void setHeight(int h) { this.height = h; } int area() { return width * height; } } class Square extends Rectangle { @Override void setWidth(int w) { this.width = this.height = w; } // ламає очікування! } // Тест що проходить для Rectangle, падає для Square: Rectangle r = new Square(); r.setWidth(4); r.setHeight(5); assert r.area() == 20; // провал! Square дає 25
LSP порушується коли підклас підсилює передумови (вимагає більше від caller) або послаблює постумови (гарантує менше). Рішення: composition замість inheritance або переосмислити ієрархію.
I

Interface Segregation Principle

Клієнт не повинен залежати від методів які він не використовує. Великий інтерфейс — розбити на менші і специфічні.

товстий інтерфейс

interface Worker { work(); eat(); sleep(); } class Robot implements Worker { work() { ... } eat() { throw new UOE(); } // ! sleep() { throw new UOE(); } // ! }

розділені інтерфейси

interface Workable { work(); } interface Feedable { eat(); } interface Sleepable { sleep(); } class Human implements Workable, Feedable, Sleepable {} class Robot implements Workable {}
Spring Data: JpaRepository extends PagingAndSortingRepository extends CrudRepository extends Repository. Можна залежати від мінімального інтерфейсу — класичний ISP.
D

Dependency Inversion Principle

Залежати від абстракцій, не від конкретних реалізацій. High-level модулі не повинні залежати від low-level. Обидва залежать від абстракцій.

порушення

class RouteService { // залежність від конкретного private MySQLRouteRepo repo = new MySQLRouteRepo(); // не можна замінити без правки }

правильно

class RouteService { // залежність від абстракції private final RouteRepository repo; RouteService(RouteRepository r) { this.repo = r; // DI } }
Spring DI реалізує DIP: всі залежності через інтерфейси, Spring сам вирішує яку конкретну реалізацію підставити. Тести легко замінюють реалізацію на mock.
сигнали

Сигнали порушення SOLID

  • SRP: клас "God object" — знає і робить все. Метод на 200 рядків. "і" у описі класу.
  • OCP: switch/if по типу об'єкту. При додаванні нового типу — правиш існуючий код у кількох місцях.
  • LSP: UnsupportedOperationException в реалізації методу інтерфейсу. instanceof перед використанням підтипу.
  • ISP: реалізація кидає UnsupportedOperationException або залишає методи порожніми. "Stub" реалізації.
  • DIP: new ConcreteClass() всередині бізнес-логіки. Неможливо написати unit test без реальної бази даних.
практика

SOLID у Spring проекті

// SRP: кожен шар має одну відповідальність @RestController // HTTP mapping тільки @Service // бізнес-логіка тільки @Repository // DB доступ тільки // OCP: нові стратегії без зміни існуючого коду interface RoutingStrategy { ... } @Component class NearestNeighbor implements RoutingStrategy { ... } @Component class TwoOpt implements RoutingStrategy { ... } // нова стратегія → новий @Component // DIP: залежність від інтерфейсів через @Autowired @Autowired PoiRepository repo; // не JpaPoiRepository @Autowired RouteService svc; // не RouteServiceImpl // ISP: Spring Data дозволяє залежати від мінімального interface PoiSummaryRepository extends Repository<Poi, Long> { List<Poi> findByCity(String city); // тільки потрібні методи }
На співбесіді: SRP — одна причина для зміни (не один метод), OCP через Strategy/поліморфізм, LSP — Rectangle/Square антиприклад, ISP — товсті інтерфейси ламають клієнтів, DIP — залежати від абстракцій = Spring DI.
