Token Bucket — найпопулярніший для API rate limiting. Дозволяє burst (короткочасний пік) при незаповненому bucket. AWS API Gateway, Stripe — використовують token bucket.

":" .. math.floor(time/60) local count = redis.call("INCR", key) if count == 1 then redis.call("EXPIRE", key, 60) -- 60 секунд end return count // Sliding Window через ZSET // Score = timestamp, member = request_id // Видаляємо старі: ZREMRANGEBYSCORE key 0 (now-window) // Рахуємо: ZCARD key // Додаємо: ZADD key now request_id // Spring: Bucket4j або Resilience4j rate limiter // Bucket4j підтримує Redis backend
Distributed rate limiting (кілька нод) — потрібен shared state (Redis). Локальний rate limiter на кожній ноді дасть N×limit ефективно. При 10 інстансах і ліміті 100 req/min → 1000 req/min реально.
caching

Caching стратегії (системний рівень)

// Рівні кешування (від клієнта до DB): // 1. Browser cache (HTTP Cache-Control) // 2. CDN (Cloudflare, CloudFront) // 3. API Gateway cache // 4. Application cache (Redis) // 5. DB query cache (L2 Hibernate) // 6. DB buffer pool (shared_buffers) // Cache invalidation strategies: // TTL: простий, може бути stale // Event-based: точний, складніше // Write-through: завжди актуальний, overhead // Write-behind: async, ризик втрати // HTTP Cache-Control Cache-Control: max-age=3600, public // 1 год, кешувати скрізь Cache-Control: max-age=0, must-revalidate // перевіряти кожен раз Cache-Control: no-store // не кешувати (особисті дані) ETag: "abc123" // conditional request
Cache hit ratio > 90% — хороший результат. 99%+ — відмінно. Нижче 80% — кеш не ефективний або TTL занадто короткий.
CDN

CDN і edge caching

Content Delivery Network — географічно розподілені edge servers. Запит іде до найближчого edge замість origin server.

  • Статика: images, JS, CSS, fonts — основний use case. Cache раз, роздавай з edge назавжди.
  • Dynamic CDN: Cloudflare Workers, CloudFront Functions — виконання на edge.
  • API caching: GET endpoints з Cache-Control → кешується на edge.
// Cache invalidation при deploy // Стратегія 1: hash у filename (Webpack, Vite) // bundle.a3f9b2.js → browser cache forever // при зміні → новий hash → новий URL → cache miss // Стратегія 2: explicit invalidation // aws cloudfront create-invalidation --paths "/*" // Стратегія 3: versioned URL // /api/v2/routes → старий URL не ламається
consistent hashing

Consistent Hashing

Розподіл ключів між нодами так що при додаванні/видаленні ноди перерозподіляється мінімум ключів.

// Звичайний hash: key % N серверів // При зміні N → переміщуються майже всі ключі → cache miss // Consistent hashing: // 1. Кільце від 0 до 2^32 // 2. Сервери розміщуються на кільці за hash(server_id) // 3. Ключ → hash(key) → рухаємось по кільцю → перший сервер // 4. При додаванні сервера: тільки ключі між попереднім і новим переміщуються // 5. При видаленні: ключі переходять до наступного // Virtual nodes: кожен сервер = K точок на кільці // Більш рівномірний розподіл // Потужніший сервер = більше virtual nodes // Використовується в: // - Cassandra (розподіл даних по нодах) // - Redis Cluster (hash slots) // - CDN (вибір edge server) // - Distributed caches
