optimistic locking

@Version — оптимістичне блокування

Без блокування рядку в DB. При update — перевіряємо що version не змінилась. Якщо хтось встиг раніше — OptimisticLockException.

@Entity class Account { @Id private long id; private BigDecimal balance; @Version // Hibernate управляє автоматично private int version; } // Hibernate генерує: // UPDATE accounts SET balance=?, version=2 WHERE id=1 AND version=1 // Якщо version вже 2 (хтось змінив) → 0 rows affected → exception
Коли використовувати: read-heavy з рідкісними конфліктами. Не блокує рядки → висока паралельність. При конфлікті — retries або повідомлення користувачу. Підходить для більшості web-застосунків.

Обробка OptimisticLockException

@Retryable(retryFor = OptimisticLockingFailureException.class, maxAttempts = 3) @Transactional public void transferFunds(long fromId, long toId, BigDecimal amount) { Account from = repo.findById(fromId).orElseThrow(); Account to = repo.findById(toId).orElseThrow(); from.debit(amount); to.credit(amount); } // @Retryable з spring-retry — простий retry при конфлікті
pessimistic locking

Pessimistic locking

Блокує рядок в DB на час транзакції. Підходить при частих конфліктах або critical sections. SELECT ... FOR UPDATE.

// Spring Data @Lock(LockModeType.PESSIMISTIC_WRITE) Optional<Account> findById(Long id); // → SELECT * FROM accounts WHERE id=? FOR UPDATE // LockModeType варіанти PESSIMISTIC_READ // SELECT FOR SHARE — дозволяє паралельні reads PESSIMISTIC_WRITE // SELECT FOR UPDATE — ексклюзивний // Через EntityManager em.find(Account.class, id, LockModeType.PESSIMISTIC_WRITE); // З таймаутом (PostgreSQL) Map<String, Object> hints = Map.of( "javax.persistence.lock.timeout", 5000 // 5 секунд ); em.find(Account.class, id, LockModeType.PESSIMISTIC_WRITE, hints);
Pessimistic locking може призвести до deadlock якщо два потоки захоплюють локи в різному порядку. Завжди набувай локи в фіксованому порядку (наприклад, за id).
cascade

CascadeType

Визначає які операції з parent entity каскадуються на child entities.

CascadeTypeЩо каскадує
PERSISTem.persist(parent) → persist(children)
MERGEem.merge(parent) → merge(children)
REMOVEem.remove(parent) → remove(children)
REFRESHem.refresh(parent) → refresh(children)
DETACHem.detach(parent) → detach(children)
ALLвсі вище разом
@Entity class Order { @OneToMany(mappedBy = "order", cascade = CascadeType.ALL) private List<Item> items; } // persist(order) → автоматично persist(items) Order order = new Order(); order.addItem(new Item("Book")); repo.save(order); // Item теж збережеться
CascadeType.REMOVE при @ManyToMany — небезпечно. Видалення одного entity може видалити entities що шеряться між іншими. Для ManyToMany зазвичай тільки PERSIST і MERGE.
orphanRemoval

orphanRemoval

Автоматично видаляє child entity якщо вона більше не має parent. Доповнює CascadeType.REMOVE але для disconnection, не delete parent.

@OneToMany(mappedBy = "order", cascade = CascadeType.ALL, orphanRemoval = true) private List<Item> items; // Видалити item з колекції → DELETE SQL Order order = repo.findById(orderId).get(); order.getItems().remove(item); // orphan → DELETE FROM items WHERE id=? // CascadeType.REMOVE vs orphanRemoval: // REMOVE: видаляємо PARENT → видаляються children // orphanRemoval: прибираємо child з колекції → видаляється child
orphanRemoval доступний тільки для @OneToOne і @OneToMany — де child "належить" одному parent. Для @ManyToMany — концептуально не підходить.

// Увімкнути L2 cache spring.jpa.properties.hibernate.cache.use_second_level_cache=true spring.jpa.properties.hibernate.cache.region.factory_class=... // @Cache на entity @Entity @Cache(usage = CacheConcurrencyStrategy.READ_WRITE) class Category { ... } // рідко змінюється → хороший кандидат // @Cache на колекцію @OneToMany @Cache(usage = CacheConcurrencyStrategy.READ_ONLY) private Set<Tag> tags;
L2 cache + розподілена система — небезпека stale data. При UPDATE через JDBC (нативний SQL, не HQL) — Hibernate не знає про зміну → L2 cache не інвалідується. Тільки через @Modifying query або явна інвалідація.
Spring Data

Spring Data Specifications

Динамічні запити через JPA Criteria API без написання рядків JPQL. Compositional — можна комбінувати через and(), or().

// Repository extends JpaSpecificationExecutor interface OrderRepo extends JpaRepository<Order, Long>, JpaSpecificationExecutor<Order> {} // Specification = один предикат class OrderSpecs { static Specification<Order> hasStatus(String status) { return (root, query, cb) -> cb.equal(root.get("status"), status); } static Specification<Order> totalGreaterThan(BigDecimal min) { return (root, query, cb) -> cb.greaterThan(root.get("total"), min); } } // Комбінування Specification<Order> spec = OrderSpecs.hasStatus("PAID") .and(OrderSpecs.totalGreaterThan(new BigDecimal("100"))); repo.findAll(spec, PageRequest.of(0, 20));
Querydsl

Querydsl — type-safe альтернатива

Генерує Q-класи з entity. Type-safe запити — помилки при компіляції, автодоповнення IDE.

// Генерований Q-клас (APT) // QOrder.order — для Order entity // Repository extends QuerydslPredicateExecutor interface OrderRepo extends JpaRepository<Order, Long>, QuerydslPredicateExecutor<Order> {} // Запит QOrder order = QOrder.order; BooleanExpression predicate = order.status.eq("PAID") .and(order.total.gt(new BigDecimal("100"))) .and(order.createdAt.after(LocalDateTime.now().minusDays(30))); repo.findAll(predicate);
Specifications vs Querydsl: Specifications — стандарт Spring Data, менше залежностей. Querydsl — краще IDE підтримка, type-safe, але потрібна APT генерація. Для простих запитів — Specifications достатньо.
На співбесіді: @Version = version column, UPDATE WHERE version=old, OptimisticLockException. CascadeType.REMOVE vs orphanRemoval (delete parent vs remove from collection). L1 = PC scope, L2 = SessionFactory scope. Specifications для динамічних запитів.
