архітектура

Topic, partition, offset

Kafka зберігає повідомлення в topics. Topic поділений на partitions — впорядковані, незмінні логи. Кожне повідомлення в partition має offset — монотонно зростаючий номер.

// Topic: "route-events" // ┌─ Partition 0 ──────────────────────────────┐ // │ [0] [1] [2] [3] [4] [5] ... │ // └────────────────────────────────────────────┘ // ┌─ Partition 1 ──────────────────────────────┐ // │ [0] [1] [2] [3] ... │ // └────────────────────────────────────────────┘ // ┌─ Partition 2 ──────────────────────────────┐ // │ [0] [1] ... │ // └────────────────────────────────────────────┘ // Ключові властивості: // Порядок гарантований ВСЕРЕДИНІ partition // Між partitions — порядок НЕ гарантований // Retention: повідомлення зберігаються N днів (default 7) // Consumer може читати з будь-якого offset
Кількість partitions = рівень паралелізму. Один consumer group може читати максимум 1 consumer на partition. 3 partitions → max 3 паралельних consumers. Збільшити partitions можна, зменшити — ні (без recreation topic).
consumer groups

Consumer groups і rebalancing

Consumer group — логічна група consumers що читають один topic. Кожна partition призначається рівно одному consumer в групі. Різні groups — незалежні (кожна читає всі messages).

// 3 partitions, group "route-processor": // consumer-1 → partition 0 // consumer-2 → partition 1 // consumer-3 → partition 2 // Якщо consumer-2 впав → rebalancing: // consumer-1 → partition 0, 1 // consumer-3 → partition 2 // Якщо consumers > partitions: // consumer-4 → idle (немає partition) // Не має сенсу більше consumers ніж partitions // Незалежні groups: // "route-processor" → читає topic незалежно // "route-analytics" → читає той самий topic з початку
Rebalancing — STW для споживачів. Під час rebalancing всі consumers в group зупиняються. Minimize rebalancing: збільш session.timeout.ms, зменш heartbeat.interval.ms, використовуй Cooperative Sticky Assignor (Kafka 2.4+).

Producer: partitioning і acks

Як вибирається partition

// 1. Якщо є key: hash(key) % numPartitions // Один key → завжди одна partition → гарантований порядок для key producer.send(new ProducerRecord<>("topic", userId.toString(), event)); // 2. Без key: round-robin між partitions producer.send(new ProducerRecord<>("topic", event)); // 3. Explicit partition producer.send(new ProducerRecord<>("topic", 0, key, event));

acks — acknowledgment від broker

// acks=0: fire and forget, найшвидший, може загубити spring.kafka.producer.acks=0 // acks=1: лідер отримав, але replica може не встигнути spring.kafka.producer.acks=1 // acks=all (-1): всі ISR (in-sync replicas) підтвердили // найбезпечніший, трохи повільніший spring.kafka.producer.acks=all spring.kafka.producer.retries=3 spring.kafka.producer.properties.enable.idempotence=true
acks=all + enable.idempotence=true — безпечний default для продакшн producer. Idempotence гарантує exactly-once на рівні producer → broker (без дублікатів при retry).

// At-least-once: commit після обробки // spring.kafka.consumer.enable-auto-commit=false (default false) @KafkaListener(topics = "route-events") void handle(RouteEvent event, Acknowledgment ack) { process(event); // якщо впало тут → повторна обробка ack.acknowledge(); // commit offset тільки після успіху } // При падінні після process() але до ack → дублікат // Рішення: idempotent consumer (перевіряємо чи вже оброблено)
exactly-once

Exactly-once semantics (EOS)

Справжній exactly-once: повідомлення обробляється рівно один раз, навіть при retry і failover. Вимагає transactional producer + transactional consumer або idempotent consumer.

// Варіант 1: Idempotent consumer (найпростіше) @KafkaListener(topics = "route-events") @Transactional void handle(RouteEvent event, Acknowledgment ack) { if (processedEventRepo.exists(event.getId())) { ack.acknowledge(); return; // вже оброблено, ігноруємо } process(event); processedEventRepo.save(event.getId()); // позначаємо як оброблено ack.acknowledge(); } // Варіант 2: Kafka Transactions (read-process-write) spring.kafka.consumer.isolation-level=read_committed spring.kafka.producer.transaction-id-prefix=route-tx- // Consumer читає тільки committed messages // Producer пише атомарно: commit + forward offset
Kafka Transactions (варіант 2) мають overhead і складніша конфігурація. Для більшості випадків idempotent consumer (варіант 1) + isolation-level=read_committed достатньо.

error handling

DLQ і error handling

// Dead Letter Topic: повідомлення що не вдалось обробити @Bean public DefaultErrorHandler errorHandler(KafkaTemplate<Object, Object> kafkaTemplate) { DeadLetterPublishingRecoverer recoverer = new DeadLetterPublishingRecoverer(kafkaTemplate, (record, ex) -> new TopicPartition( record.topic() + ".DLT", // route-events.DLT record.partition())); FixedBackOff backOff = new FixedBackOff(1000L, 3); // retry: 3 рази з паузою 1 секунда return new DefaultErrorHandler(recoverer, backOff); } // Після 3 невдалих спроб → повідомлення в DLT // Окремий consumer обробляє DLT і алертить/логує @KafkaListener(topics = "route-events.DLT") void handleDeadLetter(ConsumerRecord<?, ?> record) { log.error("Dead letter: topic={}, offset={}, value={}", record.topic(), record.offset(), record.value()); alerting.send("Kafka DLT message"); }
На співбесіді: partition = паралелізм (max consumers = partitions), consumer group offset per partition, acks=all+idempotence для надійності, at-least-once = commit після обробки + idempotent consumer для exactly-once, rebalancing = STW для group.
