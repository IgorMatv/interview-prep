Web-scoped beans не можна вприснути напряму в singleton. Spring автоматично генерує scoped proxy — делегат що кожен раз звертається до правильного екземпляру з поточного контексту.
пастка

Prototype в Singleton — пастка

Якщо вприснути prototype bean у singleton — Spring інжектує один раз при створенні singleton. Prototype фактично стає singleton.

@Component @Scope("prototype") class Worker { /* stateful */ } @Service // singleton class MyService { @Autowired private Worker worker; // ОДИН екземпляр навжди! // Prototype не має сенсу тут }

Три правильних рішення

// 1. ApplicationContext.getBean() (не елегантно) @Autowired ApplicationContext ctx; void doWork() { Worker w = ctx.getBean(Worker.class); // новий кожен раз } // 2. ObjectProvider (Spring-way) @Autowired ObjectProvider<Worker> workerProvider; void doWork() { Worker w = workerProvider.getObject(); // новий кожен раз } // 3. @Lookup (Spring генерує override метода) @Lookup protected Worker createWorker() { return null; } // Spring замінює
ObjectProvider — найчистіший варіант. Lazy, thread-safe, не прив'язує до ApplicationContext напряму.
профілі

@Profile — середовища

Активує beans тільки для конкретного профілю (dev, test, prod). Дозволяє мати різні реалізації для різних середовищ.

// Реалізація для dev @Component @Profile("dev") class StubEmailService implements EmailService { public void send(String to, String body) { log.info("[STUB] Email to {}: {}", to, body); } } // Реалізація для prod @Component @Profile("!dev") // будь-який профіль крім dev class SmtpEmailService implements EmailService { ... } // Активація профілю: // application.properties: spring.profiles.active=dev // або JVM arg: -Dspring.profiles.active=prod // або в тестах: @ActiveProfiles("test")
Можна комбінувати: @Profile({"dev", "test"}), @Profile("!prod"), @Profile("dev & cloud") (Spring Expression).
conditional

@Conditional

Більш гнучкий варіант @Profile. Bean реєструється тільки якщо умова виконується. @Profile сам реалізований через @Conditional.

// Свій Condition class KafkaEnabledCondition implements Condition { @Override public boolean matches(ConditionContext ctx, AnnotatedTypeMetadata meta) { String enabled = ctx.getEnvironment().getProperty("kafka.enabled"); return "true".equalsIgnoreCase(enabled); } } // Використання @Component @Conditional(KafkaEnabledCondition.class) class KafkaEventPublisher implements EventPublisher { ... } // Або @ConditionalOnProperty (Spring Boot) @ConditionalOnProperty(name = "kafka.enabled", havingValue = "true")
Spring Boot

@ConditionalOn* — Spring Boot

Spring Boot надає готові умовні анотації — основа autoconfiguration.

// По property @ConditionalOnProperty(name = "feature.cache", havingValue = "true", matchIfMissing = false) // Клас присутній на classpath @ConditionalOnClass(RedisTemplate.class) // Клас відсутній @ConditionalOnMissingClass("com.hazelcast.core.HazelcastInstance") // Bean зареєстрований @ConditionalOnBean(DataSource.class) // Bean відсутній — дозволяє override @ConditionalOnMissingBean(CacheManager.class) // Профіль активний @ConditionalOnExpression("${app.mode} == 'cloud'") // Веб-середовище @ConditionalOnWebApplication
@ConditionalOnMissingBean — ключ до extensibility. Autoconfiguration реєструє beans тільки якщо ти сам не визначив свою реалізацію. Це дозволяє override будь-якого autoconfigured bean.
вибір bean

@Primary і @Qualifier

Коли є кілька beans одного типу — Spring не знає який вибрати. @Primary і @Qualifier вирішують неоднозначність.

// Дві реалізації @Component @Primary // обирається за замовчуванням class RedisCache implements Cache { ... } @Component @Qualifier("local") // іменована реалізація class InMemoryCache implements Cache { ... } // Вприскування @Autowired Cache cache; // RedisCache (Primary) @Autowired @Qualifier("local") Cache localCache; // InMemoryCache // Всі реалізації одразу @Autowired List<Cache> allCaches; // [Redis, InMemory]
На співбесіді: singleton vs prototype (і пастка з prototype в singleton), @Profile vs @Conditional, @ConditionalOnMissingBean як механізм override, @Primary vs @Qualifier.
