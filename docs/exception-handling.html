Error — чому не перехоплюємо

Error сигналізує про критичну проблему JVM або середовища — стан при якому продовження роботи неможливе або безглузде. Перехоплення OutOfMemoryError зазвичай лише відтерміновує крах і ускладнює діагностику.

Є рідкісні легітимні випадки перехоплення Error — наприклад, в тест-фреймворках або контейнерах де треба залогувати і gracefully shutdown. Але в бізнес-логіці — ніколи.
класифікація

Checked vs Unchecked

Checked (Exception)

Компілятор вимагає обробки — catch або throws. Очікувані, відновлювані ситуації.

IOException SQLException ParseException InterruptedException CloneNotSupportedException

Unchecked (RuntimeException)

Компілятор не вимагає обробки. Зазвичай — помилки програмування.

NullPointerException IllegalArgumentException IllegalStateException IndexOutOfBoundsException ConcurrentModificationException

Коли що використовувати — правило Блоха

  • Checked — якщо caller може розумно відновитись. Приклад: файл не знайдено — можна запропонувати альтернативу.
  • Unchecked — якщо помилка через неправильне використання API. Приклад: null переданий у метод що не приймає null.
// Checked: caller може обробити — файл недоступний public Config loadConfig(Path p) throws IOException { return parseFile(p); // caller вирішує що робити } // Unchecked: помилка програміста — null неприпустимий public void process(Order order) { Objects.requireNonNull(order, "order must not be null"); }
Тренд в сучасному Java: Spring і більшість фреймворків обгортають checked у unchecked (наприклад, DataAccessException). Checked exceptions у великих API захаращують сигнатури і заважають лямбдам.
механіка

try-with-resources (TWR)

Java 7+. Автоматично закриває ресурси що реалізують AutoCloseable. close() викликається навіть при виключенні — в блоці finally вручну.

до Java 7 — ненадійно

InputStream in = null; try { in = new FileInputStream(f); // ... } finally { if (in != null) in.close(); // може кинути! // вихідний exception загублений }

try-with-resources

try (InputStream in = new FileInputStream(f)) { // ... } // in.close() — автоматично // suppressed exceptions збережені

Кілька ресурсів — закриваються у зворотному порядку

try ( Connection conn = dataSource.getConnection(); PreparedStatement ps = conn.prepareStatement(sql); ResultSet rs = ps.executeQuery() ) { while (rs.next()) { ... } } // Закриття: rs → ps → conn (зворотний порядок)

Suppressed exceptions

// Якщо в try і в close() виникли exceptions: // primary exception — від try блоку // secondary — від close() → додається як suppressed try { Exception[] suppressed = e.getSuppressed(); // обидва доступні }
TWR вирішує проблему "exception swallowing" — у старому finally коді exception з close() перекривав оригінальний. З TWR обидва зберігаються через getSuppressed().

Власний AutoCloseable

class ManagedConnection implements AutoCloseable { private final Connection conn; @Override public void close() throws Exception { conn.close(); System.out.println("connection closed"); } } try (var mc = new ManagedConnection()) { mc.doSomething(); } // close() викличеться автоматично
синтаксис

multi-catch і finally

multi-catch (Java 7+)

// До Java 7: дублювання try { ... } catch (IOException e) { log(e); throw new ServiceException(e); } catch (SQLException e) { log(e); throw new ServiceException(e); } // Java 7+: multi-catch try { ... } catch (IOException | SQLException e) { log(e); throw new ServiceException(e); } // e — effectively final, тип = LUB типів (Comparable&Serializable...)

finally — гарантоване виконання

// finally виконується завжди крім System.exit() і JVM crash try { return 1; } finally { System.out.println("runs!"); // виконається перед return } // ПАСТКА: return у finally перекриває exception try { throw new RuntimeException(); } finally { return 42; // exception загублений! }

Порядок catch — від вужчого до ширшого

// COMPILE ERROR: FileNotFoundException вже покрита IOException catch (IOException e) { ... } catch (FileNotFoundException e) { ... } // недосяжний код! // ПРАВИЛЬНО: від вужчого catch (FileNotFoundException e) { ... } catch (IOException e) { ... }
патерни

Exception chaining та wrapping

При перехопленні низькорівневого exception і киданні вищого — завжди передавай оригінал як cause. Інакше стек трейс нижнього рівня буде втрачено.

// ПОГАНО: загубили оригінал catch (SQLException e) { throw new ServiceException("DB error"); // SQLException і де вона виникла — загублено! } // ДОБРЕ: chaining catch (SQLException e) { throw new ServiceException("DB error", e); // e.getCause() → оригінальний SQLException } // Розгортаємо ланцюжок while (e.getCause() != null) e = e.getCause();

Checked → Unchecked wrapping (sneaky throw)

// Проблема: checked exception в лямбді list.stream() .map(s -> parse(s)) // COMPILE ERROR якщо parse кидає IOException // Рішення 1: обгортка в unchecked list.stream() .map(s -> { try { return parse(s); } catch (IOException e) { throw new UncheckedIOException(e); // спеціальний wrapper } }) // UncheckedIOException — стандартний wrapper для IOException // (Java 8, java.io.UncheckedIOException)
UncheckedIOException — офіційний wrapper в JDK. Для інших checked exceptions роби власний unchecked wrapper або використовуй RuntimeException(cause).
практика

Власні винятки

Свої exception класи дозволяють розрізняти ситуації і надавати контекстну інформацію. Мінімальна реалізація — 4 конструктори.

// Unchecked — рекомендований підхід public class OrderNotFoundException extends RuntimeException { private final long orderId; public OrderNotFoundException(long orderId) { super("Order not found: " + orderId); this.orderId = orderId; } public OrderNotFoundException(long orderId, Throwable cause) { super("Order not found: " + orderId, cause); this.orderId = orderId; } public long getOrderId() { return orderId; } }

Ієрархія своїх винятків

// Базовий клас для домену public abstract class DomainException extends RuntimeException { protected DomainException(String msg) { super(msg); } protected DomainException(String msg, Throwable cause) { super(msg, cause); } } // Конкретні підкласи public class EntityNotFoundException extends DomainException { ... } public class ValidationException extends DomainException { ... } public class ConflictException extends DomainException { ... } // Spring: один handler для всіх @ExceptionHandler(DomainException.class) public ResponseEntity<?> handle(DomainException e) { ... }
антипатерни

Типові помилки

1. Порожній catch — "swallowing"

// ПОГАНО: exception зникає безслідно try { riskyOperation(); } catch (Exception e) { // нічого } // ДОБРЕ: хоча б логуй catch (Exception e) { log.error("Failed to process", e); throw new ServiceException("Processing failed", e); }

2. Catch Exception або Throwable

// ПОГАНО: перехоплює OutOfMemoryError теж! catch (Throwable t) { ... } // ПОГАНО: перехоплює NullPointerException, CME тощо catch (Exception e) { ... } // ДОБРЕ: конкретний тип catch (IOException e) { ... }

3. throw e без cause при wrapping

// ПОГАНО: стек трейс від SQLException загублено catch (SQLException e) { throw new ServiceException(e.getMessage()); // без cause! } // ДОБРЕ catch (SQLException e) { throw new ServiceException("DB error", e); // з cause }

4. return у finally

// ПОГАНО: exception буде проковтнутий! try { throw new RuntimeException("important!"); } finally { return; // exception зник назавжди }
