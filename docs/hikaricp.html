навіщо

Навіщо connection pool

Встановлення DB з'єднання — дорога операція: TCP handshake, аутентифікація, SSL negotiation (~50-100ms). Connection pool тримає готові з'єднання і повторно використовує їх.

// Без пулу: нове з'єднання на кожен запит // 100 RPS → 100 нових з'єднань/сек → overhead // З пулом: беремо готове з'єднання (~0.5ms) // після запиту повертаємо в пул // HikariCP — default в Spring Boot 2+ // Найшвидший пул: ~28 мікросекунд на отримання з'єднання // Проти DBCP2/c3p0/Tomcat pool
HikariCP — default в Spring Boot 2+, не потрібна додаткова залежність. Налаштовується через spring.datasource.hikari.*.
конфігурація

Ключові параметри

spring.datasource.hikari.maximum-pool-size=10 # максимальна кількість з'єднань (default 10) spring.datasource.hikari.minimum-idle=5 # мінімальна кількість idle з'єднань # рекомендація HikariCP: = maximum-pool-size (fixed pool) spring.datasource.hikari.connection-timeout=30000 # скільки ms чекати на з'єднання з пулу (default 30s) # при перевищенні: SQLTransientConnectionException spring.datasource.hikari.idle-timeout=600000 # скільки ms idle з'єднання живе перед закриттям (10 min) spring.datasource.hikari.max-lifetime=1800000 # максимальний час життя з'єднання (30 min) # має бути < DB server timeout (PostgreSQL: idle_in_transaction_session_timeout) spring.datasource.hikari.keepalive-time=120000 # ping idle з'єднання щоб не закрив firewall (2 min) spring.datasource.hikari.pool-name=HikariPool-main
розмір пулу

Розмір пулу — формула HikariCP

HikariCP рекомендує: pool size = Tn × (Cm - 1) + 1, де Tn — максимальна кількість потоків, Cm — максимальна кількість одночасних з'єднань на потік.

// Проста формула від HikariCP wiki: // pool_size = cpu_cores * 2 + disk_spindles // Для SSD без spindles: cpu_cores * 2 + 1 // Приклад: 4 CPU, SSD // pool_size = 4 * 2 + 1 = 9 ≈ 10 // Але "правило 10" від HikariCP: // Більшість застосунків не потребують > 10 з'єднань // 10 з'єднань × 10 мс на запит = 1000 QPS
Більше з'єднань ≠ швидше. PostgreSQL має overhead на кожне з'єднання (~5MB RAM). 100 з'єднань на 4-ядерному сервері → context switching overhead перевищує користь. Оптимум зазвичай 10-20.

Pool size і thread pool

// ВАЖЛИВО: pool_size ≥ thread_pool_size // Якщо 200 threads і 10 з'єднань → 190 threads чекають // → connection-timeout exceptions під навантаженням // Але і навпаки: thread_pool_size > pool_size // → природній backpressure (threads чекають на з'єднання) // Spring Boot: tomcat.threads.max (default 200) // HikariCP: maximum-pool-size (default 10) // → потенційна проблема під навантаженням // Рішення: або збільшити pool-size // або зменшити tomcat threads server.tomcat.threads.max=50 spring.datasource.hikari.maximum-pool-size=50

моніторинг

Моніторинг і метрики

// Spring Actuator + Micrometer: автоматично management.endpoints.web.exposure.include=metrics,health management.metrics.tags.application=my-app // GET /actuator/metrics/hikaricp.connections // Ключові метрики: // hikaricp.connections.active — зараз використовуються // hikaricp.connections.idle — вільні // hikaricp.connections.pending — очікують (!) → pool замалий // hikaricp.connections.timeout — не дочекались → critical // hikaricp.connections.acquire — час отримання з'єднання // Grafana дашборд: HikariCP dashboard id=6083 // Логування статистики (dev) logging.level.com.zaxxer.hikari.HikariConfig=DEBUG logging.level.com.zaxxer.hikari=TRACE
pending > 0 постійно — пул замалий або є connection leak. timeout > 0 — критично, запити відкидаються. Обидва потребують негайної уваги.
connection leak

Connection leak

З'єднання взяте з пулу але не повернуте. Пул поступово вичерпується → нові запити чекають → timeout.

// Причини: // 1. Connection не закрито при exception // 2. Транзакція відкрита без commit/rollback // 3. Код в @Async без @Transactional правильно закриває // Виявлення: leak detection spring.datasource.hikari.leak-detection-threshold=5000 // Якщо з'єднання не повернуте за 5 секунд → WARNING зі стек-трейсом // Приклад виводу: // WARNING: Connection leak detection triggered for connection // com.zaxxer.hikari.pool.ProxyConnection@7a2c... // Stack trace of connection borrower: // at com.example.service.OrderService.processOrder(...) ← тут проблема
leak-detection-threshold — перший крок при підозрі на connection leak. Встанови 2-3x більше типового часу запиту. Це не вирішує, але вказує де проблема.
антипатерни

Антипатерни і правильне налаштування

1. Занадто великий пул

// ПОГАНО: 200 з'єднань на 4-ядерний PG сервер maximum-pool-size=200 // → PG витрачає ~1GB RAM на з'єднання // → context switching overhead // → db-side connection limit exceeded

2. connection-timeout занадто великий

// ПОГАНО: 30 секунд чекати на з'єднання connection-timeout=30000 // → user чекає 30 секунд перед помилкою // ДОБРЕ: fail fast connection-timeout=3000 // 3 секунди → 503 для user

3. max-lifetime > DB server timeout

// PostgreSQL idle connection timeout (pg.conf) // idle_in_transaction_session_timeout = 10min // max-lifetime має бути < 10min max-lifetime=540000 // 9 хвилин — менше ніж DB timeout
На співбесіді: pool тримає готові з'єднання щоб уникнути overhead встановлення, HikariCP рекомендує fixed pool (minimum-idle = maximum-pool-size), правило ~10 з'єднань на CPU core, connection-timeout = fail fast, leak-detection для діагностики.
