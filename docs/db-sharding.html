Vertical vs Horizontal scaling

  • Vertical (scale up) — більший сервер: більше CPU, RAM, швидший SSD. Простіше, але дорого і має ліміт.
  • Horizontal (scale out) — більше серверів. Потрібна складніша архітектура, але безмежно масштабується.
// Типові ліміти single PostgreSQL node: // ~50K-100K simple queries/sec // ~10TB data (manageable) // ~64 cores, 1TB RAM max (практично) // // Коли вертикальне масштабування не допомагає: // - дані не вміщуються на диску одного сервера // - write QPS перевищує можливості одного сервера // - geo-distribution (users в різних регіонах)
replication

Replication — спочатку це

Перш ніж шардувати — реплікація. Primary (write) + Replicas (read). Вирішує більшість read-heavy проблем.

// PostgreSQL replication через Spring // Primary: accepts writes // Read replicas: handle read traffic (eventual consistency) // Spring: routing DataSource // @Transactional(readOnly=true) → replica // @Transactional → primary // AbstractRoutingDataSource для routing class ReadWriteDataSource extends AbstractRoutingDataSource { protected Object determineCurrentLookupKey() { return TransactionSynchronizationManager.isCurrentTransactionReadOnly() ? "replica" : "primary"; } } // Ліміти реплікації: // - writes все ще на одному primary // - replication lag → stale reads // - не вирішує data size проблему
партиціонування

Партиціонування vs Sharding

  • Partitioning — поділ таблиці на частини всередині одного DB сервера. PostgreSQL Table Partitioning. Покращує query performance, maintenance.
  • Sharding — горизонтальне partitioning між кількома DB серверами. Кожен shard = окремий сервер з підмножиною даних.
-- PostgreSQL Table Partitioning (не sharding) CREATE TABLE orders ( id BIGINT, user_id BIGINT, created_at TIMESTAMPTZ ) PARTITION BY RANGE (created_at); CREATE TABLE orders_2024 PARTITION OF orders FOR VALUES FROM ('2024-01-01') TO ('2025-01-01'); CREATE TABLE orders_2025 PARTITION OF orders FOR VALUES FROM ('2025-01-01') TO ('2026-01-01'); -- Query автоматично йде в правильну партицію (partition pruning) SELECT * FROM orders WHERE created_at BETWEEN '2024-06-01' AND '2024-07-01'; -- Сканує тільки orders_2024, не orders_2025
hash sharding

Hash sharding

// shard_id = hash(shard_key) % num_shards // shard_key: user_id, order_id, tenant_id // Приклад: 4 shards user_id=100 → hash(100) % 4 = 0 → Shard 0 user_id=101 → hash(101) % 4 = 1 → Shard 1 user_id=200 → hash(200) % 4 = 0 → Shard 0 // Переваги: // - Рівномірний розподіл // - Немає hot spots (якщо key рівномірний) // Проблема: JOIN між shards // SELECT o.*, u.name FROM orders o JOIN users u ON o.user_id = u.id // orders і users можуть бути на різних shards! // Рішення: denormalize, або шардувати по одному ключу (user_id)

Range і Directory sharding

Range sharding

// user_id 1-1M → Shard 0 // user_id 1M-2M → Shard 1 // ... // Переваги: range queries ефективні (один shard) // Проблеми: hot spots (нові users завжди в останній shard)

Directory (Lookup) sharding

// Lookup table: user_id → shard_id // Зберігається в Redis або окремій DB // user_id=100 → Redis lookup → shard_id=2 → query Shard 2 // Переваги: гнучко, можна rebalance без resharding // Проблеми: додатковий lookup, SPOF якщо Redis впаде
Cassandra використовує Consistent Hashing — всі nodes на кільці, key → hash → найближча node. Resharding мінімально переміщує дані.
проблеми

Resharding і hot spots

Resharding — додати новий shard

// hash(key) % 4 → hash(key) % 5 при додаванні shard // Майже всі ключі переміщуються! (4/5 = 80%) // Consistent Hashing вирішує: тільки K/N ключів переміщуються // де K=кількість ключів, N=кількість нових nodes

Hot spots — celebrity problem

// Shard key = user_id // celebrity user_id=1 → всі reads/writes на Shard 0 // Shard 0 overloaded, інші idle // Рішення: // 1. Додати random суфікс: "1_001", "1_002"... для hot key // 2. Caching на application рівні (Redis) // 3. Окремий shard/tier для celebrity users
вибір

Коли sharding, альтернативи

  • Спочатку: вертикальне масштабування → replication → partitioning
  • Sharding тільки коли: дані не вміщуються або write QPS перевищує можливості primary
  • Типовий поріг: ~10TB або write throughput > 10K/sec для PostgreSQL
// Альтернативи sharding: // Vitess: sharding middleware для MySQL // Citus: distributed PostgreSQL // CockroachDB: auto-sharding NewSQL // Cassandra: native sharding через consistent hashing // DynamoDB: managed sharding // Проблеми sharding: // - Cross-shard joins: дорогі або неможливі // - Cross-shard transactions: 2PC або saga // - Rebalancing: складне і ризиковане // - Operational complexity: +N серверів
На співбесіді: спочатку vertical → replication → partitioning → sharding (в такому порядку), hash sharding = рівномірний розподіл, consistent hashing мінімізує resharding overhead, hot spots = celebrity problem → random suffix або caching.
