persistence context

Persistence Context (PC)

Persistence Context — кеш першого рівня. Зберігає всі managed entities в рамках транзакції. Один EntityManager = один PC. В Spring — PC живе в рамках транзакції.

// PC як identity map: key = (EntityClass, id) Order o1 = repo.findById(1L).get(); Order o2 = repo.findById(1L).get(); o1 == o2; // true! Другий запит → з PC кешу, не з DB // Але між транзакціями — різні PC @Transactional void tx1() { Order o = repo.findById(1L).get(); } @Transactional void tx2() { Order o = repo.findById(1L).get(); // нова PC → SQL запит }
OSIV (Open Session in View) — Spring Boot за замовчуванням відкриває EntityManager на весь HTTP запит (не тільки транзакцію). Це дозволяє lazy loading у View шарі, але тримає DB з'єднання довше. Вимикають: spring.jpa.open-in-view=false.

EntityManager vs Spring Data Repository

// Spring Data Repository — обгортка над EntityManager // findById() → em.find() // save() нового → em.persist() // save() існуючого (з id) → em.merge() // delete() → em.remove() // Пряме використання EM @PersistenceContext private EntityManager em; Order order = em.find(Order.class, id); em.persist(order); em.flush();
dirty checking

Dirty Checking

Hibernate порівнює стан managed entity з snapshot при завантаженні. При flush — генерує UPDATE для змінених полів. Автоматично, без явного save().

@Transactional void updateOrder(long id, BigDecimal newTotal) { Order order = repo.findById(id).orElseThrow(); order.setTotal(newTotal); // просто сетер! // repo.save(order) НЕ ПОТРІБНО // При commit: Hibernate знайде різницю → UPDATE SQL } // Як Hibernate відслідковує: // 1. При завантаженні — зберігає snapshot стан // 2. При flush — порівнює current state з snapshot // 3. Якщо різниця — генерує UPDATE
Overhead dirty checking: при великих колекціях managed entities в PC — Hibernate порівнює всі при flush. При batch обробці (тисячі entities) — використовуй em.clear() або stateless session.

@DynamicUpdate — UPDATE тільки змінених колонок

// За замовчуванням: UPDATE з усіма полями // UPDATE orders SET total=?, status=?, created_at=? WHERE id=? // З @DynamicUpdate: тільки змінені @Entity @DynamicUpdate class Order { ... } // UPDATE orders SET total=? WHERE id=? // Коли корисно: entity з багатьма полями, часті часткові оновлення // Мінус: не можна кешувати prepared statement
операції

persist, merge, remove

// persist: Transient → Managed // INSERT виконається при flush // Якщо entity вже Managed — ігнорується // Якщо Detached — IllegalArgumentException em.persist(newOrder); // merge: Detached → Managed (копія!) // Повертає НОВИЙ managed екземпляр // Оригінальний detached залишається detached Order managed = em.merge(detachedOrder); // не той самий об'єкт! managed.setTotal(newTotal); // tracked detachedOrder.setTotal(newTotal); // НЕ tracked // remove: Managed → Removed // DELETE виконається при flush // Якщо Detached → IllegalArgumentException em.remove(managedOrder);
Пастка merge: em.merge() повертає новий об'єкт. Якщо продовжувати працювати з оригінальним detached — зміни не збережуться. Завжди використовуй повернутий екземпляр.

Spring Data: save() — persist чи merge?

// SimpleJpaRepository.save(): public <S extends T> S save(S entity) { if (entityInformation.isNew(entity)) { em.persist(entity); // id == null → persist return entity; } else { return em.merge(entity); // id != null → merge } } // isNew() перевіряє: id == null (або @Version == null) // Власна логіка: implements Persistable<ID>
flush

flush і FlushMode

Flush — синхронізація стану PC з базою (генерація SQL). Не те саме що commit. Flush може відбутись кілька разів в рамках однієї транзакції.

// FlushModeType FlushModeType.AUTO // flush перед query і перед commit (default) FlushModeType.COMMIT // flush тільки перед commit // AUTO: Hibernate робить flush перед кожним JPQL/HQL запитом // якщо є pending changes для таблиці що запитується @Transactional void example() { order.setStatus("PAID"); // dirty repo.findByStatus("PAID"); // AUTO: flush перед цим запитом! // інакше findByStatus не побачить наше оновлення } // Примусовий flush em.flush(); // COMMIT FlushMode — менше SQL, але може не бачити своїх змін em.setFlushMode(FlushModeType.COMMIT);
FlushMode.AUTO — Hibernate сам вирішує коли flush. Іноді це призводить до несподіваних SQL запитів. show_sql=true допомагає зрозуміти що відбувається.
управління PC

detach і clear

// detach: Managed → Detached (один entity) em.detach(order); order.setTotal(newTotal); // не відслідковується // clear: видаляє ВСІ entities з PC em.clear(); // після clear — всі раніше managed стають detached // Типовий use case: batch обробка @Transactional void processBatch(List<Long> ids) { int count = 0; for (long id : ids) { Order order = em.find(Order.class, id); processOrder(order); em.flush(); if (++count % 50 == 0) { em.clear(); // звільняємо PC, уникаємо OOM } } }
Без em.clear() при batch обробці — PC накопичує тисячі managed entities. Dirty checking стає O(n) при кожному flush. Пам'ять росте. clear() кожні N записів вирішує це.
антипатерни

Типові помилки

1. Модифікація detached entity без merge

// ПОГАНО Order order = orderService.findById(id); // detached поза @Transactional order.setStatus("CANCELLED"); // не збережеться! // немає save() → зміна загублена // ДОБРЕ: все в одній транзакції @Transactional void cancel(long id) { Order order = repo.findById(id).orElseThrow(); order.setStatus("CANCELLED"); // managed → dirty checking зробить UPDATE }

2. LazyInitializationException

// ПОГАНО: доступ до lazy collection поза транзакцією Order order = orderService.findById(id); // транзакція закрита order.getItems().size(); // LazyInitializationException! // Рішення: JOIN FETCH або @Transactional
На співбесіді: 4 стани entity (transient/managed/detached/removed), PC = L1 cache + identity map, dirty checking без явного save(), merge() повертає новий managed екземпляр, flush vs commit.
