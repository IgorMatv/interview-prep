патерн

Database per service

Кожен мікросервіс має власну БД. Інші сервіси не можуть звертатись напряму до чужої БД — тільки через API.

  • Loose coupling — можна змінити схему без впливу на інші сервіси
  • Technology choice — OrderService → PostgreSQL, SearchService → Elasticsearch, UserService → MongoDB
  • Failure isolation — падіння DB одного сервісу не блокує інші
  • Independent scaling — масштабуємо тільки той сервіс що потрібно
// OrderService: orders DB (PostgreSQL) // PaymentService: payments DB (PostgreSQL) // SearchService: elasticsearch // SessionService: Redis // NotificationService: MongoDB (flexible schema) // Join між сервісами = API call або event-driven sync // "Дай мені дані замовлення + дані користувача" → // API Composition або CQRS read model
антипатерн

Shared database — чому не варто

Кілька сервісів в одній БД — типовий антипатерн при переході до мікросервісів.

// Проблеми shared DB: // 1. Runtime coupling: OrderService читає таблицю users напряму // → UserService не може змінити схему без координації // // 2. Scaling: все масштабується разом // → не можна масштабувати тільки один сервіс // // 3. Технологічна прив'язка: // → всі сервіси мусять використовувати ту саму DB // // 4. Відсутність ізоляції: // → довгий запит в одному сервісі блокує інший
При першому переході до мікросервісів прийнятно тимчасово мати shared DB. Але це технічний борг — eventual goal: database per service.
CQRS

CQRS — Command Query Responsibility Segregation

Розділення моделей для читання і запису. Write side (Command) оптимізований для транзакцій. Read side (Query) оптимізований для запитів.

// Write side: нормалізована DB, строга consistency // POST /orders → OrderCommand → PostgreSQL (normalized) // Read side: денормалізована view, оптимізована для запитів // GET /orders?userId=1 → OrderQuery → ElasticSearch / Redis // або denormalized PostgreSQL table // Синхронізація write → read через events // OrderCreated → update read model // OrderUpdated → update read model // Без повного CQRS: просто різні моделі в одному сервісі // CommandModel: Order entity для транзакцій // QueryModel: OrderSummaryDto для читання (SELECT з JOIN)
CQRS не обов'язково означає 2 DB. Простий варіант: різні моделі в одному сервісі (write entity, read DTO). Повний CQRS з окремими DB — тільки коли є чітка потреба в різному масштабуванні.
Event Sourcing

Event Sourcing

Замість зберігання поточного стану — зберігаємо послідовність подій. Стан = результат replay всіх подій. Незмінна, append-only история.

// Традиційний підхід: зберігаємо стан orders table: {id, status="PAID", total=100} // Як дійшли до PAID? Невідомо. // Event Sourcing: зберігаємо події order_events: {orderId=1, type="OrderCreated", data={total:100}, ts=10:00} {orderId=1, type="PaymentMade", data={amount:100}, ts=10:01} {orderId=1, type="OrderShipped", data={courier:DHL}, ts=10:05} // Поточний стан = apply всіх подій Order order = events.stream() .reduce(new Order(), Order::apply);
Event Sourcing

Переваги і ціна Event Sourcing

  • Повна аудит-трасса — кожна зміна, хто і коли
  • Time travel — стан системи в будь-який момент
  • Debug — replay подій → відтворити будь-який стан
  • Projections — нові read моделі з існуючих подій
  • Природній Kafka/messaging — події готові до публікації
  • Складність — значно складніший ніж традиційний CRUD
  • Eventual read — read model може бути застарілою
  • Schema evolution — зміна формату старих подій складна
  • Snapshots потрібні — при тисячах подій replay стає повільним
  • GDPR проблема — "право на забуття" складне при immutable log
Event Sourcing — не срібна куля. Підходить для: фінансові транзакції, аудит, складна бізнес-логіка з багатьма станами. Не підходить для: простий CRUD, коли аудит не потрібен.
проекції

Read model і проекції

Проекція — read model побудована з events. Можна мати кілька проекцій з одного event store для різних потреб.

// Event store → кілька проекцій // OrderEvents → // OrderListView (PostgreSQL): id, status, total — для списку // OrderDetailView (MongoDB): повні деталі — для сторінки // OrderAnalytics (Elasticsearch): для пошуку і звітів // Проекція в Java (спрощено) @EventHandler void on(OrderCreatedEvent event) { orderViewRepo.save(new OrderView(event.getId(), "CREATED")); } @EventHandler void on(PaymentMadeEvent event) { orderViewRepo.findById(event.getOrderId()) .ifPresent(v -> { v.setStatus("PAID"); orderViewRepo.save(v); }); } // Rebuild projection: replay всіх подій з початку // Нова проекція = replay → побудувати з нуля
вибір

Коли що застосовувати

// CQRS без Event Sourcing — найчастіший варіант // Коли: read/write навантаження сильно різне // Коли: read потребує складних JOIN або іншої DB // Коли: потрібен окремий search (Elasticsearch) // Event Sourcing — складна бізнес-логіка // Коли: потрібна повна аудит-трасса (banking, fintech) // Коли: багато станів і переходів між ними // Коли: потрібна можливість replay і time travel // Просто CRUD — більшість сервісів // Коли: CRUD достатній, немає складної бізнес-логіки // Коли: аудит можна додати через @CreatedDate/@ModifiedDate
На співбесіді: Database per service = loose coupling + technology freedom, CQRS = різні моделі читання і запису (не обов'язково 2 DB), Event Sourcing = append-only log подій як source of truth, projections = read models з подій. ES ускладнює — тільки при чіткій потребі.
