механізм

@SpringBootApplication

Мета-анотація що об'єднує три ключові анотації.

@SpringBootApplication // = @SpringBootConfiguration // + @EnableAutoConfiguration // + @ComponentScan // @SpringBootConfiguration = @Configuration // Позначає клас як джерело @Bean definitions // @ComponentScan // Сканує поточний пакет і підпакети для @Component // @EnableAutoConfiguration — ключова магія // Завантажує всі autoconfig класи з classpath // Виключити конкретну autoconfig @SpringBootApplication(exclude = {DataSourceAutoConfiguration.class})
@ComponentScan без аргументів сканує пакет класу з @SpringBootApplication. Саме тому головний клас треба класти в кореневий пакет — щоб всі підпакети були відскановані.
механізм

Як завантажуються autoconfiguration

// Spring Boot 2.x: META-INF/spring.factories org.springframework.boot.autoconfigure.EnableAutoConfiguration=\ com.example.MyAutoConfiguration,\ com.example.AnotherAutoConfiguration // Spring Boot 3.x: META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports com.example.MyAutoConfiguration com.example.AnotherAutoConfiguration // @AutoConfiguration в Spring Boot 3.x @AutoConfiguration // = @Configuration + ordering support @ConditionalOnClass(RedisTemplate.class) class RedisAutoConfiguration { @Bean @ConditionalOnMissingBean public RedisTemplate<?, ?> redisTemplate(RedisConnectionFactory factory) { return new RedisTemplate<>(); } }
Spring Boot завантажує сотні autoconfig класів. Але більшість не реєструють beans завдяки @ConditionalOn* — якщо немає Redis на classpath, RedisAutoConfiguration нічого не створить.
умови

@ConditionalOn* у autoconfig — приклад

Подивимось як реально влаштована DataSourceAutoConfiguration.

@AutoConfiguration @ConditionalOnClass({ DataSource.class, EmbeddedDatabaseType.class }) // завантажується тільки якщо DataSource на classpath @EnableConfigurationProperties(DataSourceProperties.class) // зв'язує spring.datasource.* з DataSourceProperties class DataSourceAutoConfiguration { @Bean @ConditionalOnMissingBean // тільки якщо не визначено своєї public DataSource dataSource(DataSourceProperties props) { return props.initializeDataSourceBuilder().build(); } } // Ланцюжок: spring-boot-starter-data-jpa тягне: // → hibernate-core → HibernateJpaAutoConfiguration // → HikariCP → DataSourceAutoConfiguration // → spring-data-jpa → JpaRepositoriesAutoConfiguration
практика

Override autoconfig

Будь-яку autoconfig можна перевизначити — саме для цього @ConditionalOnMissingBean.

// Override DataSource — своя конфігурація пулу @Configuration class DatabaseConfig { @Bean // твій Bean → autoconfig не створить свій (MissingBean) public DataSource dataSource() { return DataSourceBuilder.create() .type(HikariDataSource.class) .url("jdbc:postgresql://...") .maximumPoolSize(20) .build(); } } // Override ObjectMapper (Jackson) @Bean public ObjectMapper objectMapper() { return new ObjectMapper() .registerModule(new JavaTimeModule()) .disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS); } // Виключити autoconfig повністю @SpringBootApplication(exclude = SecurityAutoConfiguration.class)

// @ConfigurationProperties — типізована конфігурація @ConfigurationProperties(prefix = "app.route") @Component class RouteProperties { private int maxStops = 20; private Duration cacheTtl = Duration.ofHours(1); // getters + setters (або @ConstructorBinding) } // application.properties: app.route.max-stops=15 app.route.cache-ttl=2h // Пріоритет конфігурації (від вищого до нижчого): // 1. JVM -D props: -Dspring.datasource.url=... // 2. Environment variables: SPRING_DATASOURCE_URL // 3. application-{profile}.properties // 4. application.properties // 5. @PropertySource // 6. просунуто

Власна autoconfig (для бібліотек)

// 1. Autoconfig клас @AutoConfiguration @ConditionalOnClass(RouteOptimizer.class) @EnableConfigurationProperties(RouteOptimizerProperties.class) class RouteOptimizerAutoConfiguration { @Bean @ConditionalOnMissingBean public RouteOptimizer routeOptimizer(RouteOptimizerProperties props) { return new RouteOptimizer(props.getMaxStops()); } } // 2. Зареєструвати: // resources/META-INF/spring/ // org.springframework.boot.autoconfigure.AutoConfiguration.imports com.example.RouteOptimizerAutoConfiguration
Всі Spring Boot starters влаштовані саме так. Залежність + autoconfig + @ConditionalOnMissingBean = "convention over configuration" без жертви гнучкістю.
діагностика

Діагностика: що і чому завантажилось

// Actuator: список всіх beans // GET /actuator/beans // Звіт autoconfiguration: що застосовано і чому // GET /actuator/conditions (потрібен actuator) // Або при старті: --debug // або logging.level.org.springframework.boot.autoconfigure=DEBUG // Виведе CONDITIONS EVALUATION REPORT: // Positive matches (застосовані): // RedisAutoConfiguration: // - @ConditionalOnClass found required class 'RedisTemplate' // Negative matches (не застосовані): // MongoAutoConfiguration: // - @ConditionalOnClass did not find required class 'MongoClient'
На співбесіді: @SpringBootApplication = @Configuration + @ComponentScan + @EnableAutoConfiguration, autoconfig завантажується з imports файлу через SPI, @ConditionalOnMissingBean дозволяє override, пріоритет конфігурації (env vars > application.properties).
