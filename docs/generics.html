Variance — чому List<Dog> не є List<Animal>

Масиви в Java covariant: Dog[] можна присвоїти Animal[]. Generics — invariant: List<Dog> не є List<Animal>. Це не баг — це захист.

// Масиви: covariant — але небезпечно! Animal[] animals = new Dog[3]; animals[0] = new Cat(); // компілюється! // → ArrayStoreException в runtime // Generics: invariant — компілятор захищає List<Animal> animals = new ArrayList<Dog>(); // → COMPILE ERROR: incompatible types // Немає можливості покласти Cat в Dog-список
Чому invariant? Якби List<Dog> дозволявся як List<Animal>, то через посилання типу List<Animal> можна було б додати Cat — і це зламало б типову безпеку при читанні як Dog.

Три види variance в Java

ВидСинтаксисЧитанняЗапис

covariance

? extends T — читаємо, не пишемо

List<? extends Animal> — це список якогось конкретного підтипу Animal, але ми не знаємо якого. Тому можна читати як Animal, але не можна додавати — бо не знаємо точний тип.

List<? extends Animal> animals = new ArrayList<Dog>(); // Читання — ОК: завжди повернеться Animal або підтип Animal a = animals.get(0); // компілюється // Запис — ЗАБОРОНЕНО animals.add(new Dog()); // COMPILE ERROR animals.add(new Animal()); // COMPILE ERROR animals.add(null); // єдиний дозволений варіант // Чому? Може всередині ArrayList<Cat>! // Додавання Dog зламало б типову безпеку.

Практично: метод що читає колекцію

// Підраховує суму — не змінює колекцію double sum(List<? extends Number> list) { double total = 0; for (Number n : list) total += n.doubleValue(); return total; } // Тепер приймає List<Integer>, List<Double>, List<Long> sum(List.of(1, 2, 3)); // Integer sum(List.of(1.5, 2.5)); // Double // Без wildcard — тільки Number, не підтипи! double sumStrict(List<Number> list) { ... } sumStrict(List.of(1, 2)); // COMPILE ERROR
? extends T розширює API: метод починає приймати списки будь-яких підтипів T, а не тільки T.
contravariance

? super T — пишемо, читаємо як Object

List<? super Dog> — це список, який точно може містити Dog (або більш загальний тип). Можна безпечно додавати Dog та його підтипи. Читання повертає тільки Object.

List<? super Dog> dogs = new ArrayList<Animal>(); // Запис — ОК: Dog і його підтипи точно вміщуються dogs.add(new Dog()); // ОК dogs.add(new Labrador()); // ОК (підтип Dog) dogs.add(new Animal()); // COMPILE ERROR // Читання — тільки Object Object o = dogs.get(0); // ОК Dog d = dogs.get(0); // COMPILE ERROR // Може бути ArrayList<Object> — не знаємо

Практично: метод що заповнює колекцію

// Кладе елементи в колекцію — нічого не читає void fill(List<? super Integer> dest, int count) { for (int i = 0; i < count; i++) dest.add(i); } List<Integer> ints = new ArrayList<>(); List<Number> nums = new ArrayList<>(); List<Object> objects = new ArrayList<>(); fill(ints, 5); // ОК fill(nums, 5); // ОК fill(objects, 5); // ОК
PECS

Producer Extends, Consumer Super

Мнемоніка від Joshua Bloch (Effective Java). Питай себе: ця колекція постачає елементи в код, чи приймає елементи від коду?

Producer → використовуй extends

Колекція виробляє елементи — ти читаєш з неї. Тип елементів гарантовано підтип T.

copy(List<? extends T> src, ...) // src постачає
Consumer → використовуй super

Колекція споживає елементи — ти пишеш в неї. Гарантовано вмістить T і підтипи.

copy(..., List<? super T> dest) // dest споживає

Collections.copy() — класичний PECS

// Реальна сигнатура з JDK public static <T> void copy( List<? super T> dest, // Consumer: пишемо List<? extends T> src) // Producer: читаємо { for (int i = 0; i < src.size(); i++) dest.set(i, src.get(i)); // читаємо src, пишемо dest } // Можна копіювати Dog → Animal list! List<Dog> dogs = ...; List<Animal> animals = ...; Collections.copy(animals, dogs); // ОК
Коли нічого не підходить: якщо потрібно і читати і писати — використовуй invariant List<T> без wildcard.
практика

Практичні приклади PECS

Власний метод з PECS

// Додає елементи з src в dest — PECS повністю public static <T> void addAll( List<? super T> dest, List<? extends T> src) { for (T item : src) dest.add(item); } // Тепер гнучко: List<Integer> ints = List.of(1, 2, 3); List<Number> result = new ArrayList<>(); addAll(result, ints); // Integer → Number: ОК

Comparator і super

// sort приймає Comparator<? super T> // бо Comparator споживає елементи для порівняння void sort(List<T> list, Comparator<? super T> cmp) // Можна сортувати List<Dog> компаратором для Animal Comparator<Animal> byName = Comparator.comparing(Animal::name); List<Dog> dogs = ...; dogs.sort(byName); // ОК завдяки super

Unbounded wildcard: ?

// Коли тип взагалі не важливий void printSize(List<?> list) { System.out.println(list.size()); } // ? = ? extends Object // Читаємо тільки як Object, не пишемо // Відрізняється від List<Object>: printSize(List.of(1, 2, 3)); // ОК void bad(List<Object> l) {} bad(List.of(1, 2)); // COMPILE ERROR
type erasure

Як працює type erasure

Generics в Java — compile-time механізм. Компілятор перевіряє типи, а потім стирає їх — у байткоді залишається Object (або upper bound). JVM нічого не знає про List<String> — тільки List.

source code

List<String> list = new ArrayList<>(); list.add("hello"); String s = list.get(0);

після erasure (bytecode)

List list = new ArrayList(); list.add("hello"); String s = (String) list.get(0); // ← cast вставляє компілятор
Erasure — це ціна зворотньої сумісності. Generics з'явились у Java 5, а мільярди рядків коду з raw types мали продовжувати працювати.

Erasure з bounded type parameter

// Source: class Box<T extends Comparable<T>> { T value; int compareTo(Box<T> other) { return value.compareTo(other.value); } } // Після erasure: class Box { // T → Comparable (upper bound) Comparable value; int compareTo(Box other) { return value.compareTo(other.value); } }
наслідки

Що ламає type erasure

1. Не можна перевірити тип параметра в runtime

// COMPILE ERROR: illegal generic type for instanceof if (list instanceof List<String>) { } // ОК: unbounded wildcard if (list instanceof List<?>) { } // ОК в Java 16+ з pattern matching (але тільки raw) if (obj instanceof List<?> l) { // l — List, не List<String> }

2. Не можна створити масив generic типу

// COMPILE ERROR new List<String>[10]; new T[10]; // Workaround: cast через raw type (unchecked warning) @SuppressWarnings("unchecked") List<String>[] arr = (List<String>[]) new List[10]; // Краще: ArrayList замість масиву List<List<String>> list = new ArrayList<>();

3. Overloading по generic типу неможливий

// COMPILE ERROR: erasure однакова void process(List<String> list) {} void process(List<Integer> list) {} // Обидва → process(List list) після erasure // Рішення: різні імена методів void processStrings(List<String> list) {} void processIntegers(List<Integer> list) {}

4. Не можна використовувати T в static контексті

class Box<T> { static T defaultValue; // COMPILE ERROR static T create() { } // COMPILE ERROR // static shared між всіма Box<?>, T невідомий }
антипатерни

Типові помилки

1. Raw types

// ПОГАНО: raw type — втрачаємо типову безпеку List list = new ArrayList(); list.add("hello"); list.add(42); // компілюється без помилок! String s = (String) list.get(1); // ClassCastException runtime // ДОБРЕ: завжди вказуй тип параметр List<String> list = new ArrayList<>();

2. Неправильне застосування wildcard

// ПОГАНО: invariant там де потрібен covariant double sum(List<Number> list) { ... } sum(List.of(1, 2, 3)); // COMPILE ERROR // ДОБРЕ: PECS — читаємо, тому extends double sum(List<? extends Number> list) { ... } sum(List.of(1, 2, 3)); // ОК

3. Heap pollution через varargs

@SafeVarargs // тільки якщо метод СПРАВДІ безпечний public static <T> List<T> of(T... elements) { return Arrays.asList(elements); } // Без @SafeVarargs — unchecked warning на кожен виклик // Heap pollution: T[] може бути Object[] реально
На співбесіді: поясни PECS через Collections.copy() — src producer (extends), dest consumer (super). Потім скажи що erasure означає: instanceof по generic неможливий, масив generic неможливий, overloading по generic неможливий. Це все що треба.
