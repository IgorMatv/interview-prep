class loading

ClassLoader ієрархія

ClassLoader завантажує .class файли в JVM. Ієрархія делегує завантаження вгору — Parent Delegation Model.

// Java 11+ ієрархія: // Bootstrap ClassLoader (C++, null в Java) // └── java.lang, java.util, java.io... // // Platform ClassLoader (раніше Extension) // └── java.sql, java.xml, java.security... // // Application ClassLoader (System) // └── classpath, module-path класи // // Custom ClassLoaders // └── OSGi, Tomcat, URLClassLoader... // Отримати ClassLoader класу String.class.getClassLoader(); // null = Bootstrap MyClass.class.getClassLoader(); // Application CL Thread.currentThread().getContextClassLoader();

Parent Delegation Model

При запиті на завантаження класу — спочатку делегуємо батьківському ClassLoader. Клас завантажується батьком якщо він його знає. Це запобігає підміні системних класів.

// Алгоритм loadClass(): // 1. Перевірити кеш (вже завантажений?) // 2. Делегувати parent.loadClass() // 3. Якщо parent не знайшов → findClass() сам // 4. ClassNotFoundException якщо ніхто не знайшов // Custom ClassLoader: override findClass(), не loadClass() class MyLoader extends ClassLoader { @Override protected Class<?> findClass(String name) throws ClassNotFoundException { byte[] bytes = loadBytesFromCustomSource(name); return defineClass(name, bytes, 0, bytes.length); } }
фази

Завантаження і linking класу

// 5 фаз завантаження класу: // 1. Loading: читаємо .class файл → Class об'єкт у Metaspace // 2. Verification: перевірка bytecode // - Структура файлу коректна? // - Типи операцій відповідають операндам? // - Немає переповнення стеку? // 3. Preparation: статичні поля → default values // static int x; → x = 0 (не значення з ініціалізатора) // 4. Resolution: символьні посилання → прямі посилання // "com.example.Foo" → реальний Class об'єкт // 5. Initialization: виконання static ініціалізаторів // static { ... } і static поля з ініціалізаторами static int x = computeX(); // виконується тут
Клас ініціалізується при: першому зверненні до static поля/методу, створенні об'єкту, виклику через reflection. Ініціалізація thread-safe — JVM гарантує що static ініціалізатор виконується лише раз.
leak

ClassLoader leak

ClassLoader не може бути зібраний GC поки є посилання на будь-який з його класів або об'єктів. В hot-deploy сценаріях (Tomcat reload) — типова причина Metaspace OOM.

// Причини ClassLoader leak: // 1. ThreadLocal з об'єктом завантаженим цим CL static ThreadLocal<MyClass> tl = new ThreadLocal<>(); // Якщо потік живе довше ніж webapp — CL не збереться // 2. Static поле в загальній бібліотеці посилається на webapp клас static List<Listener> listeners = new ArrayList<>(); // listener з webapp CL → CL не збереться після undeploy // 3. Кеш з сильними посиланнями на класи // Діагностика: // jmap -clstats <pid> — статистика ClassLoader // VisualVM → Classes → ClassLoader statistics // Heap dump → аналіз ClassLoader об'єктів
Симптом ClassLoader leak: після кожного hot-deploy Metaspace зростає і не звільняється. Врешті — OutOfMemoryError: Metaspace.
JIT

Інтерпретатор і JIT компілятор

JVM спочатку інтерпретує bytecode, потім JIT компілює "гарячий" код у native machine code. Tiered Compilation — 5 рівнів.

// Tiered Compilation (default Java 8+) // Level 0: Interpreter — повільно, але починає одразу // Level 1: C1 compiled, no profiling // Level 2: C1 compiled, limited profiling // Level 3: C1 compiled, full profiling // Level 4: C2 compiled — агресивна оптимізація // Порогові значення (приблизно): // ~2000 invocations → C1 // ~10000 invocations → C2 // Флаги -XX:+TieredCompilation // default on -XX:CompileThreshold=10000 // поріг для C2 -XX:+PrintCompilation // лог компіляції -XX:+UnlockDiagnosticVMOptions -XX:+PrintInlining // inlining
Warmup: перші хвилини після старту JVM — код ще інтерпретується або компілюється C1. C2 оптимізації вступають пізніше. Benchmark без warmup — некоректний benchmark.
JIT

Ключові JIT оптимізації

Inlining

Маленькі методи вбудовуються в місце виклику. Усуває overhead виклику і відкриває більше оптимізацій.

// До inlining: int result = add(a, b); // Після inlining (концептуально): int result = a + b; // код add() вбудовано // Ліміт inlining: метод <35 bytecodes (приблизно) -XX:MaxInlineSize=35 -XX:FreqInlineSize=325

Loop unrolling

// JIT може розгорнути цикл: // for (i=0; i<4; i++) process(a[i]); // → process(a[0]); process(a[1]); process(a[2]); process(a[3]); // Менше перевірок умови, краще vectorization

Dead code elimination

// JIT видаляє код що ніколи не виконується if (false) { veryExpensiveComputation(); } // → порожньо // Небезпека для мікро-бенчмарків: JIT може видалити // весь код що не має observable side effects! // Рішення: JMH (Java Microbenchmark Harness)
escape analysis

Escape Analysis

JIT аналізує чи об'єкт "виходить" за межі методу або потоку. Якщо ні — може оптимізувати розміщення.

// Об'єкт не "escapes" метод: void compute() { Point p = new Point(1, 2); // p не повертається і не зберігається int dist = p.distance(); use(dist); } // JIT може: Stack Allocation або Scalar Replacement // Stack Allocation: p живе у Stack, не в Heap // → немає GC pressure, швидко // Scalar Replacement: p "розкладається" на примітиви // → int x = 1; int y = 2; (без об'єкту взагалі!) // Перевірити EA оптимізацію: -XX:+DoEscapeAnalysis // default on -XX:+EliminateAllocations // scalar replacement, default on // Lock elision: якщо об'єкт не escapes → synchronized видаляється! synchronized(localObject) { ... } // localObject не escapes → lock elision
Lock elision — практичний наслідок EA. StringBuffer (synchronized) у локальній змінній компілюється так само ефективно як StringBuilder, бо JIT видаляє всі локи.

Warmup і Deoptimization

Warmup

JVM потребує часу щоб "прогрітись". Перші запити повільніші бо код ще інтерпретується або компілюється. Для мікросервісів — проблема при cold start.

// CDS (Class Data Sharing) — прискорює старт // Зберігає метадані класів у shared archive -XX:+UseAppCDS -XX:SharedArchiveFile=app.jsa // AOT (Ahead of Time) — Java 9-16, experimental // GraalVM Native Image — компіляція у native binary // (instant startup, менше пам'яті, але без JIT оптимізацій)

Deoptimization

JIT оптимізує на основі профілю (спекулятивно). Якщо припущення порушується — відбувається deopt: повернення до інтерпретатора або перекомпіляція.

// Приклад: JIT припустив що list завжди ArrayList // Оптимізував виклик без vtable lookup // Потім з'явився LinkedList → deoptimization // Megamorphic call site: >2 типів → JIT не може optimizuvati void process(Shape shape) { shape.area(); } // Якщо викликається з Circle, Rect, Triangle → megamorphic // Виявити deopt: -XX:+TraceDeoptimization // або JFR event: jdk.Deoptimization
На співбесіді: Parent Delegation Model (навіщо), Tiered Compilation (C1 + C2), Escape Analysis → Stack Allocation + Scalar Replacement + Lock Elision. Deoptimization — спекулятивні оптимізації JIT.
