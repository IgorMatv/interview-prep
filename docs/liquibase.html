Навіщо Liquibase

Версіонований контроль схеми БД. Як Git для SQL. Дозволяє відтворити точний стан схеми на будь-якому середовищі і відкатитись при помилці.

  • Відтворюваність — dev/staging/prod мають однакову схему
  • Аудит — хто і коли змінив схему
  • Rollback — можливість відкатити зміни
  • CI/CD — міграції як частина деплою
spring.jpa.hibernate.ddl-auto=update — зручно для розробки, але ніколи не використовуй у продакшн. Hibernate може дропнути колонку або таблицю. Тільки Liquibase або Flyway для продакшн.

Liquibase vs Flyway

// Liquibase: XML/YAML/JSON/SQL changesets, rollback вбудований // Flyway: тільки SQL файли, простіший, rollback тільки Pro // Spring Boot підтримує обидва // Liquibase — більш гнучкий, Flyway — простіший для початку
структура

Changeset і changelog

-- resources/db/changelog/db.changelog-master.yaml databaseChangeLog: - includeAll: path: db/changelog/changes/ -- resources/db/changelog/changes/001-create-orders.yaml databaseChangeLog: - changeSet: id: 001-create-orders # унікальний ID author: ihor # автор changes: - createTable: tableName: orders columns: - column: name: id type: BIGINT autoIncrement: true constraints: primaryKey: true - column: name: status type: VARCHAR(50) constraints: nullable: false - column: name: total type: DECIMAL(19,2) rollback: - dropTable: tableName: orders
Changeset — незмінний після застосування. Liquibase зберігає MD5 кожного changeset у DATABASECHANGELOG. Якщо змінити застосований changeset — Liquibase кине ValidationFailedException. Нові зміни → новий changeset.

Spring Boot інтеграція

-- SQL формат changeset (альтернатива YAML) --liquibase formatted sql --changeset ihor:002-add-user-id ALTER TABLE orders ADD COLUMN user_id BIGINT; --rollback ALTER TABLE orders DROP COLUMN user_id;
zero-downtime

Zero-downtime патерни

При rolling deploy нова і стара версія застосунку можуть працювати одночасно. Міграція має бути сумісна з обома.

Додати нову NOT NULL колонку без downtime

-- НЕПРАВИЛЬНО: блокує таблицю, стара версія не знає колонки ALTER TABLE orders ADD COLUMN priority INT NOT NULL DEFAULT 0; -- ПРАВИЛЬНО: 3 деплої -- Деплой 1: додаємо nullable (без DEFAULT → зворотньо сумісно) ALTER TABLE orders ADD COLUMN priority INT; -- Деплой 2: нова версія коду пише priority, бекфіл існуючих UPDATE orders SET priority = 0 WHERE priority IS NULL; -- Деплой 3: додаємо NOT NULL constraint (дані вже заповнені) ALTER TABLE orders ALTER COLUMN priority SET NOT NULL;
У PostgreSQL ADD COLUMN ... DEFAULT ... з незмінним DEFAULT не блокує з PostgreSQL 11+ (instant). Але при змінному DEFAULT або NOT NULL без DEFAULT — таблиця блокується. Завжди перевіряй версію DB.
rename

Rename колонки — expand/contract

Перейменування колонки без downtime потребує 4 кроки: нова колонка → синхронізація → переключення коду → видалення старої.

-- Крок 1: додаємо нову колонку (nullable) ALTER TABLE users ADD COLUMN full_name VARCHAR(200); -- Крок 2: бекфіл існуючих даних UPDATE users SET full_name = name WHERE full_name IS NULL; -- Крок 2b: тригер для синхронізації під час міграції -- (для великих таблиць що активно пишуться) -- Крок 3: новий код пише в обидві колонки -- старий код читає зі старої, новий — з нової -- Крок 4: після повного переходу ALTER TABLE users DROP COLUMN name;
Ніколи не роби RENAME COLUMN в production при rolling deploy — стара версія застосунку одразу зламається. Expand/contract — єдиний безпечний шлях.
rollback

Rollback стратегії

-- Rollback через Liquibase CLI liquibase rollback --tag=v1.0.0 liquibase rollbackCount 3 -- відкатити 3 changeset liquibase rollbackToDate 2024-01-15 -- Автоматичний rollback (Liquibase генерує) -- createTable → dropTable -- addColumn → dropColumn -- createIndex → dropIndex -- Ручний rollback (для складних операцій) changeSet: id: 003-update-data changes: - update: tableName: orders column: name: status value: ACTIVE where: status = 'NEW' rollback: - update: tableName: orders column: name: status value: NEW where: status = 'ACTIVE'
DROP TABLE і DROP COLUMN не мають автоматичного rollback — дані загублені. Завжди архівуй дані перед drop або роби drop через expand/contract (спочатку перестаємо використовувати, потім видаляємо).
команди

Команди і антипатерни

// Maven/Gradle команди ./gradlew liquibaseUpdate // застосувати нові changesets ./gradlew liquibaseStatus // які ще не застосовані ./gradlew liquibaseValidate // перевірити changelog ./gradlew liquibaseDiff // diff між схемою і changelog ./gradlew liquibaseGenerateChangelog // генерувати з існуючої схеми // Теги для rollback ./gradlew liquibaseTag -PliquibaseCommandValue=v2.0.0

Антипатерни

  • Змінювати застосований changeset — ValidationFailedException при наступному запуску
  • Один великий changelog — важко знайти зміну, конфлікти при паралельній роботі
  • Пропускати rollback — при помилці деплою не можна відкатити
  • DDL і DML в одному changeset — важко rollback, різні timeout-и
  • Логіка в SQL changeset — PL/pgSQL процедури важко тестувати і дебажити
На співбесіді: changeset незмінний після застосування, zero-downtime = expand/contract (nullable → backfill → NOT NULL), rollback для структурних змін (не для даних), Liquibase vs Flyway (rollback, формати).
