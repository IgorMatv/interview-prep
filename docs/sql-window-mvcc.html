window

OVER і PARTITION BY

Window function обчислює значення для кожного рядку відносно "вікна" — набору пов'язаних рядків. На відміну від GROUP BY — рядки не згортаються.

-- Синтаксис function_name() OVER ( PARTITION BY col1, col2 -- групи (як GROUP BY але рядки зберігаються) ORDER BY col3 -- порядок всередині вікна ROWS BETWEEN ... AND ... -- frame (опціонально) ) -- Приклад: зарплата і середня по відділу в одному рядку SELECT name, department, salary, AVG(salary) OVER (PARTITION BY department) AS dept_avg, salary - AVG(salary) OVER (PARTITION BY department) AS diff_from_avg FROM employees; -- Результат: -- Alice | Engineering | 5000 | 4500 | +500 -- Bob | Engineering | 4000 | 4500 | -500 -- Carol | HR | 3000 | 3500 | -500
Window functions виконуються після WHERE, GROUP BY, HAVING — але до ORDER BY і LIMIT. Тому не можна фільтрувати за window function у WHERE — потрібен підзапит або CTE.
ranking

ROW_NUMBER, RANK, DENSE_RANK

-- ROW_NUMBER: унікальний номер, без прогалин SELECT name, salary, ROW_NUMBER() OVER (ORDER BY salary DESC) AS rn FROM employees; -- 1, 2, 3, 4, 5 (завжди унікальний) -- RANK: однакові значення → однаковий rank з прогалиною SELECT name, salary, RANK() OVER (ORDER BY salary DESC) AS rnk FROM employees; -- 1, 2, 2, 4, 5 (пропущена 3) -- DENSE_RANK: без прогалин SELECT name, salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS drnk FROM employees; -- 1, 2, 2, 3, 4 -- Топ-1 в кожному відділі (типове питання на співбесіді) SELECT * FROM ( SELECT *, ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS rn FROM employees ) sub WHERE rn = 1;
navigation

LAG, LEAD, Running total

-- LAG: значення попереднього рядку -- LEAD: значення наступного рядку SELECT order_date, total, LAG(total) OVER (ORDER BY order_date) AS prev_total, LEAD(total) OVER (ORDER BY order_date) AS next_total, total - LAG(total) OVER (ORDER BY order_date) AS daily_change FROM orders WHERE user_id = 42; -- LAG з default value (якщо немає попереднього) LAG(total, 1, 0) OVER (ORDER BY order_date) -- LAG(col, offset, default) -- Running total (наростаючий підсумок) SELECT order_date, total, SUM(total) OVER ( ORDER BY order_date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) AS running_total FROM orders;

NTILE і PERCENT_RANK

-- NTILE: розбити на N рівних груп (квантилі) SELECT name, salary, NTILE(4) OVER (ORDER BY salary) AS quartile FROM employees; -- quartile: 1=нижній, 4=верхній -- PERCENT_RANK: відносна позиція 0..1 SELECT name, salary, PERCENT_RANK() OVER (ORDER BY salary) AS pct FROM employees;
CTE

CTE — Common Table Expressions

Іменований підзапит що можна використовувати кілька разів. Покращує читабельність складних запитів.

-- Базовий CTE WITH active_users AS ( SELECT id, name, email FROM users WHERE last_login > NOW() - INTERVAL '30 days' ), high_value_orders AS ( SELECT user_id, SUM(total) AS total_spent FROM orders WHERE status = 'PAID' GROUP BY user_id HAVING SUM(total) > 1000 ) SELECT u.name, u.email, hvo.total_spent FROM active_users u JOIN high_value_orders hvo ON u.id = hvo.user_id ORDER BY hvo.total_spent DESC; -- CTE можна посилатись на попередній CTE в ланцюжку WITH step1 AS (...), step2 AS (SELECT * FROM step1 WHERE ...), step3 AS (SELECT * FROM step2 JOIN ... ) SELECT * FROM step3;
У PostgreSQL до 12: CTE — optimization fence (матеріалізується завжди). З PostgreSQL 12+: планувальник сам вирішує. Якщо хочеш явно: WITH cte AS MATERIALIZED (...) або NOT MATERIALIZED.
recursive

Recursive CTE — дерева і графи

Для ієрархічних структур: категорії, коментарі, оргструктура. Складається з anchor (початок) і recursive member.

-- Таблиця: categories(id, name, parent_id) -- Завдання: знайти всіх нащадків категорії 1 WITH RECURSIVE category_tree AS ( -- Anchor: початковий рядок SELECT id, name, parent_id, 0 AS depth FROM categories WHERE id = 1 UNION ALL -- Recursive member: приєднуємо дітей SELECT c.id, c.name, c.parent_id, ct.depth + 1 FROM categories c JOIN category_tree ct ON c.parent_id = ct.id ) SELECT * FROM category_tree ORDER BY depth, name; -- Breadcrumb (шлях до кореня) WITH RECURSIVE breadcrumb AS ( SELECT id, name, parent_id, name AS path FROM categories WHERE id = 42 UNION ALL SELECT c.id, c.name, c.parent_id, c.name || ' > ' || b.path FROM categories c JOIN breadcrumb b ON c.id = b.parent_id ) SELECT path FROM breadcrumb WHERE parent_id IS NULL;
MVCC

MVCC в PostgreSQL

Multi-Version Concurrency Control — кожна транзакція бачить snapshot даних на момент її початку. Writers не блокують readers, readers не блокують writers.

-- Як PostgreSQL зберігає версії: -- Кожен рядок має xmin (transaction що створила) -- і xmax (transaction що видалила/оновила) -- UPDATE насправді INSERT нового рядку + "видалення" старого UPDATE orders SET status = 'PAID' WHERE id = 1; -- Старий рядок: xmax = поточна tx -- Новий рядок: xmin = поточна tx, xmax = null -- Транзакція читає тільки рядки де: -- xmin <= snapshot_xid AND (xmax = null OR xmax > snapshot_xid) -- Перевірити xmin/xmax SELECT xmin, xmax, * FROM orders WHERE id = 1;
MVCC дозволяє читати без блокувань — навіть якщо хтось пише в той самий рядок, читач побачить стару версію. Але старі версії займають місце → потрібен VACUUM для прибирання.
VACUUM

VACUUM і table bloat

Мертві версії рядків (dead tuples) займають місце. VACUUM їх прибирає і звільняє місце для нових рядків (але не повертає OS без VACUUM FULL).

-- Автовакуум: PostgreSQL запускає автоматично -- Можна запустити вручну VACUUM orders; -- прибирає dead tuples VACUUM ANALYZE orders; -- + оновлює статистику VACUUM FULL orders; -- повертає місце OS (блокує!) -- Перевірити bloat SELECT relname, n_dead_tup, n_live_tup, round(n_dead_tup::numeric / nullif(n_live_tup,0) * 100, 2) AS dead_pct, last_autovacuum FROM pg_stat_user_tables WHERE schemaname = 'public' ORDER BY n_dead_tup DESC; -- Transaction ID wraparound — критична проблема! -- PostgreSQL має 32-bit transaction IDs (2 млрд) -- При підході до ліміту → autovacuum FREEZE -- Якщо не встигне → DB shutdown! -- Моніторити: age(datfrozenxid) для кожної БД
На співбесіді: window functions = обчислення без згортання рядків, ROW_NUMBER vs RANK vs DENSE_RANK, recursive CTE для дерев, MVCC = кожна транзакція бачить snapshot (readers не блокують writers), VACUUM прибирає dead tuples.
