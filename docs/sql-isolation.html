аномалії

Аномалії конкурентного доступу

  • Dirty read — читаємо незакомічені зміни іншої транзакції
  • Non-repeatable read — один і той самий SELECT дає різні результати в рамках транзакції (хтось змінив рядок і закомітив)
  • Phantom read — повторний SELECT повертає нові рядки (хтось додав рядки що підпадають під WHERE)
  • Serialization anomaly — результат не відповідає жодному можливому послідовному виконанню транзакцій
PostgreSQL і dirty read: навіть на рівні READ UNCOMMITTED — PostgreSQL ніколи не дозволяє dirty read. Мінімальний рівень ефективно = READ COMMITTED.

READ COMMITTED

READ COMMITTED детально

Default у PostgreSQL. Кожен окремий SQL-вираз бачить snapshot на момент свого початку (не початку транзакції). Тому можливий non-repeatable read.

-- Non-repeatable read при READ COMMITTED -- TX1: BEGIN; SELECT balance FROM accounts WHERE id = 1; -- → 1000 -- [TX2 робить UPDATE balance=500, COMMIT] SELECT balance FROM accounts WHERE id = 1; -- → 500 (інший результат!) COMMIT; -- REPEATABLE READ: обидва SELECT повернуть 1000 -- snapshot фіксується на початку транзакції BEGIN ISOLATION LEVEL REPEATABLE READ; SELECT balance FROM accounts WHERE id = 1; -- → 1000 -- [TX2 робить UPDATE, COMMIT] SELECT balance FROM accounts WHERE id = 1; -- → 1000 (той самий snapshot) COMMIT;
При REPEATABLE READ якщо TX1 намагається UPDATE рядок що вже змінила TX2 і закомітила — PostgreSQL кине ERROR: could not serialize access due to concurrent update. Потрібен retry.
SELECT FOR UPDATE

SELECT FOR UPDATE — pessimistic locking

Блокує вибрані рядки до кінця транзакції. Інші транзакції чекають або отримують помилку.

-- Класичний use case: дебет рахунку BEGIN; SELECT balance FROM accounts WHERE id = 42 FOR UPDATE; -- блокуємо рядок -- Тепер ніхто не може UPDATE цей рядок UPDATE accounts SET balance = balance - 100 WHERE id = 42; COMMIT; -- Варіанти FOR UPDATE -- ексклюзивний lock FOR SHARE -- shared lock (кілька читачів, блокує writers) FOR NO KEY UPDATE -- не блокує FK reads FOR KEY SHARE -- shared lock на key columns -- NOWAIT: не чекати, одразу помилка якщо lock зайнятий SELECT * FROM orders WHERE id = 1 FOR UPDATE NOWAIT; -- SKIP LOCKED: пропустити заблоковані рядки SELECT * FROM jobs WHERE status = 'PENDING' FOR UPDATE SKIP LOCKED LIMIT 10;
SKIP LOCKED — стандартний патерн для queue processing. Кожен worker бере 10 невідкладних задач без блокування інших workers.
deadlock

Deadlock в PostgreSQL

PostgreSQL автоматично виявляє deadlock і завершує одну з транзакцій з помилкою ERROR: deadlock detected.

-- Класичний deadlock -- TX1: TX2: BEGIN; BEGIN; UPDATE accounts SET ... UPDATE accounts SET ... WHERE id = 1; -- lock 1 WHERE id = 2; -- lock 2 UPDATE accounts SET ... UPDATE accounts SET ... WHERE id = 2; -- чекає TX2 WHERE id = 1; -- чекає TX1 -- → DEADLOCK! -- PostgreSQL вибирає жертву і rollback -- Запобігання: фіксований порядок блокування -- Завжди блокувати в одному порядку (за id ASC) UPDATE accounts SET ... WHERE id IN (1, 2) ORDER BY id; -- або два UPDATE в порядку зростання id
При deadlock PostgreSQL логує деталі в pg_log. Перевіряй: SELECT * FROM pg_locks; і SELECT * FROM pg_stat_activity; для моніторингу активних lock-ів.
advisory locks

Advisory Locks

Application-level locks на довільний integer ключ. Не пов'язані з конкретними рядками — для координації між додатками.

-- Session-level: тримається до явного unlock або відключення SELECT pg_advisory_lock(12345); -- отримати lock SELECT pg_advisory_unlock(12345); -- звільнити -- Transaction-level: автоматично звільняється при commit/rollback SELECT pg_advisory_xact_lock(12345); -- Non-blocking: повертає false якщо lock зайнятий SELECT pg_try_advisory_lock(12345); -- Shared lock SELECT pg_advisory_lock_shared(12345); -- Типовий use case: scheduled job (один виконавець) BEGIN; SELECT pg_try_advisory_xact_lock(hashtext('daily-report')); -- якщо false → інший процес вже виконує -- якщо true → виконуємо задачу COMMIT;
queue патерн

SKIP LOCKED — queue processing

-- Таблиця-черга задач CREATE TABLE jobs ( id BIGSERIAL PRIMARY KEY, status VARCHAR(20) DEFAULT 'PENDING', payload JSONB, created_at TIMESTAMPTZ DEFAULT NOW() ); -- Worker: атомарно бере і блокує задачі WITH next_jobs AS ( SELECT id FROM jobs WHERE status = 'PENDING' ORDER BY created_at LIMIT 5 FOR UPDATE SKIP LOCKED -- пропускаємо зайняті іншими workers ) UPDATE jobs SET status = 'PROCESSING' WHERE id IN (SELECT id FROM next_jobs) RETURNING *; -- Кілька workers можуть виконувати паралельно -- SKIP LOCKED гарантує: кожна задача одному worker
PostgreSQL як message queue — простий і надійний підхід для невеликих обсягів задач. При великих throughput (тисячі задач/сек) — краще Kafka або RabbitMQ.
На співбесіді: 4 аномалії (dirty/non-repeatable/phantom/serial), READ COMMITTED = snapshot per statement, REPEATABLE READ = snapshot per transaction, SELECT FOR UPDATE = row-level lock, SKIP LOCKED для queue, PostgreSQL auto-detects deadlock.
