CAP

CAP теорема

Розподілена система може гарантувати тільки 2 з 3: Consistency, Availability, Partition tolerance. При network partition треба вибирати між C і A.

CP
Consistency + Partition tolerance
MongoDB, HBase, ZooKeeper
→ жертвує availability
AP
Availability + Partition tolerance
Cassandra, DynamoDB, CouchDB
→ eventual consistency
Partition tolerance — обов'язкова для реальної розподіленої системи (мережа ЗАВЖДИ може розпастись). Тому реальний вибір: CP або AP при partition. CA — тільки для single-node.

На практиці

  • PostgreSQL — CA (single node), CP при реплікації з synchronous commit
  • MySQL + synchronous replication — CP
  • Cassandra — AP за замовчуванням, можна tunable consistency
  • Redis Cluster — AP (async replication → можлива втрата даних)
  • Kafka — CP (ISR + acks=all)
PACELC

PACELC — розширення CAP

CAP описує тільки поведінку при partition. PACELC додає: навіть без partition є trade-off між Latency і Consistency.

// PACELC: If Partition → A or C, Else → L or C // // DynamoDB: PA/EL → доступна при partition, низька latency без // Cassandra: PA/EL → те саме, tunable // MongoDB: PC/EC → консистентна, вища latency // PostgreSQL:PC/EC → strong consistency, synchronous // MySQL: PC/EC
На практиці важливіші питання: яка consistency model потрібна? (strong, eventual, read-your-writes), яка latency прийнятна, і який throughput потрібен. CAP — корисна шпаргалка, але не повна картина.

Back-of-envelope розрахунки

// Корисні константи: // 1 байт = 8 біт // 1 KB = 10^3, 1 MB = 10^6, 1 GB = 10^9, 1 TB = 10^12 // 1 секунда = 10^9 наносекунд // 1 день = 86400 ≈ 10^5 секунд // 1 місяць ≈ 2.5 × 10^6 секунд // 1 рік ≈ 3 × 10^7 секунд // Приклад: Twitter-подібний сервіс // 300M active users, 50% daily → 150M DAU // Кожен читає 100 tweets/day → 15B reads/day // 15B / 86400 ≈ 175K reads/sec // Запис: 1% users пишуть 2 tweets/day // 1.5M × 2 / 86400 ≈ 35 writes/sec // Read/Write ratio ≈ 5000:1 → read-heavy! // Storage: 1 tweet ≈ 280 chars = 280 bytes ≈ 300 bytes // 35 writes/sec × 300 bytes = 10 KB/sec // За рік: 10 KB × 3×10^7 ≈ 300 GB text/year

// Один сервер (Spring Boot, 8 cores): // ~10K-50K HTTP requests/sec (залежно від операції) // ~10 concurrent DB connections → ~1K DB queries/sec // PostgreSQL single node: // ~10K-50K simple queries/sec // ~1K complex queries/sec // ~100 transactions/sec з важкими joins // Redis: // ~100K-1M GET/SET operations/sec (single node) // Latency: < 1ms для більшості операцій // Kafka: // ~1M messages/sec per broker (з правильним tuning) // Latency: ~5-15ms end-to-end // Загальні правила: // Vertical scaling: до ~64 cores, 1TB RAM // Horizontal scaling: stateless сервіси → легко // DB sharding: від ~10M users або 10TB // CDN: статика, ~95% cache hit ratio
reliability

SLA, SLO, SLI

  • SLI (Service Level Indicator) — вимірювана метрика: latency p99, error rate, availability
  • SLO (Service Level Objective) — внутрішня ціль: p99 latency < 200ms, availability 99.9%
  • SLA (Service Level Agreement) — зовнішній договір з штрафами: 99.9% uptime або credit
// Availability і downtime per year: // 99% → 3.65 days downtime/year // 99.9% → 8.7 hours downtime/year ("three nines") // 99.99% → 52 minutes downtime/year ("four nines") // 99.999%→ 5 minutes downtime/year ("five nines") // Error budget: 99.9% → 0.1% = 8.7 hours/year // Витрачаємо на planned maintenance, incidents // Типові latency SLO: // p50 < 50ms, p99 < 200ms, p999 < 1s // p99.9 важливіший ніж average для UX
p99 vs average: середня latency 50ms може ховати p99=2s. 1% users з 2s — при 10K QPS = 100 users/sec отримують погану відповідь. Завжди дивись на percentiles, не середнє.
інтерв'ю

Структура SD інтерв'ю

Стандартний фреймворк для будь-якого System Design питання. 45 хвилин → чіткий розподіл часу.

  • 1. Clarify (5 хв) — уточнити вимоги: DAU, QPS, latency SLO, consistency requirements. Не починай проектувати до розуміння масштабу.
  • 2. Back-of-envelope (5 хв) — порахуй: QPS reads/writes, storage/year, bandwidth. Це визначить архітектурні рішення.
  • 3. High-level design (10 хв) — клієнт → LB → сервіси → DB. Компоненти і взаємодія.
  • 4. Deep dive (15 хв) — інтерв'юер фокусує: DB schema, caching strategy, або конкретний компонент.
  • 5. Bottlenecks (5 хв) — SPOF, масштабування, failover.
Ключова порада: говори вголос про trade-offs. "Ми можемо використати X, але тоді Y. Альтернатива — Z з іншим trade-off." Інтерв'юер оцінює мислення, не правильну відповідь.
