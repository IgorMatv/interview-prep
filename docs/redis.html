Spring Cache

@Cacheable, @CacheEvict, @CachePut

// Увімкнути кешування @EnableCaching @SpringBootApplication class App { ... } // @Cacheable: повертає з кешу або виконує і кешує @Cacheable(value = "pois", key = "#id") public Poi findById(long id) { return poiRepo.findById(id).orElseThrow(); // при повторному виклику — з Redis, не з DB } // @CacheEvict: видаляємо при зміні @CacheEvict(value = "pois", key = "#poi.id") public void update(Poi poi) { poiRepo.save(poi); } // @CachePut: завжди виконує і оновлює кеш @CachePut(value = "pois", key = "#result.id") public Poi save(Poi poi) { return poiRepo.save(poi); } // Умовне кешування @Cacheable(value = "routes", condition = "#stops.size() > 3") @Cacheable(value = "routes", unless = "#result == null")
@Cacheable — AOP proxy. Self-invocation не спрацює (як і @Transactional). Метод повинен бути public і викликатись через Spring bean.
патерни

Cache-aside, Write-through, Write-behind

Cache-aside (Lazy loading) — найпоширеніший

// Читання: спочатку кеш, потім DB public Route getRoute(long id) { Route cached = redis.opsForValue().get("route:" + id); if (cached != null) return cached; // cache hit Route route = routeRepo.findById(id).orElseThrow(); // cache miss redis.opsForValue().set("route:" + id, route, Duration.ofHours(1)); return route; } // Запис: DB + інвалідація кешу public void updateRoute(Route route) { routeRepo.save(route); redis.delete("route:" + route.getId()); // evict }

Write-through — синхронний запис в обидва

public void save(Route route) { routeRepo.save(route); // DB redis.opsForValue().set("route:" + route.getId(), // Cache route, Duration.ofHours(1)); // Кеш завжди актуальний, але подвійний запис }
distributed lock

Distributed lock через Redisson

Коли кілька нод застосунку не повинні виконувати одну операцію одночасно — distributed lock через Redis.

// Redisson: готова реалізація distributed locks // implementation 'org.redisson:redisson-spring-boot-starter:...' @Autowired RedissonClient redisson; void processUniqueJob(long jobId) { RLock lock = redisson.getLock("job-lock:" + jobId); boolean locked = false; try { locked = lock.tryLock(0, 30, TimeUnit.SECONDS); // 0 = не чекати якщо зайнятий // 30 = auto-release після 30 секунд if (!locked) return; // хтось вже обробляє doWork(jobId); } finally { if (locked) lock.unlock(); } } // RedLock алгоритм: lock на 3+ незалежних Redis нодах // Для максимальної надійності при failover
Простий SETNX без Redisson — небезпечно: клієнт може впасти після lock, TTL не спрацює правильно при clock skew. Використовуй Redisson — він вирішує ці проблеми.
TTL і eviction

TTL, eviction policies, антипатерни

// Eviction policies (redis.conf: maxmemory-policy) noeviction // OOM error при переповненні (default) allkeys-lru // видаляти найменш використовувані з усіх volatile-lru // LRU тільки серед ключів з TTL allkeys-random // випадкові з усіх volatile-ttl // видаляти ключі з найменшим TTL allkeys-lfu // LFU (least frequently used), Redis 4+ // Для кешу: allkeys-lru або allkeys-lfu // Для сесій: volatile-ttl або volatile-lru

Антипатерни

  • Кешувати без TTL — пам'ять заповниться, eviction видалить довільні ключі
  • Thundering herd — одночасне закінчення TTL у багатьох ключів → шторм запитів до DB. Рішення: random jitter до TTL
  • Cache stampede — cache miss під навантаженням → всі одночасно йдуть в DB. Рішення: Probabilistic Early Expiration або mutex lock
  • Великі значення — об'єкти >1MB блокують Redis event loop. Розбий або стисни
  • KEYS * в продакшн — O(n) блокує Redis. Використовуй SCAN
