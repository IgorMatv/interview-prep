HashMap — що всередині

Це масив (Node[] table) де кожен елемент — голова ланцюжка або корінь червоно-чорного дерева.

// Спрощено: HashMap<K,V> transient Node<K,V>[] table; // масив бакетів int size; // кількість пар int threshold; // size > threshold → resize float loadFactor = 0.75f; // за замовчуванням static class Node<K,V> { final int hash; final K key; V value; Node<K,V> next; // linked list }

threshold = capacity × loadFactor — при 16 бакетах і 0.75 це 12. Щойно додається 13-й елемент — масив подвоюється до 32.

Чому 0.75? Математичний компроміс між пам'яттю та швидкістю. При 1.0 — мало resize, але багато колізій. При 0.5 — мало колізій, але вдвічі більше пам'яті. 0.75 ≈ ln(2) — оптимальна точка для random hashing.
алгоритм

hash() і вибір бакету

Java не використовує hashCode() напряму — робить perturbation: XOR старших 16 біт з молодшими. Це зменшує колізії коли capacity маленька.

// Реальний код HashMap.hash() static final int hash(Object key) { int h; return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16); } // Індекс бакету (n = table.length, завжди 2^k) int index = (n - 1) & hash; // бітова маска ≡ hash % n
Чому capacity завжди 2^k? Щоб замість повільного hash % n використовувати бітовий (n-1) & hash. Це працює виключно при n = 2^k.

Що відбувається при колізії

Якщо два ключі потрапляють в один бакет — новий вузол додається в хвіст ланцюжка (Java 8+, раніше в голову). Пошук: O(1) в середньому, O(n) worst case.

Якщо перевизначити hashCode() так, що всі об'єкти повертають одне значення — HashMap деградує до O(n) linked list.
оптимізація

Treeify — коли список стає деревом

Якщо в одному бакеті 8+ вузлів і загальна capacity ≥ 64 — linked list перетворюється на TreeNode (червоно-чорне дерево). O(n) → O(log n).

// Константи HashMap static final int TREEIFY_THRESHOLD = 8; static final int UNTREEIFY_THRESHOLD = 6; static final int MIN_TREEIFY_CAPACITY = 64; // Після кожного put у переповнений бакет: if (binCount >= TREEIFY_THRESHOLD - 1) treeifyBin(tab, hash); // але якщо table.length < 64 → resize замість treeify

Нюанс: MIN_TREEIFY_CAPACITY = 64

Якщо capacity < 64 і бакет переповнений — HashMap спочатку зробить resize, а не treeify. При малій таблиці краще перехешувати.

Гістерезис 8 / 6

При видаленні: якщо вузлів ≤ 6 — дерево стає списком знову. Різниця між 6 і 8 запобігає безперервним конверсіям при почерговому add/remove.

На практиці treeify майже ніколи не спрацьовує при нормальному hashCode(). Якщо бачите дерева — перевірте хеш-функцію.
критично

Resize — подвоєння масиву

Коли size > threshold — HashMap створює новий масив capacity × 2 і перерозподіляє всі елементи. Це O(n) операція.

// Спрощено: HashMap.resize() int newCap = oldCap << 1; // × 2 Node<K,V>[] newTab = new Node[newCap]; for (int j = 0; j < oldCap; j++) { Node<K,V> e = oldTab[j]; if (e != null) { // перевіряємо один біт, що додався if ((e.hash & oldCap) == 0) // індекс не змінюється: j else // новий індекс: j + oldCap } }

Розумний трюк Java 8 — без повного rehash

При подвоєнні capacity додається рівно один біт у масці індексу. Старий хеш або дає той самий індекс (новий bit = 0) або oldIndex + oldCapacity (новий bit = 1). Перераховувати хеш не потрібно.

Як уникнути зайвих resize

// Якщо знаєте розмір наперед new HashMap<>(1334); // 1000 / 0.75 + 1 // Guava (зручніше) Maps.newHashMapWithExpectedSize(1000); // Java 19+ HashMap.newHashMap(1000);
HashMap не thread-safe під час resize. У Java 6 міг виникнути infinite loop. У Java 8+ — data race та втрата даних. Використовуйте ConcurrentHashMap.
підклас

LinkedHashMap

Розширює HashMap. Кожен Entry має два додаткових поля: before і after — doubly linked list поверх hash table.

static class Entry<K,V> extends HashMap.Node<K,V> { Entry<K,V> before, after; // doubly linked list } // Два режими: new LinkedHashMap<>() // insertion order new LinkedHashMap<>(16, 0.75f, true) // access order (LRU)

LRU Cache — класична реалізація

accessOrder = true + перевизначити removeEldestEntry. При кожному get або put звернений елемент переміщується в хвіст. Голова = найменш нещодавно використаний.

int MAX_SIZE = 100; var lruCache = new LinkedHashMap<K, V>( 16, 0.75f, true) { // accessOrder! @Override protected boolean removeEldestEntry( Map.Entry<K, V> eldest) { return size() > MAX_SIZE; // видаляємо LRU } };

TreeMap

Реалізує NavigableMap через червоно-чорне дерево. Ключі завжди відсортовані. hashCode() не використовується взагалі — тільки compareTo / Comparator.

// Природній порядок new TreeMap<String, Integer>() // Reverse order new TreeMap<>(Comparator.reverseOrder()) // NavigableMap API map.firstKey() // мінімальний ключ map.lastKey() // максимальний ключ map.floorKey("cat") // найбільший ≤ "cat" map.ceilingKey("cat") // найменший ≥ "cat" map.subMap("a", "m") // підмножина [a, m) map.headMap("m") // усе < "m" map.tailMap("m") // усе ≥ "m"
O(log n)
get / put / remove
null key
NullPointerException

Коли TreeMap, а не HashMap

  • Потрібна sorted iteration
  • Range queries: subMap, headMap, tailMap
  • Closest-match: floorKey, ceilingKey
  • Ключ не має нормального hashCode()
Класичний баг: TreeMap використовує тільки compareTo. Якщо Comparator повертає 0 для двох різних об'єктів — другий мовчки перезапише перший, навіть якщо equals() повертає false.

Правило вибору

  • Просто key-value без гарантій порядку → HashMap
  • Потрібен порядок вставки або LRU-cache → LinkedHashMap
  • Sorted keys або range queries → TreeMap
  • Concurrent read/write → ConcurrentHashMap
  • Read-heavy, write-rare → CopyOnWriteArrayMap (рідко)
На співбесіді: намалюй структуру HashMap (масив + list/tree), поясни hash(), treeify умову, скажи що resize — O(n). Потім поясни різницю трьох Map через "порядок і складність".
